    コードレビュー結果: S09「単品を発注する」¶

    レビュー対象¶

    • コミット範囲: 4bdbb2a..67c1e7b（3 コミット、25 ファイル、+1,529 行）
    • バックエンド: PurchaseOrder エンティティ、UseCase、Prisma リポジトリ、API ルート
    • フロントエンド: PurchaseOrderForm コンポーネント、App.tsx 画面遷移
    • E2E テスト: 5 シナリオ

    総合評価¶

    TDD サイクルが丁寧に回され、レイヤードアーキテクチャの依存方向も正しく守られた良い実装。ドメイン用語との一致度が高く、テストがドキュメントとして十分機能している。一方で、(1) 発注 API に送信する数量がユーザー入力値のままで自動調整後の値ではない点、(2) エラーハンドリングの欠如、(3) undefined as unknown キャストによる型安全性の破壊が主要な改善課題。

    改善提案（重要度順）¶

    高（マージ前に対応すべき）¶

    # 提案 箇所 指摘元 理由
    H1 発注 API に送る数量を自動調整後の値にする（現在はユーザー入力値のまま） PurchaseOrderForm.tsx:54-56 user-rep 画面表示「自動調整後: 20 本」と実際の発注数量が食い違う。業務上許容できない
    H2 fetchItemInfo / handleSubmit にエラーハンドリングを追加 PurchaseOrderForm.tsx:30-32, 47-61 programmer, architect, writer, user-rep API 失敗時にユーザーへのフィードバックがなく、画面が空白 or 無応答になる
    H3 undefined as unknown as PurchaseOrderId を型安全な設計に置換 purchase-order.ts:44 programmer, architect, writer 型安全性を破壊。将来 .value アクセスで実行時エラーの可能性
    H4 購入単位倍数調整の境界値テストを追加 purchase-order.test.ts tester quantity=9,10,11 に対する purchaseUnit=10 の on/off ポイントが未テスト

    中（対応推奨）¶

    # 提案 箇所 指摘元 理由
    M1 エラーハンドリングを文字列比較からカスタムエラークラスに変更 purchase-order-routes.ts:27-35 programmer, architect, writer メッセージ変更で壊れる脆弱な結合
    M2 fetchItemInfo で全単品一覧取得→find ではなく個別取得 API を使用 App.tsx:169-181 programmer, architect 単品数増加時のパフォーマンス劣化
    M3 仕入先名のハードコード解消 App.tsx:179 programmer, architect 仕入先 ${item.supplierId} は仮実装のまま
    M4 Repository インターフェースの findByStatus を PurchaseOrderStatus 型に purchase-order-repository.ts:15 architect 生の string を使っており値オブジェクトが活用されていない
    M5 Repository の findByStatus/findByItemIdAndStatus がドメインオブジェクトでなく Record を返す purchase-order-repository.ts:14-16 architect 他の Repository（Item, Order）との一貫性が崩れている
    M6 フロントエンドの数量調整にコメント追加「プレビュー用。正式な調整はサーバー側」 PurchaseOrderForm.tsx:35-38 programmer, writer バックエンドとのロジック重複の意図を明示

    低（改善の余地あり）¶

    # 提案 箇所 指摘元 理由
    L1 createNew テストの vi.useFakeTimers() を beforeEach にまとめる purchase-order.test.ts programmer, tester 4 テストで重複
    L2 toRecord メソッドの不要な変換を除去 purchase-order-repository-prisma.ts:79-89 programmer 入力と出力が同一構造
    L3 バックエンド Date 型とフロントエンド string 型の不一致を明示 purchase-order-usecase.ts / types/purchase-order.ts programmer, writer JSON シリアライズで暗黙変換されるが型の意図が曖昧

    矛盾事項¶

    # 視点 A 視点 B 論点 推奨判断
    1 programmer: 数量調整の重複は共有パッケージに切り出す writer: プレビュー用コメントで意図を明示すれば十分 フロントエンド/バックエンドのロジック重複 コメント追加で十分（MVP フェーズ）。共有パッケージは過剰

    エージェント別フィードバック詳細¶

    xp-programmer（高: 3 / 中: 4 / 低: 3） ### 評価サマリー TDD サイクルが丁寧に回されており、レイヤード・アーキテクチャの責務分離も既存パターンに合致した良い実装。ただし `undefined as unknown` キャストとビジネスロジック重複が設計上の負債。 ### 良い点 - テストカバレッジが充実（各層で適切にテスト） - イミュータブル設計（`receive()` が新インスタンスを返す） - バリデーションが層ごとに適切 - Props 注入によるテスタビリティ - `isSubmitting` によるダブルサブミット防止 ### 主要指摘 - `undefined as unknown as PurchaseOrderId` は型安全性を破壊 - 購入単位倍数調整がフロントエンド/バックエンドで重複（DRY 違反） - `fetchItemInfo` のエラーハンドリング欠如 - 入荷予定日のタイムゾーン問題（深夜帯で 1 日ズレの可能性）
    xp-tester（高: 1 / 中: 2 / 低: 2） ### 評価サマリー テストピラミッドのバランスは良好。ドメイン層 7、アプリ層 4、API 層 4、フロントエンド 7、E2E 5 と各層に分布。 ### 主要指摘 - 購入単位倍数調整の境界値テスト不足（on/off ポイント: 9, 10, 11） - `vi.useRealTimers()` を `afterEach` に移動すべき - 発注確定→在庫推移反映の統合テストが不足
    xp-architect（高: 2 / 中: 4 / 低: 2） ### 評価サマリー 依存方向は正しく、既存パターンとの一貫性も概ね保たれている。Repository インターフェースが Record 型を返す点と `undefined as unknown` キャストが設計上の課題。 ### 良い点 - 依存関係の方向が正しい（Domain <- Application <- Infrastructure） - 値オブジェクトの活用 - ドメインロジックの凝集 ### 主要指摘 - Repository が Record を返す設計は他の Repository との一貫性が崩れている - `findByStatus` の引数が生の `string`（`PurchaseOrderStatus` を使うべき） - トランザクション境界が不明確 - `App.tsx` の肥大化（368 行）
    xp-technical-writer（高: 2 / 中: 4 / 低: 2） ### 評価サマリー ドメイン用語との一致度が高く、テストがドキュメントとして機能している。エラーハンドリングの欠如と非自明なビジネスロジックへのコメント不足が課題。 ### 良い点 - 業務用語がそのままコードに反映 - テスト名が日本語で受入条件と対応 - イテレーション計画の設計セクションが充実 ### 主要指摘 - `createNew` の数量調整ロジックにコメントがない - API エラーレスポンス（400, 404, 500）の仕様がドキュメントに未記載 - 完了条件（DoD）と進捗率 100% の矛盾 - トランザクション ADR（タスク 0.5）が未作成
    xp-user-representative（高: 3 / 中: 2 / 低: 1） ### 評価サマリー 在庫推移→発注→在庫推移の業務フローは自然で、購入単位の自動調整プレビューは親切。ただし発注 API に送る数量が調整前の値である点と、エラーフィードバック欠如が業務上の重大な問題。 ### 良い点 - 在庫推移画面の各行に「発注」ボタンがあり導線が明確 - 購入単位の倍数調整がリアルタイムプレビュー - 発注確定後に在庫推移画面に自動復帰 ### 主要指摘 - 発注 API に送信する数量がユーザー入力値のまま（自動調整後の値ではない） - エラーハンドリングの欠如（3 イテレーション連続の指摘） - 発注確定前の確認ダイアログがない（誤操作リスク）
