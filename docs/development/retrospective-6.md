## KPT 分析

### Keep（続けること）

- **DDD レイヤー構成パターンの再利用**: shipping アプリを products → orders → inventory → purchasing と同じパターンで構築。5 つ目の Django アプリも一貫した構成で迷いなく実装できた。パターン再利用による高速化が定着
- **インサイドアウト TDD の徹底**: 全 3 ストーリーをドメイン → アプリケーション → インフラ → プレゼンテーションの順に TDD で実装。308 テスト全パス、カバレッジ 96% を維持
- **2 イテレーション連続 13SP 達成**: IT5 に続き IT6 でも 13SP を達成。Phase 1 平均 9.25SP に対して Phase 2 平均 13SP と大幅向上。パターンの蓄積と習熟効果が明確に表れた
- **OrderStatus の適切な拡張**: PREPARING/SHIPPED を追加し、CONFIRMED → PREPARING → SHIPPED の遷移をドメインエンティティで管理。不正遷移のガード（ValueError）も網羅
- **XP マルチパースペクティブレビューの実施と対応**: IT5/IT6 の成果物に対してコード品質・テスト品質・設計整合性の観点からレビューを受け、高優先度指摘 H-1〜H-7 を全件修正した
- **アプリ間連携の深化**: ShippingService が OrderRepository と ShipmentRepository を注入し、出荷→注文ステータス更新→出荷記録作成を一連の処理として実現。ドメイン境界を越えた協調パターンが成熟

### Problem（問題点）

- **IT5 Try の一部持ち越し**: SonarQube Quality Gate チェック（5IT 連続持ち越し）、FakeRepository 準拠テスト、テストヘルパー共通化（conftest.py）が最終イテレーションでも未実施。リリース完了時点で「技術的負債リスト」として明示化が必要
- **レビュー指摘で発見された ORM/ドメイン不整合**: OrderStatus に preparing/shipped が ORM choices に含まれていなかった（H-1）。ドメイン拡張時に ORM 側の同期を忘れるリスクが構造的に存在する
- **datetime.now() の使用**: ShippingService で timezone.now() ではなく datetime.now() を使用していた（H-3）。Django プロジェクトではタイムゾーン対応が必須。静的解析ルールの追加が必要
- **PurchaseOrder の DB 復元時バリデーション問題**: Repository からの復元時に __post_init__ バリデーションが走り、過去日付の発注が復元できない問題があった（H-2）。reconstruct() パターンで解決したが、同様の問題が他のエンティティでも潜在的に存在
- **テスト名と内容の矛盾**: test_変更期限超過でエラー が実際には正常系のテストだった（H-5）。テスト名とアサーションの整合性をレビューで担保する必要がある
- **コミット粒度の改善が不十分**: IT5 の 1 コミットから改善したが、まだストーリー単位の大粒度。TDD サイクル（Red-Green-Refactor）単位のコミットには到達しなかった

### Try（次のリリースに向けて）

| アクション | 優先度 | 期待効果 |
| :--- | :--- | :--- |
| SonarQube を導入し品質ゲートを CI に組み込む | 高 | 5IT 連続持ち越しの技術的負債を解消 |
| ORM choices とドメイン Enum の自動同期チェックを追加 | 高 | ORM/ドメイン不整合の構造的防止 |
| datetime.now() → timezone.now() のリントルール追加 | 中 | タイムゾーン関連バグの予防 |
| reconstruct() パターンを Order エンティティにも適用 | 中 | DB 復元時の一貫性確保 |
| FakeRepository 準拠テストの追加 | 中 | Repository 実装の品質保証 |
| テストヘルパー共通化（conftest.py） | 低 | テストコードの DRY 改善 |
