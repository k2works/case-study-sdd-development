## Keep（継続すること）

- **TDD サイクルの厳守**: Order モデルの enum 移行・cancel!・change_delivery_date! を全てテストファーストで実装。リファクタリングの安全網として機能
- **Service Object パターンの一貫した活用**: OrderService を ShippingService/PurchaseOrderService と同じ構造で実装。設計の一貫性により新機能追加のコストが低い
- **IT5 Try の反映**: 確認ダイアログ（data-turbo-confirm）をキャンセル処理に適用。cancellable? によるビジネスルールの一箇所集約を実現
- **3 エージェント並列レビュー**: programmer/tester/architect の 3 並列でレビューを実施。API リミットを回避しつつ多角的フィードバックを取得
- **バッファ候補の消化**: S06（届け先コピー: 3SP）をバッファ候補に設定していたが、全 14SP を消化できた

---

## Problem（問題点）

- **Order enum 移行時の参照漏れ**: `Order::STATUSES` 定数を View で参照していたため、enum 移行後に既存テストが 3 件失敗。grep で事前に全参照箇所を確認すべきだった
- **OrderService の未使用 DI**: `current_date` を注入可能にしたが実際には使用せず、レビューで YAGNI 違反を指摘された。使用しない段階で DI を実装するのは過剰設計
- **在庫引当の明示的な解放処理が未実装**: ドメインモデル設計では `cancel` 時に在庫引当解除が責務とされているが、現実装では StockForecastService が `status:ordered` のみを計算対象にするため、自動的に解除される。設計と実装の乖離がある
- **カバレッジ微減**: 95.85% → 95.68%（-0.17%）。新規ファイル追加に対してテストは書いたが、View の一部パスがカバーされていない
