# イテレーション 6 ふりかえり

## 対象

| 項目 | 内容 |
|------|------|
| イテレーション | IT6 |
| 期間 | 2026-06-01 から 2026-06-12 まで |
| 対象ストーリー | `US-07`, `US-08` |
| 実績 SP | 10 / 10 |

## Keep

- `US-07` と `US-08` を同一イテレーションで完了し、 `Release 1.2` の体験改善機能を計画どおり 10 SP で完了できた。
- Backend と Frontend の受け入れ観点を先に固定したことで、届け先再利用と届け日変更をそれぞれフルスタックで着地できた。
- `developing-review` で出た入力検証、照合正規化、派生表示再取得の指摘を同一イテレーション内で解消し、全テストと型検査の通過まで持ち込めた。

## Problem

- 再注文履歴の照合条件に表記揺れの観点が不足しており、レビュー段階でメールアドレスと電話番号の正規化が必要になった。
- 届け日変更の入力不正と、変更後に再取得すべき管理画面データの範囲が初回実装時に十分固定されていなかった。
- `tracking-progress`、報告書、 GitHub 同期、ドキュメントインデックス更新は今回も実装完了後にまとめて実施しており、終了処理の即時性はまだ弱い。

## Try

| アクション | 担当 | 期限 | 期待効果 |
|-----------|------|------|----------|
| 顧客照合と日付入力のような表記揺れ / 不正入力観点を、受け入れ基準からテストケースへ明示的に落とし込む | - | 次回同種ストーリー着手時 | レビュー起点の入力系手戻り削減 |
| 管理画面の更新系実装では、変更対象だけでなく依存表示の再取得一覧を実装前に洗い出す | - | 次回管理画面改修時 | 派生表示の更新漏れ防止 |
| イテレーション Day 9 の終了処理に `tracking-progress`、報告書、 GitHub 同期、 `operating-docs` を固定タスクとして前倒しで入れる | - | 次イテレーション終了時 | 計画書と運用状態の反映遅れ防止 |
