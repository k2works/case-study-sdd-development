# イテレーション 7 ふりかえり

## KPT

### Keep（続けること）

1. **TDD サイクルの徹底**: バックエンド（CustomerQueryPort, GetCustomerDetailUseCase, DashboardQueryService）をすべて Red-Green-Refactor で実装。テストが安全網として機能し、ダッシュボードの大規模リファクタリングでもリグレッションなし
2. **マルチパースペクティブレビューの効果**: 5 エージェント並列レビューでダッシュボードの API パス不整合（実行時バグ）、SecurityConfig デッドルール、設計書との乖離を発見。テスト不在のコードに潜むバグを検出
3. **レビュー指摘の即時対応**: シードデータの DRY リファクタリング、ダッシュボードの責務分離を同イテレーション内で完了。技術的負債の蓄積を防止
4. **既存パターンの再利用**: CustomerQueryPort は BundlingQueryService/ShipmentQueryService のパターンを踏襲。フロントエンドも OrderAdminPage のテーブル+詳細パターンを得意先管理で再利用
5. **Clock 注入パターン**: DashboardQueryService に Clock を注入し、テストでの日付制御を実現。テスタビリティの向上

### Problem（問題点）

1. **テスト・リリース準備タスクが未完了**: 計画の 18.5h 分（統合テスト、E2E、回帰テスト、ArchUnit、リリース準備）が未着手。実装に集中した結果、品質保証フェーズに入れなかった
2. **ダッシュボードの API パス不整合が長期間放置**: IT4 で実装されたダッシュボードに、フロントエンドと SecurityConfig のパス不整合が存在していた。TDD を徹底していれば防げた問題
3. **設計書（S-100）と実装の乖離**: ダッシュボードが設計書の 3 カード構成を満たしていなかった。イテレーション 4 の実装時に暫定対応のまま放置されていた
4. **日本語 Nested クラス名の非互換**: テストの `@Nested class 受注サマリ` がコンパイルエラーに。環境依存の問題を事前に把握できていなかった

### Try（次に試すこと）

1. **統合テスト・回帰テストをバッファイテレーションで優先実施**: 認可テスト（ロール別アクセス制御）と全フェーズ回帰テストを最優先で完了する
2. **実装後のレビューを標準プロセス化**: 今回のレビューで発見したダッシュボードの問題は、実装直後にレビューしていれば数イテレーション前に修正できた
3. **API パスの一致をテストで保証**: Controller テスト（統合テスト）でエンドポイントの疎通を必ず検証する。SecurityConfig とのパス一致もテストで保証する
4. **設計書との差異を明示的に管理**: 暫定実装の場合は TODO コメントまたはイテレーション計画に明記し、設計書との差異を可視化する
