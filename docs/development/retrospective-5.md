## Keep（継続すること）

- **TDD サイクル（Red → Green → Refactor）の厳守**: ShippingService の出荷処理・在庫消費ロジックを段階的にテストファーストで実装。複雑なビジネスロジックでも安全に開発できた
- **Service Object パターンの踏襲**: PurchaseOrderService と同じパターンで ShippingService を実装。設計の一貫性によりコードの予測可能性が向上
- **5 エージェント並列レビューによる多角的フィードバック**: programmer, tester, architect, technical-writer, user-representative の 5 視点から重要な問題（トランザクション境界、排他制御）を発見
- **FEFO（先期限先出）の在庫消費ロジック**: 賞味期限の近い在庫から消費する業務ルールをドメインサービスに正しく実装。複数ロットにまたがる消費もテストで保証

---

## Problem（問題点）

- **`ship_all` のトランザクション境界設計が初期実装で不十分**: 個別トランザクションで実装し、一括出荷時に部分的に出荷される可能性があった。レビューで発見し単一トランザクション（All-or-Nothing）に修正
- **在庫消費の排他制御（悲観ロック）が初期設計で考慮されていなかった**: TOCTOU 競合のリスクをレビューで指摘され、`FOR UPDATE` による悲観ロックを追加
- **ステータスバッジ helper の DRY 化が見送り**: 発注一覧・受注一覧・出荷一覧の 3 画面でステータスバッジが重複しているが、各画面のバッジ色が異なるためハードコードのまま残した
- **E2E 統合テスト・StockForecast 連携テストが見送り**: 境界値テストで代替したが、発注→入荷→出荷のフルフローの自動テストは未実施
