## KPT 分析

### Keep（続けること）

- **新規 Django アプリの DDD レイヤー構成の確立**: purchasing アプリを IT1 の products パターンに倣い、domain/entities.py → domain/value_objects.py → domain/interfaces.py → models.py → repositories.py → services.py → views.py の順序でインサイドアウト TDD で構築した。14 ファイル・598 行を一貫した構成で実装できた
- **アプリ間連携パターンの確立**: PurchasingService が inventory の StockLotRepository を注入し、入荷時に在庫ロットを自動作成するクロスアプリ連携を実現。ドメインの境界を維持しつつアプリケーション層で協調するパターンが定着した
- **StockForecastService の後方互換拡張**: incoming_schedule パラメータをオプショナル（デフォルト None）で追加し、既存テスト 261 件を壊さずに入荷予定を在庫推移に反映した。IT3 で 0 固定だった incoming_planned が実データで動くようになった
- **ベロシティ 13SP の達成**: 平均 9.25SP に対して 40% 超の 13SP を達成。IT4 ふりかえりでリスクとして指摘した「未達の可能性が高い」を覆した。新規アプリのスキャフォールドが Phase 1 で蓄積したパターン再利用により効率化された結果
- **テスト 46 件追加（261→261+46 件の基盤構築）**: 購入ドメイン 16 件、リポジトリ 10 件、サービス 6 件、View 12 件、在庫推移拡張 2 件と、テストピラミッドのバランスが良い
- **仕入先の自動設定**: 単品マスタに紐づく仕入先を発注登録時に自動設定する UX を実装。UI 設計書 A-10 の仕様通り、ユーザーの入力負荷を軽減した

### Problem（問題点）

- **IT4 Try 項目の持ち越し**: SonarQube Quality Gate チェック（3 イテレーション連続持ち越し）、FakeRepository 準拠テスト、テストヘルパー共通化（conftest.py）が IT5 でも未対応。新規アプリの実装を優先した結果、技術的負債の返済が後回しになっている
- **View 層のリッチ化が不十分**: 発注一覧画面で item_name / supplier_name を表示するために `_enrich_orders()` ヘルパーを View 層に配置した。これはドメインエンティティに表示用データを動的に追加する ad-hoc なパターンで、ReadModel やプロジェクション層が不在のため View が太くなる傾向がある
- **品質期限アラートの実装がシンプルすぎる**: ExpiryAlertView が全単品をループして find_active_by_item_id → is_near_expiry を呼ぶ N+1 パターン。単品数が増えると性能問題が発生する可能性がある。StockLotRepository に find_near_expiry() メソッドを追加すべきだった
- **IT5 計画書の成功基準チェックボックスが未更新**: 実装は完了したが、iteration_plan-5.md の成功基準チェックボックスが `[ ]` のまま。計画書を「生きたドキュメント」として維持する規律が不足
- **コミット粒度が大きい**: IT5 全体を 1 コミットで実装した。TDD サイクルごとのコミット（Red-Green-Refactor 単位）が徹底されなかった。コミット粒度の改善が必要

### カバレッジ低下の分析

IT5 でカバレッジが 98% → 96% に低下した要因:

1. **View 層のコード増加**: purchasing の 4 画面（161 行）のうち、エラーハンドリングパスがテストでカバーされていない
2. **Repository の更新パスの一部**: ArrivalRepository の save() 更新パス（既存 ID での update）がテストされていない
3. **テストから実行されない他アプリコード**: 全体カバレッジ計算で他アプリの View コードが分母に含まれる

96% は十分高いが、下降トレンドが 2IT 連続なので注意が必要。
