## KPT 分析

### Keep（続けること）

- **レイヤードアーキテクチャの一貫性**: IT3 で導入した OrderService パターンを IT4 の全 View に適用。レビュー指摘で ORM 直接アクセスを排除し、全 View を Service 経由に統一した
- **XP マルチパースペクティブレビューの威力**: 5 エージェント（プログラマー・テスター・アーキテクト・テクニカルライター・ユーザー代表）を並列起動し、重要度「高」5 件・「中」5 件・「低」4 件の指摘を得た。特に H1（ORM 直接アクセス排除）は単独レビューでは見落としやすい設計一貫性の問題を検出できた
- **ドメインエンティティへの責務集約**: OrderStatus.label プロパティの追加により、日本語表示をドメイン層に集約。テンプレート側の `get_status_display()` への依存を排除し、ORM と切り離した
- **Phase 1 MVP の計画通り完了**: 4 イテレーション・37SP を達成率 100% で完了。ベロシティの安定（標準偏差 1.1SP）により予測可能な開発を実現
- **既存パターンの効率的な流用**: US-015（受注一覧）のフィルタパターンを US-016（注文履歴）に流用し、実装効率を上げた。3 ストーリーを 1 コミットで同時実装できた
- **テスト増分の対象が明確**: IT4 の +20 テストは全てサービス層と View 統合テスト。ドメイン層は IT1-IT3 で十分にカバーされており、新たな変更がないため妥当

### Problem（問題点）

- **View の ORM 直接アクセスが初回実装に混入**: IT4 の初回コミットで StaffOrderDetailView、OrderHistoryView、OrderHistoryDetailView、OrderCompleteView の 4 箇所が ORM Model を直接使用していた。IT3 で OrderService を導入済みにもかかわらず、新規 View でパターンを一貫できなかった
- **FakeRepository と DjangoRepository のソート基準不一致**: FakeOrderRepository は `delivery_date` 降順、DjangoOrderRepository は `created_at` 降順でソートしていた。Fake がテストで常にパスするが本番で異なる動作をするリスクがあった
- **テスト増分の減少**: IT1-IT3 は毎回 +60 以上のテストを追加したが、IT4 は +20 にとどまった。ドメイン層の新規テストが 0 件で、テスターから「ドメイン層ユニットテストの不足」を指摘された
- **IT3 Try の SonarQube・ADR が未完了**: IT3 ふりかえりの Try 項目「SonarQube Quality Gate チェック」「設計乖離 ADR 記録」が IT4 でも完了しなかった。2 イテレーション連続で持ち越し
- **カバレッジが 99% → 98% に微減**: View 層のコード増加に対してテストカバレッジが追いつかなかった。98% は十分高いが、下降トレンドの初兆候
- **レビュー保留項目の残存**: M3（テストヘルパー共通化）、M4（計画書のファイル名乖離）、L2（通貨カンマ表示）、L3（ナビゲーションのグループ分け）が未対応のまま

## Phase 2 に向けて

### 技術的な引き継ぎ

1. **仕入ドメインは新規**: Phase 2（US-008, US-009）は仕入先・入荷という新しい集約が必要。IT1 の products パターン（ドメイン→インフラ→プレゼンテーション）を踏襲する
2. **在庫引当の連動**: US-009（入荷受入）で StockLot の actual_quantity を更新する際、StockForecastService との連携が必要
3. **FakeRepository の準拠テスト**: IT4 で判明した Fake/Django の乖離問題を、IT5 の新規 Repository で再発させない

### ベロシティ見込み

- IT5 計画: 13 SP（US-008: 5SP + US-009: 5SP + US-012: 3SP）
- 平均ベロシティ: 9.25 SP → 13SP は 40% 超過
- **リスク**: IT3（11SP）が最高実績。13SP は未達の可能性が高い
- **緩和策**: US-012（品質期限アラート 3SP）を IT6 に移動してベロシティに収める選択肢を検討
