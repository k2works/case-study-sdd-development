# イテレーション 4 ふりかえり

## 対象

| 項目 | 内容 |
|------|------|
| イテレーション | IT4 |
| 期間 | 2026-05-04 から 2026-05-15 まで |
| 対象ストーリー | `US-06`, `US-09`, `US-09B` |
| 実績 SP | 9 / 9 |

## Keep

- `US-06`、`US-09`、`US-09B` を同一イテレーションで完了し、入荷から出荷準備完了までの現場オペレーションを一連で成立できた。
- `purchase-order`、`inventory`、`shipping` を跨ぐ統合観点を Backend テストで固定でき、在庫不足解消から結束完了までの状態遷移を安全に変更できる形にできた。
- Frontend の管理画面に `入荷管理` と `出荷管理` を追加し、既存の仕入ワークベンチ構成を崩さずに業務画面を拡張できた。

## Problem

- `release_plan.md` と GitHub Project 側の `IT4` 状態更新は実装完了と同時に反映されず、終了処理が後段にずれた。
- `US-10` と `US-00` を `IT5` に送ったため、 Phase 2 全体としてはまだ `Release 1.1` の完了条件を満たしていない。
- 実行確認は通ったが、入荷登録から出荷確定までの end-to-end 業務シナリオは引き続き手動確認中心で、自動化余地が残っている。

## Try

| アクション | 担当 | 期限 | 期待効果 |
|-----------|------|------|----------|
| `IT5` 開始前に `US-10` の状態遷移と `Release 1.1` 完了条件を明文化する | - | IT5 Day 1 | 出荷確定の実装ずれを防ぐ |
| イテレーション終了時は `iteration_plan` 更新と同日に `release_plan`、 `index`、 GitHub Project をまとめて反映する | - | IT5 終了時 | 計画書と運用状態の同期遅れをなくす |
| 入荷から出荷確定までの主要業務シナリオを E2E または API シナリオとして 1 本追加する | - | IT5 Day 4 | Phase 2 完了時の回帰検知力を高める |
