## Keep（続けたいこと）

1. **TDD サイクルの徹底** — 全イテレーションで Red-Green-Refactor を実践し、高品質を維持できた
2. **バッファイテレーションの活用** — 品質保証タスクを専用イテレーションで集中処理する戦略が有効だった
3. **テストカバレッジ目標の設定** — 80% 目標により、テスト不足を早期に検出できた
4. **ArchUnit によるアーキテクチャ保護** — レイヤー違反を自動検出する仕組みが有効

## Problem（問題だったこと）

1. **IT7 から IT8 へのタスク移動** — 実装イテレーションに品質保証タスクを含めると消化しきれない。計画時にバッファを想定すべき
2. **E2E テストの実行環境** — アプリケーション全体の起動が必要で、CI/CD パイプラインの整備が今後の課題
3. **Clock 注入の範囲判断** — 全 LocalDateTime.now()/LocalDate.now() を対象にすべきか、スコープ外を許容するか基準が曖昧だった

## Try（次に試したいこと）

1. **E2E テストの CI 統合** — Docker Compose で全スタックを起動し、CI パイプラインで E2E テストを自動実行
2. **品質保証タスクの各イテレーションへの分散** — バッファイテレーションに集中させず、各 IT で少しずつ実施
3. **テストカバレッジのブランチカバレッジ改善** — 76.9% を 80% 以上に引き上げ

## プロジェクト全体のふりかえり

### 成功要因

- **段階的リリース戦略** — Phase 1-3 の段階的リリースにより、各フェーズで安定したデリバリーを実現
- **ヘキサゴナルアーキテクチャ** — ドメイン層の独立性が保たれ、機能追加が容易だった
- **AI アシスタントの活用** — コード生成・テスト作成・リファクタリングが加速された

### 改善点

- **初期のベロシティ推定精度** — 初回推定 12-16 SP に対し実績平均 11.9 SP。概ね正確だが、イテレーションごとの振れ（8-16 SP）が大きい
- **レビュー工程の省略** — 1 人開発のため peer review がなく、AI レビューで代替したが、本番運用では人間レビューが必要
