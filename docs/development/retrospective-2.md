# イテレーション 2 ふりかえり

## 対象

| 項目 | 内容 |
|------|------|
| イテレーション | IT2 |
| 期間 | 2026-04-06 から 2026-04-17 まで |
| 対象ストーリー | `US-02`, `US-03`, `US-04` |
| 実績 SP | 11 / 11 |

## Keep

- `US-02`、`US-03`、`US-04` を縦切りで完了し、 MVP の主要導線を 1 イテレーションで成立できた。
- Backend / Frontend / E2E のテスト実行環境を復旧し、以後の回帰確認を自動テストで維持できる状態に戻せた。
- 開発レビューと UI/UX レビューの指摘を同イテレーション内で吸収し、出荷日算出、期間入力、通信失敗、管理画面の業務導線を改善できた。

## Problem

- `release_plan.md` と `docs/development/index.md` への進捗反映が開発完了より後ろ倒しになり、計画と実績の整合が一時的に崩れた。
- 管理画面は `US-03` と `US-04` の初回実装時点で、役割分離、一覧 + 詳細の同時可視、失敗時の導線が不足していた。
- Backend の開発サーバー実行では DI が不安定になり、`tsx watch` 前提の起動確認まで含めないと検知できない差分があった。

## Try

| アクション | 担当 | 期限 | 期待効果 |
|-----------|------|------|----------|
| イテレーション終了日の作業項目に `release_plan.md` と `docs/development/index.md` の更新を固定で入れる | - | IT3 終了時 | 進捗サマリーの遅延を防ぐ |
| 管理画面系のストーリーでは、受け入れテストに `通信失敗` と `業務切替` の観点を最初から含める | - | IT3 Day 2 | UI/UX の後戻りを減らす |
| Backend の `npm run dev` 確認を Definition of Done に追加し、テスト通過だけでなく watch 実行時の起動も確認する | - | IT3 Day 5 | 実行環境差分による障害を早期に検知する |
