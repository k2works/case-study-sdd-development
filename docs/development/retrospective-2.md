## KPT 分析

### Keep（続けること）

- **TDD Inside-out アプローチの徹底**: ドメイン層 → インフラ層 → プレゼンテーション層の順序で、テストファーストを維持できた
- **マルチパースペクティブレビュー**: 5+2 エージェントの並列レビューで致命的バグ（DeliveryDate 復元バグ）を早期発見できた
- **Conventional Commits の遵守**: コミット単位が機能ごとに明確で、変更追跡が容易
- **Django Generic View の活用**: ListView/DetailView で簡潔な実装を実現
- **値オブジェクトの自己バリデーション**: 不正な状態のオブジェクトが生成できない設計を orders にも適用

### Problem（問題点）

- **Application 層の欠落**: View にビジネスロジックが集中し、4 層アーキテクチャの「アプリケーション層」が未実装。API 追加時にコピペが必要になる構造
- **DeliveryDate の DB 復元バグ**: バリデーションが新規作成時と DB 復元時を区別していなかった。レビューなしでは見逃していた可能性が高い
- **商品詳細画面の注文ボタン忘れ**: 購買フローの断絶という致命的な UI 欠陥をレビューで初めて発見。実装時にユーザーフロー全体を通して確認する習慣が不足
- **注文確認画面の省略**: UI 設計ドキュメントの C-07 を省略したが、ADR に記録していない
- **設計ドキュメントとの乖離**: 受注集約の実装が設計ドキュメントと複数箇所で不一致（ステータス値、メソッド名等）

### Try（次に試すこと）

| アクション | 担当 | 期限 | 期待効果 |
| :--- | :--- | :--- | :--- |
| Application 層（OrderService）の導入 | 開発 | IT3 | View の責務を軽減し、API 追加時の再利用を可能にする |
| ファクトリメソッドパターンの標準化 | 開発 | IT3 | DB 復元時のバリデーションスキップを一貫した方法で実装 |
| 実装後のユーザーフローウォークスルー | 開発 | IT3 以降 | UI の導線漏れを実装段階で検出 |
| 設計ドキュメントとの差分を ADR に記録 | 開発 | IT3 | 意図的な省略と見落としを区別可能にする |
