## Keep（続けること）

- **TDD インサイドアウト開発の継続**: IT1 に引き続き、ドメイン層 → アプリケーション層 → インフラ層 → API 層の順序で TDD 開発を実施した。Product 集約のテスト（21 件）が設計を駆動し、ProductComposition の追加・削除ロジックがテストで仕様化された。IT2 でバックエンド +46 件のテストが追加され、累計 127 件に到達した。

- **マルチパースペクティブレビューの高い投資対効果**: 5 つの XP パースペクティブ（programmer, tester, architect, writer, user-rep）による並列レビューが、CatalogController の非公開商品表示バグ、CompositionResponse の itemName 欠落、Item 存在チェック欠如、価格バリデーション不一致など高優先度 8 件を発見した。すべてリリース前に修正完了。レビュー投資は品質に対して非常に有効。

- **ヘキサゴナルアーキテクチャの一貫適用**: IT1 で確立した Port/Adapter パターンを Product 集約でも正確に再現できた。ArchUnit テストによる依存方向の自動検証が継続的に機能し、アーキテクチャの劣化を防止した。IT1 でのパターン確立への投資が IT2 で回収された。

- **IT1 レビュー指摘の着実な改善**: IT1 ふりかえりで引き継いだ M-2（物理削除問題）に対して Product に is_active パターンを導入し、L-3（登録成功時のフィードバック）も対応した。イテレーション間の改善サイクルが機能している。

---

## Problem（問題点）

- **レビュー時に重要バグが発見された（TDD テスト設計の網羅性不足）**: CatalogController が非アクティブ商品を返す設計バグ、価格バリデーション不一致（実装: 0 以上、設計書: 1〜999,999）など、TDD で防げるはずのバグが 4 件漏れた。テスト設計時に「異常系」「セキュリティ（認可）」「設計書との整合性」の観点が不足していた。

- **フロントエンドテストカバレッジの穴**: 5 ページ中 3 ページ（ProductListPage, ProductFormPage, ProductCompositionPage）のみテスト作成。ProductDetailPage と ProductCatalogPage のテストが未作成のままリリースされた。フロントエンドのテスト作成がページ実装の「おまけ」扱いになっている。

- **IT1 Try の 2 件が未達成**: 「テストカバレッジ CI 自動確認」と「モバイルファーストのレスポンシブ対応」の 2 つの Try が IT2 でも未実施。Try が具体的なタスクとしてイテレーション計画に組み込まれておらず、実行されない構造になっていた。

- **中優先度レビュー指摘の積み残し**: `@Transactional` 未付与、OpenAPI アノテーション欠落、`formatPrice` 関数の 3 箇所重複、`ProductFormPage` での `useEffect` 直接 API 呼び出し（useQuery 未使用）など 13 件の技術的負債が蓄積した。

- **設計書と実装の乖離**: 価格バリデーション（設計書: 1-999,999、実装: 0 以上）、値オブジェクト未導入（設計書で定義済みの ProductName, Price が未実装）など、設計書をリファレンスとして活用できていない。

---

## Try（次に試すこと）

- **テスト設計チェックリストの導入**: 各 US のタスク完了前に「正常系 / 異常系 / 境界値 / セキュリティ（認可）/ 設計書整合性」の 5 観点でテストケースを列挙するチェックリストを作成する。特に API 層のテストでは「認可されていないロールでのアクセス拒否」を必須テストとする。**IT3 Day 1 に導入。**

- **IT1 未達成 Try の再計画と明示的タスク化**: Try を「やりたいこと」で終わらせず、イテレーション計画のタスクに SP 外として明示的に組み込む。
  - テストカバレッジ CI 自動確認: **IT3 技術タスク（SP 外・1h）** として計画
  - レスポンシブ対応: **IT4 以降の技術タスク** として backlog に追加

- **中優先度レビュー指摘の計画的消化**: IT3 の技術タスク（SP 外）として以下を優先的に組み込む:
  - `@Transactional` 付与（全 UseCase クラス）
  - `formatPrice` の共通ユーティリティ化
  - `ProductFormPage` の useQuery 統一

- **設計書整合性チェックを Definition of Done に追加**: 各 US の完了条件に「設計書（ドメインモデル設計、データモデル設計）との整合性確認」を明示的に追加する。

- **フロントエンドテスト必須化**: 新規作成する全ページについてコンポーネントテストを必須とし、ページ実装とテスト作成を同一タスクとして見積もる。IT2 で未作成の ProductDetailPage・ProductCatalogPage のテストは IT3 技術タスクで追加する。
