## KPT 分析

### Keep（続けること）

- **TDD インサイドアウトの徹底**: ドメイン層 → インフラ層 → プレゼンテーション層の順序で、新規 inventory アプリでもパターンを踏襲できた
- **Application 層パターンの導入**: OrderService・InventoryService により View の責務が明確化。IT2 ふりかえり Try を IT3 で確実に対応
- **ベロシティ超過達成**: 平均 8.5SP に対して 11SP を達成。設計済みのドメインモデルが効率的な実装を可能にした
- **既存パターンの踏襲**: products アプリの Repository・Admin パターンを inventory に適用し、一貫性を維持
- **ファクトリメソッド標準化**: `reconstruct()` パターンを DeliveryDate に適用し、DB 復元時のバリデーションスキップを安全に実装
- **ADR の記録**: OrderStatus の命名・設計乖離を ADR-001 に記録し、意図的な差異を文書化
- **テストの大幅増加**: +65 テスト（130→195）で回帰テストの安全網が強化

### Problem（問題点）

- **tox 未実行**: IT3 では `uv run pytest` + `ruff check` のみで tox を実行していない。成功基準に `uv run tox` 全パスを掲げていたが未確認
- **SonarQube 未確認**: Quality Gate OK を成功基準に含めていたが未実行
- **テンプレートパスの設計乖離**: 設計では `templates/inventory/stock_forecast.html` だが実装は `templates/shop/stock_forecast.html` に配置。shop ベーステンプレートを共有するための判断だが未記録
- **URL パスの設計乖離**: 設計では `/staff/inventory/forecast/` だが実装は `/inventory/forecast/` に配置。スタッフ認証が未実装のため `/staff/` プレフィックスを省略したが未記録
- **キャンセル画面の設計乖離**: 設計では注文番号 URL アクセス（`/shop/orders/{order_number}/cancel/`）だが、実装は注文番号検索フォーム方式（`/shop/order/cancel/`）に変更。ユーザビリティを考慮した判断だが ADR 未記録
