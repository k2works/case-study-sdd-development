# イテレーション 3 ふりかえり

## 対象

| 項目 | 内容 |
|------|------|
| イテレーション | IT3 |
| 期間 | 2026-04-20 から 2026-05-01 まで |
| 対象ストーリー | `US-00B`, `US-05` |
| 実績 SP | 8 / 8 |

## Keep

- `US-00B` と `US-05` を同一イテレーション内で完了し、花材更新から発注確定までの業務導線を 1 本で成立できた。
- 開発レビューと UI/UX レビューの指摘を同イテレーション内で吸収し、 `materialId` ベースの結合、数量算出、通信失敗時の保護、在庫から発注への CTA を改善できた。
- `npm run dev` による実行確認を完了条件へ含めたことで、テスト通過だけでは見えない CORS と開発サーバー起動差分を検知できた。

## Problem

- GitHub Project 側では `US-00` 以降の未完了ストーリーまで `Done` へ倒れており、計画ドキュメントとの同期を後追いで補正する必要があった。
- IT3 の主対象は仕入スタッフ向けだったが、初期表示や Loading / Retry の導線は最初の実装時点で十分ではなく、 UI/UX レビュー後に補強が必要だった。
- 発注候補の数量ロジックと花材マスタの識別子伝播は、レビューが入るまで受け入れ基準をぎりぎり満たす最小実装に寄っていた。

## Try

| アクション | 担当 | 期限 | 期待効果 |
|-----------|------|------|----------|
| イテレーション終了時は `release_plan.md` と GitHub Project の両方を同日に更新し、 `Done` / `Todo` のずれを残さない | - | IT4 終了時 | 計画と実行の整合性を維持する |
| 業務画面系のストーリーでは、初回の受け入れテストに `Loading`、 `Retry`、 `通信断` の観点を含める | - | IT4 Day 2 | UI/UX レビュー後の手戻りを減らす |
| 識別子を跨ぐ集約ロジックでは、表示名ではなく ID を保持する結合テストを最初から追加する | - | IT4 Day 3 | 下流機能の不整合を早期に検知する |
