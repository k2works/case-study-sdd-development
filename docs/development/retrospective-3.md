## Keep（継続すること）

- **TDD によるサービス層の段階的実装**: StockForecastService を Red → Green → Refactor で段階的に構築。良品在庫 → 入荷予定 → 引当済み → 廃棄対象 → 有効在庫と、1 つずつロジックを積み上げた
- **高いテストカバレッジ**: 94.01%（IT1: 87.29% → IT2: 90.76% → IT3: 94.01%）と継続的に向上
- **マルチパースペクティブレビューの有効活用**: コードレビュー（xp-programmer, tester, architect, technical-writer, user-representative）と UI/UX レビュー（interaction-designer, user-representative）の 2 段階レビューで、二重減算バグやアクセシビリティ問題を検出
- **品質維持日数に基づく廃棄ロジック**: ドメイン知識を正しくモデル化し、業務要件を満たす計算ロジックを実装

---

## Problem（問題点）

- **effective_stock の二重減算バグ**: 有効在庫の計算で廃棄対象を二重に減算する初期実装ミス。コードレビューで発見・修正済みだが、計算式が複雑な箇所は初期段階でバグが混入しやすい
- **空状態・エラー状態の考慮漏れ**: 単品未選択時のガイダンスメッセージや不正日付入力のエラーハンドリングが初期実装で欠如。UI/UX レビューで指摘
- **他画面とのデザイン不整合**: container-fluid と container の混在、card パターンの不統一など、画面間の一貫性が不足。共通テンプレートパターンが未確立

---

## Try（次に試すこと）

| アクション | 期待効果 |
|-----------|---------|
| 複雑な計算ロジックは境界値テストを重点的に書く | 二重減算のようなバグの早期発見 |
| UI 実装時に空状態・Loading・Error の 3 状態を最初から考慮する | レビュー指摘の事前防止 |
| 管理画面の共通テンプレートパターン（card + table-striped + table-dark ヘッダー）を確立し横展開する | 画面間の一貫性確保 |
| IT2 Try の OrderService 導入は IT4 以降で検討 | Fat Controller の段階的解消 |
