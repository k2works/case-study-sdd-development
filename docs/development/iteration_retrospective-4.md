## KPT 分析

### Keep（継続すべきこと）

**技術的成功事項**:

- **InventoryQueryPort の抽象化**: 在庫推移計算で複数集約（Stock, PurchaseOrder, Order）にまたがるクエリをポートで抽象化した設計が、テスタビリティと関心分離の両面で有効だった
- **Clock 注入パターンの採用**: IT3 の TZ バグの教訓を活かし、InventoryTransitionService に Clock を DI。テスト時の日付制御が容易になり、境界日のテストが確実に書ける
- **レビュー指摘の即時対応**: 計画レビュー→計画修正→開発→開発レビュー→修正→UI/UX レビュー→修正 のサイクルが機能し、品質の高い成果物を段階的に仕上げられた
- **US-011 の IT5 移動判断**: ベロシティ超過リスクを計画段階で検知し、入荷登録を IT5 に移動。結果的に 13SP を余裕を持って消化できた

**プロセス的成功事項**:

- **マルチパースペクティブレビューの定着**: 5 エージェント（プログラマー、テスター、アーキテクト、テクニカルライター、ユーザー代表）+ 2 エージェント（デザイナー、ユーザー代表）の並列レビューが品質向上に大きく貢献
- **設計ドキュメントの先行更新**: IT4 計画にドメインモデル図、データモデル、API 設計、UI ワイヤーフレームを含めたことで、実装時の迷いが大幅に減少
- **シードデータの充実**: 在庫・発注のシードデータを作成したことで、画面の動作確認がスムーズに

### Problem（問題点）

**技術的課題**:

- **JpaInventoryQueryPort の入荷予定・受注引当が常に 0**: 在庫推移計算の核心部分が未完成のまま。UI に「未反映」注意バナーで対処したが、仕入スタッフへの提供価値が制限されている
- **PurchaseOrderPage の state 非同期バグ**: React の `setQuantity` 後に `mutate` を呼ぶと古い値が送信される問題。レビューで発見し修正したが、TDD で防げるべきバグだった
- **StockStatus.DEGRADED の放置**: ドメインモデルに定義されているが実装での使用箇所がない。仕様が不明確なまま残存

**プロセス的課題**:

- **曜日表示・発注確認ダイアログが IT3 から 4 回指摘**: レビューで指摘されても次のイテレーションに先送りされ、同じ指摘が繰り返される構造的問題。最終的に IT4 の UI/UX 実装レビューで対応したが、もっと早く解消すべきだった
- **設計ドキュメントの追従遅れ**: domain_model.md と data-model.md の supplierId → supplierName の更新がレビュー指摘まで放置されていた。実装が先行してドキュメントが追いつかないパターン
- **ベロシティ見積もりの精度**: 計画時 16SP → レビューで 13SP に調整。SP 外タスク（15.5h）の影響を過小評価していた

### Try（次に試すこと）

| アクション | 担当 | 期限 | 期待効果 |
|-----------|------|------|---------|
| レビュー指摘で「高」判定されたものは同イテレーション内で即座に修正するルールを制定 | チーム | IT5 開始時 | 同じ指摘の繰り返しを防止（曜日表示・確認ダイアログ問題の再発防止） |
| 実装後にドメインモデル・データモデル設計ドキュメントの差分チェックをタスクに組み込む | チーム | IT5 | 設計と実装の乖離を早期検知 |
| InventoryQueryPort の入荷予定・受注引当を IT5 の冒頭タスクで実装 | - | IT5 Day 1-2 | 在庫推移の信頼性を確保し、「未反映」バナーを除去 |
| フロントエンドのユニットテスト比率を改善（現在 E2E > ユニットのアイスクリームコーン傾向） | - | IT5 | フィードバックループの高速化 |
| SP 外タスクの工数を計画時にベロシティから差し引いて実効 SP を計算する | - | IT5 計画時 | ベロシティ超過リスクの早期検知 |

## 次のイテレーションへの申し送り

### IT5 で対応すべき事項

1. **US-011 入荷登録**（3SP）: IT4 から移動。Arrival ドメインエンティティ + RegisterArrivalUseCase + 入荷画面
2. **InventoryQueryPort の実装完成**: 入荷予定（PurchaseOrder から推定）と受注引当（Order から計算）の実装。在庫推移の「未反映」バナー除去
3. **StockStatus.DEGRADED の方針決定**: ADR で使用可否を明確化
4. **フロントエンドテストの拡充**: ユニットテスト比率の改善
