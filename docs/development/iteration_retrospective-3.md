## Keep（続けること）

1. **TDD サイクルの徹底**: ドメインモデル→ユースケース→インフラ→API の順で境界値テストまで含めて TDD で進めたことで、バグの早期発見と設計の安全性を確保できた
2. **ヘキサゴナルアーキテクチャの一貫性**: IT1-IT2 で確立したパターン（Entity→UseCase→Repository→Controller）をそのまま踏襲し、開発速度が向上した
3. **UI/UX レビュー指摘の事前反映**: IT3 計画段階でレビュー指摘を取り込んだことで、二重送信防止やスケルトンローディングなどの品質向上策を最初から実装できた
4. **ロール別アクセス制御**: SecurityConfig でのロールベース認可と、フロントエンドのロール別メニュー表示を同時に実装し、セキュリティの一貫性を確保した

## Problem（問題点）

1. **フロントエンドテストの未追加**: IT3 ではバックエンドの TDD を徹底したが、フロントエンドのコンポーネントテストは追加できていない。IT4 以降でフロントエンドテストも追加する
2. **E2E テスト**: 注文→受注一覧→受付のフル E2E テストは時間の制約から未実施。IT4 で追加する

## Try（試すこと）

2. **E2E テスト**: 注文フローの E2E テストを Playwright で実装する
3. **商品画面の改善**: IT3 のレビュー指摘で残っている商品画像プレースホルダー（UI/UX-H-1）と 2 カラムレイアウト（UI/UX-H-5）の対応
