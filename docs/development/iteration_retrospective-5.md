## KPT 分析

### Keep（継続すべきこと）

**技術的成功事項**:

- **IT4 技術負債の冒頭解消**: IT4 ふりかえりの Try で「InventoryQueryPort の入荷予定・受注引当を IT5 冒頭で実装」と決めた通り Day 1-2 で完了。在庫推移画面の「未反映」バナーを除去でき、仕入スタッフへの提供価値が向上した
- **TDD で 3 つのドメイン機能を安全に追加**: Arrival エンティティ、Stock.consume（FIFO）、Order.prepare() をすべて TDD で実装。レビュー指摘（H-1〜H-4, M-1〜M-6）を計画段階で反映し、実装後の手戻りが最小限だった
- **Clock 注入パターンの継続適用**: IT3 の教訓・IT4 の成功を踏まえ、BundlingQueryService にも Clock を DI。出荷日計算のテストが確実に書けた
- **ヘキサゴナルアーキテクチャの一貫性**: 新しい結束（Bundling）機能を application/bundling パッケージに配置し、ドメイン層に依存しない形で設計。ArchUnit テストで自動検証

**プロセス的成功事項**:

- **レビュー指摘の即時対応ルールが機能**: IT4 ふりかえりで制定した「高判定は同イテレーション内で修正」ルールを遵守。IT5 開発レビューの高優先度指摘を即座に修正コミット（87cfcab）
- **設計ドキュメントの差分チェックタスク化**: タスク 0.4 で domain_model.md / data-model.md / architecture_backend.md の IT4 実装差分を明示的にチェック。設計と実装の乖離を防止
- **シードデータの整合性改善**: IT5 完了後にシードデータの不整合（PURCHASE_STAFF ユーザー不在、発注データ不足）を発見・修正。2 件 → 8 件に発注を拡充し、受注商品の全構成単品をカバー

### Problem（問題点）

**技術的課題**:

- **BundlingQueryService.getTargets が 56 行**: Checkstyle の 50 行制限を超過。コミット時の pre-commit hook で発覚し、メソッド分割で対応したが、開発中に気づくべきだった
- **BundleOrderUseCaseTest の未使用インポート**: anyInt, anyList のインポートが残存。IDE の警告を見逃していた
- **StockStatus.DEGRADED の未解決**: IT4 ふりかえりで「ADR で方針決定」とした が、IT5 でも着手できず。技術負債として残存
- **シードデータの PURCHASE_STAFF ユーザー不在**: 発注管理 API が OWNER or PURCHASE_STAFF ロールを要求するが、シードデータに PURCHASE_STAFF ユーザーがなかった。OWNER ロールでは動作するが、ロールベースのテストが不十分

**プロセス的課題**:

- **初期化順序の潜在バグ**: 受注データが在庫・発注より先に作成されていた。受注時に在庫チェックがないため動作はするが、ビジネスロジック上は不自然な順序
- **フロントエンドテストのアイスクリームコーン傾向**: IT4 ふりかえりで指摘した E2E > ユニットの問題は改善傾向にあるが、完全には解消していない

### Try（次に試すこと）

| アクション | 担当 | 期限 | 期待効果 |
|-----------|------|------|---------|
| Checkstyle を IDE/エディタで即時フィードバックする設定を全員に展開 | チーム | IT6 開始時 | コミット前にメソッド長や未使用インポートを検知 |
| StockStatus.DEGRADED の方針を ADR で決定（使うか削除するか） | - | IT6 Week 1 | 不要なドメインモデル要素を排除し、保守性向上 |
| IT6 計画時に SP 外タスクの工数見積もりを明示し、実効ベロシティを計算 | - | IT6 計画時 | 過剰コミットの防止 |
| フロントエンドのユニットテスト比率 KPI を設定（ユニット:E2E = 3:1 目標） | - | IT6 | テストピラミッドの健全化 |
