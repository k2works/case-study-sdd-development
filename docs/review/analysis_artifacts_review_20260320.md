### 総合評価
全体として、花束配送 WEB ショップの業務像から、モジュラーモノリス、Next.js 単一アプリ、PostgreSQL 中心の永続化までを一貫した粒度で定義できています。特に `orders` / `stock` / `purchasing` / `shipping` を中心にした分割は小規模チーム向けとして自然ですが、在庫整合性、認証 / 認可、届け先スナップショット、コンテキスト所有権など、後工程で手戻りになりやすい境界の定義が未完了です。

### 改善提案（重要度順）

#### 高
| # | 提案 | 指摘元 | 理由 |
|---|------|--------|------|
| 1 | 顧客の届け先再利用 / 届け日変更を成立させる認証・本人確認方式を早期に確定する | xp-architect | `GET /api/customers/{customerId}/destinations` や注文後の届け日変更導線は、顧客アカウント、ゲスト注文、ワンタイム URL などの方式が未確定だと API 契約、UI 遷移、権限制御が決められません。 |
| 2 | `orders` に届け先スナップショットを保持する方針へ修正する | xp-architect | 現在の `destination_id` 参照だけでは、届け先マスタ更新で過去注文の配送先が変わり得ます。履歴保全と配送事故防止のため、注文時点の配送先情報を正本として保持すべきです。 |
| 3 | `stock projection` 更新時の同時実行制御、再計算単位、失敗時補償を定義する | xp-architect | 注文、届け日変更、発注、入荷、出荷が同一 projection を更新するため、同期更新だけでは二重引当や整合性崩れのリスクがあります。トランザクション境界、ロック方針、再計算ジョブを決める必要があります。 |
| 4 | `supplier` の所属コンテキストを `catalog` か `purchasing` に明確化し、全ドキュメントで統一する | xp-architect | バックエンド設計では `purchasing` に仕入先管理があり、ドメイン / データモデルでは `catalog` 側に含まれています。所有権が曖昧だと変更責務と依存方向が崩れます。 |
| 5 | 非機能要件を測定可能な形で追加する | xp-architect | 現状は構成方針はあるものの、性能目標、在庫判定応答時間、RPO / RTO、可用性、バッチ完了 SLA が未定義で、実現可能性を評価できません。 |

#### 中
| # | 提案 | 指摘元 | 理由 |
|---|------|--------|------|
| 1 | 出荷業務のスコープを「対象抽出のみ」か「出荷確定まで含む」か明文化する | xp-architect | 要件・ UI は主に出荷一覧確認ですが、状態モデルとデータモデルは `出荷済み` まで含んでいます。実装対象の認識差を防ぐためです。 |
| 2 | UI / フロントエンド設計に、注文確認画面・注文完了画面・届け先選択画面の実装マッピングを追加する | xp-architect | UI 設計には存在するが、フロントエンドのルート / Feature 設計には一部抜けがあります。実装計画と画面一覧のトレーサビリティを高められます。 |
| 3 | 在庫 projection に対する検証シナリオを、同日多件注文・部分入荷・届け日変更競合まで具体化する | xp-architect | バックエンド文書ではストレスポイントと認識できていますが、テスト観点の粒度はまだ粗いです。設計段階で代表ケースを固定すると後の不具合を防げます。 |
| 4 | Next.js の Route Handler を使う場合の API 認証連携と same-origin 方針を追加する | xp-architect | 専用 BFF は作らない方針ですが、中継層を置く以上、Cookie / token の受け渡し設計を決めないと実装時にブレます。 |
| 5 | 主要なアーキテクチャ判断を ADR 候補ではなく正式 ADR に昇格させる | xp-architect | モジュラーモノリス、限定的 CQRS、単一 Next.js アプリ、同期 projection 更新開始は後で揺れやすい重要判断です。 |

#### 低
| # | 提案 | 指摘元 | 理由 |
|---|------|--------|------|
| 1 | `items.supplier_id` の単一仕入先前提が将来制約になるかを確認する | xp-architect | 現段階では妥当ですが、代替仕入先運用が入ると再設計が必要です。 |
| 2 | 出荷一覧 / 発注一覧の印刷・ CSV 出力の優先度を要件側に反映する | xp-architect | UI には表現がありますが、要件定義では明示度が低く、後でスコープ外扱いされる可能性があります。 |

### 矛盾事項

| # | 視点 A | 視点 B | 論点 | 推奨判断 |

| 1 | バックエンド設計: `purchasing` が仕入先管理を担当 | ドメイン / データモデル: `catalog` に仕入先を含める | 仕入先の所有コンテキスト | 発注変更と一緒に変わるなら `purchasing`、商品構成と一体なら `catalog`。現状は発注責務との近さから `purchasing` 寄りが自然です。 |
