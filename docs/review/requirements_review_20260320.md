# 要件定義成果物レビュー

**レビュー日**: 2026-03-20

**レビュー対象**: 要件定義書（RDRA 2.0）、ビジネスユースケース、システムユースケース、ユーザーストーリー

## レビュー対象

- docs/requirements/ 配下 4 ドキュメント

## 総合評価

RDRA 2.0 の4層構造に沿った要件定義は網羅的かつ構造的に整理されており、ビジネスドメインの理解度が高い。一方、ユーザーストーリーにおいて見積もり・ビジネス価値の明示がなく、フェーズ分けの優先順位にビジネス価値の観点から疑問がある箇所がいくつか存在する。

---

RDRA 2.0 の4層構造に忠実に従い、ビジネスユースケースからシステムユースケース、ユーザーストーリーまでのトレーサビリティが明確に確保された高品質な要件定義です。一方で、在庫推移計算の技術的複雑度に対する深掘りと、MVP フェーズにおけるアーキテクチャリスクの特定がやや不足しています。

全体として、RDRA 2.0 の4層構造に沿った要件定義が丁寧に行われており、ユーザーストーリーのトレーサビリティも確保されています。しかし、ユーザーの心理モデルとの整合や画面遷移の自然さにおいて、「得意先が自分の注文状況を確認・操作できない」という重大な体験の欠落があり、ユーザー体験全体としてはまだ改善の余地があります。

---

全体として RDRA 2.0 の 4 層構造に沿った構成は明確で、トレーサビリティも確保されています。ただし、テスト可能性の観点では、受け入れ基準の定量性が不十分な箇所、状態遷移のエッジケースの未定義、在庫推移計算ルールの検証条件の曖昧さなど、テスト設計時に「仕様を確認しに戻る」必要が生じる箇所が複数見受けられます。

---

花屋の業務フローをよく理解した上で要件がまとめられており、特に「品質維持日数を考慮した在庫推移の可視化」や「発注判断は人間が行い、システムは判断材料を提供する」という設計思想は、現場の実態に即しています。ただし、得意先（お客様）の日常的な注文体験と、スタッフの繁忙期の業務フローにおいて、いくつか見落とされている重要なシナリオがあります。

**プロダクトマネージャーの視点**

### 評価サマリー

RDRA 2.0 の4層構造に沿った要件定義は網羅的かつ構造的に整理されており、ビジネスドメインの理解度が高い。一方、ユーザーストーリーにおいて見積もり・ビジネス価値の明示がなく、フェーズ分けの優先順位にビジネス価値の観点から疑問がある箇所がいくつか存在する。

---

### 良い点

- **RDRA 2.0 の4層構造が忠実に実装されている。** システム価値 → 外部環境 → 境界 → システムの各層が明確に分離されており、トレーサビリティが確保されている。
- **ビジネスルールが具体的かつ計算式で定義されている。** 在庫推移計算ルール（`日別在庫予定数 = 前日在庫 + 当日入荷予定 - 当日受注引当 - 当日廃棄予定`）は開発チームが迷わず実装できる水準にある。
- **境界値が明確に定義されている。** 届け日の範囲、メッセージ文字数、発注数量の上限など、実装時の曖昧さを排除している。
- **状態遷移が状態図と状態遷移表の両方で定義されている。** テストケース設計の基盤として十分な情報がある。
- **「1注文 = 1商品ルール」が明記されている。** 複雑さを意図的に制限しており、MVP として適切なスコープ管理がされている。
- **トレーサビリティマトリクスが完備されている。** BUC → UC → US の対応関係が明確で、要件の抜け漏れチェックが容易。
- **利用シーンが具体的。** 「初めての注文」「リピート注文」「日次の在庫確認」「緊急の追加発注」など、実際の業務シナリオに基づいている。
- **在庫引当のタイミングが UC-007 の補足で明確化されている。** 論理引当と物理消費の区別は、この種のシステムで最も曖昧になりやすい箇所であり、事前に定義している点は優れている。

---

### 改善提案

- **【重要度: 高】ユーザーストーリーに見積もりとビジネス価値の記載がない。** 現状のストーリーには見積もり（相対ポイント）もビジネス価値（高/中/低）の評価もない。フェーズ分けの根拠が不透明であり、イテレーション計画時にチームが優先順位の判断に困る。各ストーリーにビジネス価値と相対的な見積もりを追加すべきである。

- **【重要度: 高】MVP フェーズの優先順位に疑問がある。** 経営者の最大の課題は「廃棄ロスの削減」と「受注キャパシティの拡大」である。にもかかわらず、在庫推移表示（US-009）を MVP に含めつつ、出荷管理（US-012, US-013, US-014）を Phase 2 に回している。受注から出荷までの一連のフローが完結しなければ、MVP としてエンドツーエンドの価値を提供できない。出荷管理の基本機能は MVP に含めるべきである。代わりに、届け日変更（US-008）やキャンセル（US-019）を Phase 2 に残すのは妥当。

- **【重要度: 高】得意先が自らキャンセルできるストーリーがない。** 現在のキャンセル（US-019）は受注スタッフ経由のみである。WEB ショップのユーザー体験として、得意先自身が一定条件下でキャンセルできるストーリーが必要かどうか、ビジネス判断を明確にすべき。意図的に受注スタッフ経由に限定しているなら、その理由をドキュメントに記載すべきである。

- **【重要度: 中】「届け完了」への遷移を誰がどのように行うかが曖昧。** 受注ステータスに「届け完了」があるが、対応するユースケースやストーリーが存在しない。配送スタッフが行うのか、システムが自動で行うのか、それとも得意先の受取確認をトリガーとするのか。UC-008 の後続処理として定義が必要である。

- **【重要度: 中】ユーザーストーリーが INVEST 基準の「Independent（独立）」を一部満たしていない。** US-002（花束構成定義）は US-001（商品登録）と US-003（単品登録）に依存している。US-005（注文）は US-001, US-002, US-003, US-004 すべてに依存する。依存関係を明示するか、実装順序の制約として記載すべきである。

- **【重要度: 中】決済・支払いに関する要件が完全に欠落している。** WEB ショップで注文を受け付けるにもかかわらず、決済フローの記述がない。MVP では請求書払い（既存の商取引関係）なのか、クレジットカード決済なのか、あるいはシステムスコープ外として割り切るのか。ビジネス判断を明記すべきである。

- **【重要度: 中】ストーリーのフォーマットが「として/したい/なぜなら」で統一されているが、XP のストーリーカード形式（As a / I want / So that）との対応がやや不明瞭。** 「なぜなら」の部分がビジネス価値ではなく手段の説明になっているケースがある。例えば US-002 の「花束の構成が在庫管理と結束作業の基盤となるためだ」はシステム的な理由であり、ビジネス価値（例: 「お客様に正確な花束を届けるためだ」）の方が適切。

- **【重要度: 低】廃棄処理のユースケース・ストーリーがない。** 在庫の状態遷移に「廃棄対象 → 廃棄処理」があるが、これに対応するユースケースやストーリーが存在しない。経営者の要求「廃棄ロスの削減」を実現するには、廃棄の記録と可視化も必要になる。Phase 2 以降のバックログ候補として検討すべき。

- **【重要度: 低】通知機能に関するストーリーがない。** 業務フローでは「得意先に変更完了を通知」「得意先に変更不可を通知」とあるが、通知手段（メール、画面内通知など）のストーリーがない。

---

### 懸念事項

- **MVP でエンドツーエンドの業務フローが完結しないリスク。** 受注を受け付けても出荷ができないため、MVP リリース時点で実業務に投入できない可能性がある。MVP の価値提案を「受注と在庫管理の効率化」に限定するなら、その旨を明示的に定義すべき。
- **「経営者」がマスタ管理の主アクターになっている点。** 小規模事業者では経営者がマスタ管理を兼任するのは自然だが、事業成長時にロールの分離が必要になる。現時点では問題ないが、ロール設計に拡張の余地を残しておくべき。
- **在庫推移計算のパフォーマンスリスク。** 日別在庫予定数の計算は、単品数 x 日数 x（入荷予定 + 受注引当 + 廃棄予定）の組合せで計算量が増大する。非機能要件として応答時間の目標値を定義しておくべき。
- **アクター「仕入先」の関与が一方的。** 仕入先はシステムに直接アクセスするのか、FAX/メール等の外部チャネルなのか。BUC-06 では仕入先が支援アクターとなっているが、システムとの接点が不明確。

---

### スコープ外の発見

- **要件定義の「状態モデル」セクションが2回記述されている。** `requirements_definition.md` の 844 行目と 879 行目に「### 状態モデル」が重複しており、844 行目の方は内容が空。構造上の記述ミスと思われる。
- **システムユースケースのタイトルに絵文字が使用されている。** `## UC-001: 🖥️商品を注文する🌊` のように絵文字が含まれているが、CLAUDE.md のガイドラインでは「過度な絵文字の利用は避ける」とされている。ドキュメントの一貫性の観点から削除を検討すべき。
- **BUC-13（認証・ログインする）がビジネスユースケースのトレーサビリティテーブルに含まれていない。** `business_usecase.md` のトレーサビリティセクションに BUC-13 の行が欠落している。
- **情報モデルで「受注」と「商品」の関連が `1 -- 1` になっている。** 「1注文 = 1商品」ルールの反映だが、将来的な拡張（複数商品注文）を考えると、この制約がデータモデルに組み込まれていることを設計チームに共有すべき。

**アーキテクトの視点**

### 評価サマリー

RDRA 2.0 の4層構造に忠実に従い、ビジネスユースケースからシステムユースケース、ユーザーストーリーまでのトレーサビリティが明確に確保された高品質な要件定義です。一方で、在庫推移計算の技術的複雑度に対する深掘りと、MVP フェーズにおけるアーキテクチャリスクの特定がやや不足しています。

### 良い点

- **トレーサビリティの一貫性**: BUC → UC → US の対応関係が明示されており、トレーサビリティマトリクスも完備されています。要件の抜け漏れを構造的に検出しやすい設計です。
- **ドメインルールの明示化**: 在庫推移計算ルール（`日別在庫予定数 = 前日在庫 + 当日入荷予定 - 当日受注引当 - 当日廃棄予定`）が数式レベルで定義されており、実装者が迷わない水準です。
- **「1 注文 = 1 商品」ルールの明示**: ドメインの制約を早い段階で明確にしている点は、アーキテクチャ設計・データモデル設計において非常に有益です。
- **状態遷移の網羅性**: 受注・在庫・発注の3つの状態遷移が図と表の両方で定義されており、実装時のエッジケース対応を促進します。
- **境界値の定義**: 届け日、メッセージ長、価格範囲などの制約が明記されており、バリデーション実装とテスト設計に直結します。
- **在庫引当のタイミング明記（UC-007 補足）**: 論理引当（受注時）と物理消費（結束時）の区別が明確で、在庫モデルの設計方針が読み取れます。
- **フェーズ分割の妥当性**: MVP に基盤となる商品管理・受注・在庫管理を集中させ、出荷管理と顧客体験向上を後続フェーズに回す判断は合理的です。

### 改善提案

- **【重要度: 高】在庫推移計算のパフォーマンス要件の明確化**
  在庫推移計算は「日別 × 単品数」のマトリクス計算であり、単品数の増加に対して計算量が線形以上に増加します。要件定義の段階で、以下を明確にすべきです。
  - 在庫推移の表示対象期間（何日先まで？30日？ 90日？）
  - 想定する単品数の上限（マスタ件数の規模感）
  - リアルタイム計算か、バッチ計算か（受注登録のたびに再計算するのか）
  これが不明確なまま実装に入ると、在庫推移画面がボトルネックになるリスクがあります。

- **【重要度: 高】同時操作に関する整合性要件の不足**
  以下のシナリオで競合が発生し得ますが、要件として言及がありません。
  - 受注スタッフが届け日を変更中に、別の注文が入り在庫引当が変わる
  - 2人の仕入スタッフが同時に同一単品の在庫推移を見て、それぞれ発注する
  - 結束完了登録と注文キャンセルが同時に発生する
  楽観的ロックで十分か、業務ルールレベルで排他制御が必要かを、少なくとも非機能要件として明示するか、ユースケースの拡張シナリオに追加すべきです。

- **【重要度: 中】情報モデルの「在庫」と「入荷ロット」の関係の曖昧さ**
  情報モデルでは `item "1" -- "0..1" stock` となっていますが、ドメインモデル設計では「入荷ロットごとの在庫情報」と定義されています。品質維持日数の起算は入荷日ベースなので、1単品に対して複数の在庫ロットが存在するはずです。情報モデルのカーディナリティは `item "1" -- "0..*" stock` が正しいのではないでしょうか。この不一致は在庫推移計算の実装に直接影響します。

- **【重要度: 中】UC-001 における在庫確認の欠落**
  UC-001（商品を注文する）の主成功シナリオでは、得意先が注文を確定する前に在庫が十分かどうかを確認するステップがありません。拡張 3a で「在庫がない場合」に言及していますが、在庫チェックのタイミングが曖昧です。届け日を指定した時点でリアルタイムに在庫を確認するのか、注文確定時に確認するのかを明確にすべきです。

- **【重要度: 中】得意先によるキャンセルの導線の欠落**
  UC-003 のキャンセル拡張は受注スタッフが行うとなっていますが、得意先自身が WEB ショップからキャンセルできるかどうかが不明です。ビジネスユースケースの業務フローでも、キャンセルの導線は定義されていません。得意先がセルフサービスでキャンセルできないとすると、受注スタッフの負荷に影響します。意図的なスコープ外であれば、その旨を明記すべきです。

- **【重要度: 低】UC-013 のロール別アクセス権限表に「経営者」のアクセス範囲が狭い**
  経営者は UC-009, UC-010, UC-012 にのみアクセス可能となっていますが、経営者が受注状況や在庫推移を閲覧したいというのは自然な要求です。要求モデルでも経営者は「受注キャパシティ拡大」「廃棄ロス削減」を求めているため、参照系 UC（UC-003, UC-004）への読み取りアクセスは検討すべきです。

### 懸念事項

- **在庫推移計算がシステムの技術的コア**であるにもかかわらず、その計算モデルのテスタビリティに関する言及がありません。在庫推移計算は入力パラメータが多く（在庫ロット × 品質維持日数 × 入荷予定 × 受注引当）、組合せ爆発が起きやすい領域です。ドメインモデルとして隔離し、ユニットテストで網羅的に検証できるアーキテクチャにすることが必須です。ヘキサゴナルアーキテクチャの採用が ADR で決定されている点は安心材料ですが、要件レベルで「在庫推移計算の正確性」を検証可能な受け入れ基準（具体的な計算例）があるとさらに良いです。

- **MVP フェーズで在庫推移（US-009）を含めているが、結束（US-012, US-013）はフェーズ 2 に回している点**に整合性の問題があります。在庫推移計算の「当日受注引当」は、出荷日（届け日の前日）に花材を消費する前提ですが、結束機能が MVP に含まれない場合、在庫の物理消費を記録する手段がありません。MVP 期間中の在庫推移は「引当のみ・消費なし」で計算上は正しいのか、それとも何らかの手動在庫消費手段が必要なのかを明確にすべきです。

- **通知機能の要件が不在**です。届け日変更の業務フローでは「得意先に変更完了を通知」「得意先に変更不可を通知」とありますが、通知手段（メール、画面通知、SMS 等）が定義されていません。通知はインフラ依存度が高い横断的関心事であり、後から追加すると影響範囲が大きくなります。

- **認証・認可のアーキテクチャ影響度**が過小評価されている可能性があります。6つのロールに対する RBAC（Role-Based Access Control）は、全 UC の横断的関心事です。UC-013 で定義はされていますが、JWT/セッション管理、トークンリフレッシュ、パスワードリセットなどの技術的詳細は早めに検討が必要です。

### スコープ外の発見

- **情報モデルの「受注」と「商品」のカーディナリティ** `order "1" -- "1" product` は、1注文=1商品ルールと整合しますが、将来的に複数商品注文に拡張する可能性がある場合、受注明細（OrderLine）を介した設計の方が変更容易性が高まります。現時点では意図的な制約として妥当ですが、ADR として記録しておくことを推奨します。
- **ドメインモデル設計書の Stock のカーディナリティ**（`item "1" -- "0..*" stock`）と**要件定義の情報モデル**（`item "1" -- "0..1" stock`）が不一致です。これはレビュー対象外のドキュメントの問題ですが、要件定義側の修正が必要です。
- **決済処理がスコープ外**となっている（バックエンドアーキテクチャ設計書より）ことは、要件定義書のどこにも明示されていません。「クレジットカード事前登録済み」という前提条件は、要件定義のシステムコンテキストまたはスコープ定義に記載すべきです。

**インタラクションデザイナーの視点**

### 評価サマリー

全体として、RDRA 2.0 の4層構造に沿った要件定義が丁寧に行われており、ユーザーストーリーのトレーサビリティも確保されています。しかし、ユーザーの心理モデルとの整合や画面遷移の自然さにおいて、「得意先が自分の注文状況を確認・操作できない」という重大な体験の欠落があり、ユーザー体験全体としてはまだ改善の余地があります。

---

### 良い点

- **注文フローの段階的な画面遷移が自然である**: 商品一覧 → 商品詳細 → 注文入力 → 注文確認 → 注文完了という5ステップの遷移は、EC サイトの一般的な心理モデルに合致しており、得意先が迷いにくい設計です。
- **リピート注文の「届け先コピー」がフロー内に統合されている**: UC-001 のステップ 4 から UC-011 を呼び出す構造になっており、リピーターの心理モデル（「前と同じ人に送りたい」）に合致しています。注文画面ワイヤーフレーム上でも「過去の届け先から選択」ボタンが適切に配置されています。
- **在庫推移画面の情報設計が優れている**: S-201 の在庫推移画面は、品質維持日数・購入単位・リードタイムをヘッダーに表示し、日別の内訳（前日在庫、入荷予定、受注引当、廃棄予定、在庫予定）をテーブルで一覧できる設計です。仕入スタッフの「一目で発注要否を判断したい」という心理モデルに合致しています。
- **受注ステータス遷移が明確に定義されている**: 状態遷移図・状態遷移表の両方で定義されており、「出荷準備中以降はキャンセル・届け日変更不可」というビジネスルールが UI のボタン制御に直結できる形で記述されています。
- **エラー処理・フィードバックのパターンが体系的に定義されている**: UI 設計書のエラー処理テーブルで7パターンが定義されており、エラー防止（購入単位の倍数チェック）やエラー回復（代替日提案）も考慮されています。
- **ロール別ナビゲーションが定義されている**: 6つのロールそれぞれに表示メニューが定義されており、認知負荷の最小化に貢献しています。

---

### 改善提案

- **【重要度: 高】得意先が自分の注文状況を確認できるユースケース・ストーリーが欠落している**。EC サイトの利用者は「注文したあと、ちゃんと受け付けられたか」「いつ届くのか」を確認したいという強い心理モデルを持っています。現状では注文完了画面（S-007）の受注番号が表示されるだけで、その後に注文履歴や注文詳細を確認する手段がありません。UC-013 のロール別アクセス権限でも得意先は UC-001 と UC-011 のみです。「得意先として、自分の注文履歴と注文ステータスを確認したい。なぜなら注文が正しく処理されているか不安だからだ」というストーリーの追加を提案します。

- **【重要度: 高】得意先自身による届け日変更・キャンセル操作の導線が存在しない**。要件定義の業務フローでは「得意先が届け日変更を依頼」→「受注スタッフが対応」となっており、UC-002 の主アクターも受注スタッフです。しかし、得意先がどのようにして変更やキャンセルを「依頼」するのかのチャネルが定義されていません（電話？メール？WEB 上のフォーム？）。WEB ショップシステムという文脈では、得意先が WEB 上から変更依頼を送れる導線がないと、せっかく WEB で注文できても変更は電話というちぐはぐな体験になります。少なくとも「得意先がシステム上で変更依頼を送信し、受注スタッフが処理する」というフローを明示するか、システムスコープ外であれば明確にその旨を記述すべきです。

- **【重要度: 高】注文画面（S-005）で届け日指定時に在庫不足だった場合の UI フィードバックが未定義**。UC-001 の拡張 3a で「システムが対応可能な最寄りの届け日を提案する」とありますが、注文画面のワイヤーフレームにはこのフィードバックの表示領域がありません。カレンダーピッカー上で在庫のない日をグレーアウトするのか、日付選択後にメッセージを表示するのか、具体的なインタラクションパターンを定義してください。

- **【重要度: 中】注文確認画面（S-006）から商品一覧に戻る導線がない**。現在は「修正する」（注文画面に戻る）と「注文を確定」の2つだけですが、「やっぱり別の商品にしたい」という心理モデルに対応できていません。注文画面（S-005）からも商品一覧への導線がありません。ブラウザの戻るボタンに依存するのではなく、明示的なナビゲーションを提供すべきです。

- **【重要度: 中】管理画面のナビゲーションがロールによって異なるが、ヘッダーの一貫性が保たれていない**。結束対象一覧画面（S-401）と出荷一覧画面（S-402）のナビゲーションは「受注管理 | 在庫管理 | 発注管理 | 出荷管理」ですが、他の管理画面（S-101, S-201, S-301 等）は「受注管理 | 在庫管理 | 発注管理 | 商品管理」です。ロール別ナビゲーションの定義と実際のワイヤーフレームのヘッダーが一致していないため、混乱の原因になります。

- **【重要度: 中】商品管理画面（S-501）が一覧・登録・編集・構成管理のすべてを1画面に詰め込んでおり、認知負荷が高い**。経営者の心理モデルとして「商品を探す」「商品を登録/編集する」「構成を管理する」は異なるタスクであり、それぞれの文脈に応じた画面に分割するか、タブやモーダルで切り替える設計にすることを提案します。

- **【重要度: 中】アクセシビリティに関する言及がドキュメント全体で欠如している**。ワイヤーフレームにおいて、フォームラベルとフィールドの関連付け、エラーメッセージのアクセシブルな通知方法（aria-live 等）、キーボードナビゲーションの考慮、テーブルのスコープ定義などが未定義です。WCAG 2.1 AA を最低基準とする場合、これらの方針を早期に定義しておくことを提案します。

- **【重要度: 低】在庫推移画面（S-201）で在庫不足を示す視覚的なアラートが不足している**。ワイヤーフレームでは在庫予定数が少ない行を太字で示していますが、色による強調やアイコンの付与がないため、見落とされるリスクがあります。ただし、色だけに依存しない設計にする必要があります（WCAG 1.4.1）。

- **【重要度: 低】ユーザーストーリーの US-015（届け先をコピーする）がフェーズ 3 に配置されているが、リピーター獲得の観点では MVP に含めるべき**。要求モデルで経営者の要求として「リピート率を向上させたい」が明示されており、得意先の要求として「過去の届け先を再利用して簡単にリピート注文したい」があります。リピーターが MVP で注文できても届け先コピーが使えないとなると、リピーターの体験が想定より大幅に低下します。

---

### 懸念事項

- **システムメタファーが明確に定義されていない**。要件定義・ユースケース・UI 設計を通じて「WEB ショップ」という言葉が使われていますが、このシステムが得意先にとって「オンラインの花屋」なのか「法人向け発注ポータル」なのか、メタファーが曖昧です。「得意先」という用語は B2B 的な響きですが、UI 設計は B2C の EC サイトに近い構成です。このメタファーの不一致は、将来的にナビゲーション設計やコミュニケーションの方向性がぶれるリスクがあります。「得意先」「顧客」「お客様」の使い分けを含め、システム全体のメタファーを明確にすることを推奨します。

- **得意先のアカウント新規登録後の onboarding フローが未定義**。US-018 では「登録後、自動的にログインされる」で終わっていますが、初回利用者が商品一覧に飛ばされた後にどう行動するかの導線が考慮されていません。初回利用者向けのウェルカムメッセージや利用ガイドなどが必要になる可能性があります。

- **「1注文 = 1商品」ルールがユーザーの心理モデルと大きく乖離する可能性がある**。一般的な EC サイトの利用者は「複数の商品をカートに入れてまとめて注文する」という心理モデルを持っています。この制約はビジネスルールとして明記されていますが、UI 上でどのようにこの制約を伝え、ユーザーの混乱を防ぐかが未定義です。商品詳細画面に「1回のご注文につき1つの花束をお選びください」といった明示的な説明が必要です。

- **BUC-13（認証・ログインする）のビジネスユースケースが business_usecase.md のユースケース一覧には含まれているが、全体図の PlantUML には描画されていない**。トレーサビリティの観点で不整合です。

- **商品詳細画面（S-004）に商品画像の表示がない**。花束という商品特性上、視覚的な訴求力がきわめて重要です。ワイヤーフレームにテキスト情報しかなく、画像表示の考慮が見当たりません。

---

### スコープ外の発見

- **仕入先との発注連携がシステム外に依存している**。UC-005 で「システムが発注を登録する」とありますが、実際に仕入先に発注情報を伝達する手段（FAX、メール、EDI 等）が未定義です。発注画面から発注を登録しても、仕入先への通知は別途手動で行う必要があると思われますが、この点が明示されていないため、利用者が「システムで発注したら自動的に仕入先に通知される」と誤解するリスクがあります。
- **配送管理の外部システム連携が未定義**。UC-008 で「出荷する」とありますが、配送業者との連携（送り状発行、追跡番号取得等）がシステムスコープに含まれるかどうかが不明確です。出荷一覧画面（S-402）に「出荷処理」ボタンがありますが、これが何を実行するのか（ステータス更新のみか、外部連携を含むか）が曖昧です。
- **得意先への通知手段（メール等）が未定義**。届け日変更の可否通知、注文確認メールなどの通知要件がシステムユースケースに含まれていません。
メモリの書き込みに権限がありませんでしたが、レビュー自体は上記の通り完了しています。

レビュー結果の中で特に重要度が高い3点を改めて強調しておきます。

1. **得意先の注文状況確認機能の欠落** -- これは EC サイトとしての基本体験が抜けており、最優先で対応すべきです。
2. **得意先からの変更/キャンセル依頼の導線未定義** -- WEB システムなのに変更依頼のチャネルが不明確な点は、システムの価値を損ないます。
3. **注文時の在庫不足フィードバックの UI 未定義** -- ユースケースの拡張シナリオとして定義されていますが、ワイヤーフレームに反映されていません。

**テスターの視点**

### 評価サマリー

全体として RDRA 2.0 の 4 層構造に沿った構成は明確で、トレーサビリティも確保されています。ただし、テスト可能性の観点では、受け入れ基準の定量性が不十分な箇所、状態遷移のエッジケースの未定義、在庫推移計算ルールの検証条件の曖昧さなど、テスト設計時に「仕様を確認しに戻る」必要が生じる箇所が複数見受けられます。

---

### 良い点

- **境界値の明示**: 要件定義書に「届け日: 翌日〜30 日後」「お届けメッセージ: 最大 200 文字」「発注数量: 購入単位の倍数、上限 9,999」等の境界値が明確に定義されており、境界値分析テストを直接設計できる状態です
- **状態遷移表の完備**: 受注の状態遷移が状態遷移図と状態遷移表の両方で記述されており、状態遷移テストの設計が容易です。特に「出荷準備中以降はキャンセル・届け日変更不可」というルールが明確で、テストの正常系・異常系の境界がはっきりしています
- **在庫推移計算式の明示**: `日別在庫予定数 = 前日在庫 + 当日入荷予定 - 当日受注引当 - 当日廃棄予定` が計算式として定義されており、ユニットテストで直接検証可能です
- **「1 注文 = 1 商品」ルール**: ビジネスルールが明確で、テストデータの構造設計がシンプルになります
- **トレーサビリティマトリクス**: ユーザーストーリー → システムユースケース → ビジネスユースケースの対応が整備されており、テストの網羅性確認に活用できます
- **在庫引当タイミングの明記**: UC-007 の補足に「論理引当（受注時）」「物理消費（結束時）」「引当解除（キャンセル時）」が明記されており、テストシナリオ設計に直結します
- **テスト戦略との整合**: テスト戦略ドキュメントでドメインモデルの状態遷移テストと在庫推移計算テストが具体例として挙げられており、要件定義の記述と一致しています

---

### 改善提案

#### 受け入れ基準の定量性不足

- **【重要度: 高】US-005「花束を注文する」の在庫確認に関する受け入れ基準が欠落しています。** UC-001 の拡張 3a では「指定した届け日に対応できる在庫がない場合、対応可能な最寄りの届け日を提案する」と記述されていますが、US-005 の受け入れ基準にこのケースが含まれていません。テスト設計時に「在庫不足時の注文はどうなるのか」が曖昧になります。以下の受け入れ基準を追加すべきです:
  - `指定した届け日に在庫が不足する場合、対応可能な代替の届け日が提案される`
  - `在庫不足の場合でも注文自体は可能（在庫推移上の論理引当として計上される）`か、それとも注文をブロックするのか、を明確にする必要があります

- **【重要度: 高】US-009「在庫推移を表示する」の受け入れ基準に、在庫推移計算ルールの検証条件が不足しています。** 要件定義書に定義されている計算式の各項目（前日在庫、当日入荷予定、当日受注引当、当日廃棄予定）が受け入れ基準として検証できる形になっていません。以下を追加すべきです:
  - `在庫推移は「前日在庫 + 当日入荷予定 - 当日受注引当 - 当日廃棄予定」で計算される`
  - `受注引当は出荷日（届け日の前日）に計上される`
  - `廃棄予定は入荷日 + 品質維持日数で計算される`
  - `表示期間の範囲（何日先まで表示するのか）`が未定義

- **【重要度: 高】US-010「単品を発注する」に発注数量のバリデーション受け入れ基準が不足しています。** 要件定義書の境界値で「発注数量: 購入単位の倍数。上限 9,999」と定義されていますが、US-010 の受け入れ基準にはこれが反映されていません。以下を追加すべきです:
  - `発注数量は購入単位の倍数でなければならない`
  - `購入単位の倍数でない場合、切り上げた数量が提案される`
  - `発注数量の上限は 9,999 である`

- **【重要度: 中】US-001「商品（花束）を登録する」の受け入れ基準に境界値条件が不足しています。** 要件定義書で定義されている「商品名: 最大 50 文字」「商品価格: 1 円〜999,999 円」が受け入れ基準に含まれていません。テストケース設計時に境界値分析が必要になる箇所なので明示すべきです

- **【重要度: 中】US-003「単品（花材）を登録する」も同様に、「品質維持日数: 1〜30 日」「リードタイム: 1〜14 日」「単品名: 最大 50 文字」等の境界値が受け入れ基準に反映されていません**

- **【重要度: 中】US-017「システムにログインする」の「5 回連続失敗」の定義が曖昧です。** 「連続」の定義（時間制限の有無、成功でリセットされるのか）、ロックの期間、ロック解除方法が未定義で、テストシナリオが書けません。以下を明確にすべきです:
  - ロックの持続期間（15 分、30 分、管理者解除等）
  - 成功ログインで失敗カウントがリセットされるか
  - ロック中のログイン試行に対するエラーメッセージ

#### ユースケースの異常系・エッジケース

- **【重要度: 高】UC-001「商品を注文する」に同時注文（競合）の考慮がありません。** 複数の得意先が同じ花束を同じ届け日で同時に注文した場合の在庫引当の排他制御について記述がなく、テストシナリオの設計ができません。これは統合テスト・E2E テストで検証すべき重要なエッジケースです

- **【重要度: 中】UC-006「入荷を受け入れる」に入荷数量が発注数量を超過した場合の扱いが未定義です。** 拡張で「一部入荷」は定義されていますが、超過入荷のケースが考慮されていません

- **【重要度: 中】UC-007「花束を結束する」で、在庫が不足している場合のフローが明確でありません。** 事前条件に「必要な花材の在庫がある」とありますが、結束当日に在庫が不足していた場合のエラーハンドリングが拡張として定義されていません

- **【重要度: 低】UC-002「届け日を変更する」の拡張 4a2 で「代替日を表示する」とありますが、代替日の算出ロジック（最寄りの在庫充足日？最大何日分提示？）が未定義です**

#### 状態遷移の未定義領域

- **【重要度: 高】在庫の状態遷移に「引当済み」状態がありますが、受注キャンセル時の引当解除に伴う状態遷移が状態遷移図に含まれていません。** `引当済み → 在庫あり（キャンセルによる引当解除）` の遷移パスが必要です。UC-003 拡張 5a4 で「在庫引当を解除する」と記述されているため、状態遷移図との整合が取れていません

- **【重要度: 中】発注の状態遷移に「キャンセル」状態がありません。** 発注後にキャンセルする業務フローが存在しないのか、意図的に除外しているのかが不明です。テスト設計時にネガティブテスト（存在しないはずの遷移を試みるテスト）を書くべきか判断できません

---

### 懸念事項

- **在庫推移計算の表示期間未定義**: 在庫推移を「何日先まで」表示するのかが要件定義・ユースケース・ユーザーストーリーのいずれにも記述されていません。テストデータの準備範囲に直結するため、早期に定義が必要です

- **品質維持日数「に近づく」の定義不在**: 在庫状態の「品質低下」は「品質維持日数に近づいている状態」と定義されていますが、具体的に何日前から「品質低下」になるのかが未定義です。これは在庫推移計算のテストケース設計に影響します（境界値テストが書けない）

- **得意先の「認証済みアクセス」と注文フローの関係**: UC-001「商品を注文する」の事前条件に「認証」が明示されていません。UC-013 のロール別アクセス権限表と照合すると得意先は UC-001 にアクセス可能とされていますが、未ログイン状態での商品一覧閲覧の可否が不明です。E2E テストの前提条件設計に影響します

- **MVP フェーズでの整合性リスク**: MVP に UC-003「受注を確認・管理する」は含まれていますが、キャンセル（US-019）は Phase 2 です。しかし UC-003 の拡張 5a にキャンセルが含まれています。MVP のテスト範囲でキャンセルのテストが必要か不要かが曖昧です。フェーズ分割に沿ったテスト範囲の明確化が必要です

- **テスト戦略との整合に関する懸念**: テスト戦略で E2E テストの対象シナリオに「注文フロー」「受注管理フロー」「在庫・発注フロー」の 3 つが挙げられていますが、「出荷・配送フロー」が含まれていません。Phase 2 の US-012〜014 が E2E テストで検証されるタイミングを明確にすべきです

**ユーザー代表の視点**

### 評価サマリー

花屋の業務フローをよく理解した上で要件がまとめられており、特に「品質維持日数を考慮した在庫推移の可視化」や「発注判断は人間が行い、システムは判断材料を提供する」という設計思想は、現場の実態に即しています。ただし、得意先（お客様）の日常的な注文体験と、スタッフの繁忙期の業務フローにおいて、いくつか見落とされている重要なシナリオがあります。

### 良い点

- **「1 注文 = 1 商品」ルールの明示**: 花束の注文において、1 件 1 商品という割り切りは現場のオペレーションに合っています。花束ごとに届け先やメッセージが異なるのが通常ですので、この判断は正しいです。
- **品質維持日数を軸にした在庫推移**: 花という生鮮品の特性を正しく捉えています。仕入スタッフが「何日後にどの花材がどれくらい残るか」を見ながら発注判断できるのは、まさに現場が求めていることです。
- **届け先コピー機能**: リピーターのお客様にとって、毎回同じ届け先を入力するのは大きなストレスです。この機能は確実にリピート率向上に貢献します。
- **在庫引当の論理引当/物理消費の区別**（UC-007 の補足）: 受注時と結束時で引当の扱いを分けている点は、現場の実態を正確に反映しています。
- **発注ステータスの「一部入荷」**: 花材の仕入れでは全量が揃わないことが日常的に発生するので、これは必要な状態です。
- **受注ステータス遷移表の記載**: 「出荷準備中以降はキャンセル・届け日変更不可」という制約が明記されており、受注スタッフがお客様に説明しやすいです。

### 改善提案

- **【重要度: 高】得意先が自分で注文履歴を確認できる機能が欠落しています。** 花束を贈るお客様は「去年の母の日に何を贈ったか」「先月の注文はいつ届くか」を確認したいというニーズが非常に強いです。現状のユースケースでは得意先が自分の注文状況を見る手段がありません。電話やメールで受注スタッフに問い合わせが集中し、かえって業務負荷が増えます。MVP に「得意先が自分の注文履歴・ステータスを確認する」ユーザーストーリーを追加すべきです。

- **【重要度: 高】注文キャンセルが Phase 2 に回されていますが、MVP に含めるべきです。** 実際の業務では、注文直後に「届け日を間違えた」「商品を変えたい」というお客様からの連絡は日常茶飯事です。キャンセルできない受注システムは、受注スタッフの対応を困難にし、お客様の不満にも直結します。最低限、「注文受付」ステータスでのキャンセルは MVP で対応する必要があります。

- **【重要度: 高】届け日変更の主アクターが受注スタッフのみですが、得意先自身による変更も検討すべきです。** 業務フローでは得意先が変更を「依頼」し、受注スタッフが対応する形になっていますが、WEB ショップを導入する目的の一つが「手作業の限界を解消」することです。少なくとも「注文受付」ステータスであれば得意先が自分で届け日を変更できるようにしないと、受注スタッフの負荷が軽減されません。Phase 2 であっても設計としては盛り込んでおくべきです。

- **【重要度: 中】注文確認メール・通知の仕組みが要件に含まれていません。** WEB ショップで注文したお客様が「本当に注文できたのか」不安になるのは当然です。注文受付時の確認メール、ステータス変更時の通知（出荷完了など）は、お客様の安心感に直結します。EC システムでは常識的な機能であり、少なくとも注文確認メールは MVP に含めるべきです。

- **【重要度: 中】商品一覧での「在庫なし」「注文不可」の表示が定義されていません。** UC-001 のステップ 1 で商品を選択した後、届け日を指定するステップ 3 まで在庫不足がわからない設計です。お客様の立場からすると、選んだ花束が実は注文できないと分かるのは大きなストレスです。商品一覧の段階で「直近で注文可能かどうか」が分かると、お客様の体験が大幅に改善されます。

- **【重要度: 中】在庫推移画面で「発注推奨」のアラート・ハイライトがあると仕入スタッフの作業効率が上がります。** 要件には「発注判断は人間が行う」とありますが、在庫推移を毎朝目で確認して不足を見つけるのは、単品の数が増えると見落としが起こります。「在庫が一定数を下回る日がある」単品を色分けやアラートで示す機能があると、仕入スタッフの日常業務が楽になります。

- **【重要度: 中】フローリストの結束作業画面で、花材の在庫所在（どこに何があるか）が確認できると作業効率が上がります。** 現在の US-012 は「必要な花材と数量」の表示のみですが、実際の作業場では「どの花材をどこから取るか」も重要です。ただし、この粒度は Phase 2 の実装段階で詳細化すれば十分です。

- **【重要度: 低】得意先の「法人」「個人」の区別が明確でありません。** 「得意先」という用語は一般的に法人取引を連想させますが、要件を読むと個人顧客も含むようです。ユーザーストーリーでは「個人顧客」として記載されていますが、法人顧客のまとめ注文（例: 企業の株主総会用に 50 束）のようなケースも将来的にありうるなら、早めに用語を整理しておくべきです。

- **【重要度: 低】商品の「季節限定」「販売期間」の概念が欠落しています。** 花束は季節商品の側面が強く、「春の花束」「母の日特別商品」のように期間限定で販売する商品があるのが一般的です。商品マスタに販売開始日・終了日の属性が必要になる可能性があります。

### 懸念事項

- **決済（支払い）の仕組みが完全に欠落しています。** WEB ショップで注文する以上、支払いは避けて通れません。クレジットカード決済なのか、銀行振込なのか、請求書払い（法人向け）なのか。この部分が要件定義に一切登場しないのは大きなリスクです。外部の決済サービス連携が必要であれば、技術選定やアーキテクチャ設計にも影響します。意図的にスコープ外としているなら、その旨を明示すべきです。

- **「届け完了」のステータス遷移トリガーが不明確です。** 状態遷移では「出荷済み」から「届け完了」への遷移がありますが、誰がどのタイミングでこのステータスを更新するのかが定義されていません。配送業者の配達完了通知に連動するのか、手動更新なのか。この部分が曖昧だと、お客様に「届きました」と通知するタイミングも決められません。

- **受注スタッフが在庫推移を確認する導線が弱いです。** 業務フロー「商品を注文する」では、受注スタッフが受注確認後に在庫推移を確認する手順がありますが、システムユースケースでは受注管理（UC-003）から在庫推移（UC-004）への連携が明確でありません。受注詳細画面から直接、関連する花材の在庫推移に飛べる導線が必要です。

- **繁忙期（母の日、クリスマス等）の業務量増加への対応が考慮されていません。** 花屋にとって母の日は年間売上の 2-3 割を占めることもある特別な時期です。この時期に注文が殺到した場合、受注スタッフ・フローリスト・配送スタッフの作業量が通常の数倍になります。「受注上限」「届け日ごとの注文枠」といった制御が必要になる可能性がありますが、現在の要件では考慮されていません。

### スコープ外の発見

- **情報モデルで「受注」と「商品」が 1:1 の関係になっていますが、将来的に「アレンジメント」「スタンド花」「観葉植物」等の商品種別を追加する場合、この構造で問題ないか検討が必要です。** 現時点では花束のみで問題ありませんが、ビジネス拡大を見据えるなら情報モデルの拡張性を確認しておくべきです。

- **配送に関する外部システム連携（配送業者の API 連携等）が一切言及されていません。** 自社配送なのか外部委託なのかによって、出荷管理のシステム要件が大きく変わります。

- **受注ステータスの「注文受付」と「受付済み」の 2 段階が、小規模店舗のオペレーションとして本当に必要かは検討の余地があります。** 受注件数が少ないうちは、注文即「受付済み」でも業務は回ります。受注量が増えた段階で導入する方が、初期の運用負荷を減らせます。
