# UI/UX レビュー結果 - IT4 在庫推移・発注画面

**レビュー日**: 2026-03-22
**レビュー対象**: S-201 在庫推移画面、S-301 発注画面、S-302 入荷登録画面（docs/design/ui-design.md）

## レビュー対象

- S-201: 在庫推移画面（在庫アラートルール含む）
- S-301: 発注画面（新規発注フォーム + 発注一覧）
- S-302: 入荷登録画面
- 管理画面遷移（在庫・発注管理）
- デザインシステム基盤のインタラクション状態定義

## 総合評価

在庫推移→発注→入荷の画面遷移は仕入スタッフの「確認→判断→行動」という認知プロセスに沿っており、ドメインモデル（DailyInventory）と UI カラムの 1:1 対応も良好。在庫アラートの 2 段階ルール（amber/red）はドメイン知識と視覚フィードバックが連動した合理的な設計。一方で、在庫推移→発注の**コンテキスト引き継ぎ不足**（単品 ID・アラート日付がフォームにプリフィルされない）と、購入単位バリデーションの**提案 UI 未定義**が仕入スタッフの業務効率を損なうリスクがある。

## モダンデザイン準拠サマリー

| 評価項目 | 状態 | 備考 |
|---|---|---|
| カラーシステム | OK | Emerald 系ブランドカラー + amber/red アラート色が定義済み |
| タイポグラフィ | OK | 6 レベルの Type Scale が定義済み |
| Elevation & Surface | 要改善 | Card の shadow-sm のみ。コンポーネント別エレベーション階層が未定義 |
| コンポーネント一貫性 | 要改善 | Hover/Focus/Active 状態が Input のみ定義。Button/Table Row が未定義 |
| スペーシング | OK | 8px グリッドシステム準拠 |
| レスポンシブ / Adaptive | 要改善 | 在庫推移 7 カラムのモバイル対応が「横スクロール or 縦表示」で不明確 |
| ダークモード | 未対応 | 記述なし |
| 状態デザイン（空 / Loading / Error） | OK | 空状態・スケルトン・二重送信防止が定義済み |

## 改善提案（重要度順）

### 高（実装前に対応すべき）

| # | 提案 | 箇所 | 指摘元 | 理由 |
|---|------|------|--------|------|
| H-1 | S-201→S-301 遷移時に選択中の単品 ID とアラート日付をプリフィルする | S-201, S-301 | デザイナー / ユーザー代表 | 単品の選び直しと日付記憶はエラーの元。在庫推移で見た情報が発注フォームに引き継がれないと二度手間 |
| H-2 | 在庫推移テーブルのモバイル対応を具体化（Tablet: 内訳折りたたみ 4 列、Mobile: カード形式） | S-201 | デザイナー | 7 カラムの横スクロールは操作性が著しく悪い |
| H-3 | 購入単位バリデーションのインライン提案 UI を定義（「15→20 本に切り上げますか？」） | S-301 | デザイナー / ユーザー代表 | エラーではなく提案として表示し、仕入スタッフの判断を尊重する設計が必要 |
| H-4 | コンポーネントの Hover/Focus/Active 状態をデザインシステムに追加 | デザインシステム基盤 | デザイナー | テーブル行のホバー（特にアラート行）は操作対象の視覚フィードバックに必須 |

### 中（対応推奨）

| # | 提案 | 箇所 | 指摘元 | 理由 |
|---|------|------|--------|------|
| M-1 | 在庫推移の単品ヘッダーに仕入先名を追加 | S-201 | デザイナー | 発注判断に仕入先情報はコンテキストとして重要 |
| M-2 | 発注一覧にステータスフィルタを追加 | S-301 | デザイナー | 「発注済み」の未処理分を素早く見つける必要がある |
| M-3 | S-302 入荷数量のバリデーションフィードバック定義（残数量超過、0 以下、全量入荷確認） | S-302 | デザイナー | IT5 で実装予定だが仕様は先に確定すべき |
| M-4 | ダッシュボードの在庫アラートカードにアラート対象単品名（上位 3 件）を表示 | S-100 | デザイナー | カードから S-201 への遷移時に該当単品をフィルタ |
| M-5 | 在庫推移テーブルの初期表示期間を「当日から 14 日間」に定義 | S-201 | デザイナー | リードタイム（最長 14 日）をカバーする合理的なデフォルト |
| M-6 | 在庫推移の曜日表示を追加（「03/23(月)」形式） | S-201 | ユーザー代表 | 花屋は週末の需要パターンを意識して発注判断する |

### 低（改善の余地あり）

| # | 提案 | 箇所 | 指摘元 | 理由 |
|---|------|------|--------|------|
| L-1 | 受注引当・廃棄予定を絶対値表示にし、ヘッダーで方向性を示す | S-201 | デザイナー | マイナス符号は冗長で認知負荷を増やす |
| L-2 | アラート行に aria-label を付与し、バッジに role="status" を追加 | S-201 | デザイナー | スクリーンリーダーでのアラート認知 |
| L-3 | 在庫推移グラフ表示オプション（sparkline） | S-201 | デザイナー | Phase 2 候補。トレンド把握にグラフが有効 |

## 矛盾事項

なし。デザイナーとユーザー代表の指摘は概ね一致している。
