# UI/UX レビュー結果

**レビュー日**: 2026-03-21
**レビュー対象**: docs/design/ui-design.md（UI 設計 - フレール・メモワール WEB ショップシステム）

## レビュー対象

- 画面一覧（顧客向け 7 画面、管理画面 12 画面）
- 画面遷移図（全体、顧客向け、管理画面の受注管理、在庫・発注管理）
- 画面イメージ（PlantUML salt 図 18 画面分）
- インタラクション設計（注文フロー、受注管理フロー、在庫・発注フロー）
- エラー処理・フィードバック設計
- ロール別ナビゲーション

## 総合評価

ドメインモデルとユースケースに忠実に対応した堅実な UI 設計であり、業務フローに沿った画面遷移とインタラクション設計がよく整理されている。一方で、顧客向け画面が「業務システム」の見た目になっており EC サイトとしての購買体験が成立していない点、モダンデザインシステムへの準拠がほぼ未定義である点、レスポンシブ対応・アクセシビリティ・状態デザイン（ローディング、空状態等）の記述が欠落している点が実装フェーズでの手戻りリスクとなる。管理画面は基本的な業務フローには沿っているが、日常業務を楽にする工夫（一括操作、視覚的アラート、ダッシュボード）が不足している。

## モダンデザイン準拠サマリー

| 評価項目 | 状態 | 備考 |
|---|---|---|
| カラーシステム | 要改善 | ブランドカラー #2d6a4f は存在するが UI 設計ドキュメントに体系的な記述がない |
| タイポグラフィ | 要改善 | 太字（`<b>`）のみ。Type Scale（Display / Headline / Title / Body / Label）が未定義 |
| Elevation & Surface | 要改善 | salt 図の `{+}` ボックスのみ。Surface 階層（Card, Sheet, Dialog）が未定義 |
| コンポーネント一貫性 | 要改善 | ボタン・テーブルは統一的だが、ナビゲーションラベルが画面間で不一致 |
| スペーシング | 要改善 | 8px グリッド等のスペーシングルールが未定義 |
| レスポンシブ / Adaptive | 要改善 | ブレークポイント、レイアウト変更方針が未記述 |
| ダークモード | 未対応 | 記述なし |
| 状態デザイン（空 / Loading / Error） | 要改善 | エラー処理テーブルはあるが、空状態・ローディング状態のデザインがない |

## 改善提案（重要度順）

### 高（リリース前に対応すべき）

| # | 提案 | 箇所 | 指摘元 | 理由 |
|---|------|------|--------|------|
| H-1 | 商品一覧画面をテーブル形式からカードグリッド形式に変更し、商品画像エリアを追加する | S-003 商品一覧画面 | デザイナー / ユーザー代表 | 花束はビジュアルが購買意欲に直結する商品。テーブル形式では EC サイトとしての訴求力が著しく低い。顧客は見た目の印象で花束を選ぶ |
| H-2 | 空状態（Empty State）のデザインを全一覧画面に定義する | 全一覧画面 | デザイナー | 検索結果 0 件、注文履歴なし等のシナリオでユーザーに「壊れている」印象を与える。次のアクション導線も必要 |
| H-3 | レスポンシブ対応の方針を記述する（ブレークポイント定義、顧客向けはモバイルファースト、管理画面はデスクトップファースト） | UI 設計全体 | デザイナー / ユーザー代表 | 得意先はスマートフォンから注文する可能性が高い。管理画面の在庫推移テーブルも横スクロール問題がある |
| H-4 | ローディング状態のデザインを定義する（スケルトンスクリーン、ボタン disabled + スピナー） | 全画面 | デザイナー / ユーザー代表 | React Query による非同期データフェッチが前提のアーキテクチャだが、ユーザーへのフィードバックが未設計。注文確定ボタンの二重送信防止も必須 |
| H-5 | 商品詳細画面（S-004）に高品質な商品画像表示エリアを追加する | S-004 商品詳細画面 | ユーザー代表 | 花束を贈る場面で最も重要な「この花束の見た目」が伝わらない。画像なしでは購買決定ができない |
| H-6 | 注文確定ボタンの二重クリック防止とローディング表示を設計する | S-006 注文確認画面 | ユーザー代表 | 注文が重複すると業務に大きな影響。ボタン disabled 化 + スピナー表示が必須 |

## 矛盾事項

| # | 視点 A | 視点 B | 論点 | 推奨判断 |
|---|--------|--------|------|----------|
| 1 | デザイナー: salt 図の制約上、テーブル形式は設計段階では許容される | ユーザー代表: テーブル形式では花束を選ぶ体験が成立しない | 商品一覧の表示形式 | **ユーザー代表を優先**。salt 図はあくまで構造定義であり、実装時のビジュアル表現を制約するものではない。ただし設計ドキュメントに「実装時はカードグリッド形式で表示」の注釈を追記すべき |
| 2 | デザイナー: 商品管理画面は 3 関心を分離すべき | ユーザー代表: 画面遷移を増やすと操作が面倒。1 画面で完結する方が業務効率が良い場合もある | S-501 の画面分割 | **タブ or アコーディオン形式**で関心を視覚的に分離しつつ、同一画面内で完結させるのが妥協点 |

<summary>xp-user-representative（高: 3 / 中: 4 / 低: 3）</summary>

### 評価サマリー
管理画面は花屋の基本的な業務フローに沿っており、受注管理・在庫管理・発注管理の一連の流れは理解しやすい。しかし顧客向け画面が「業務システム」の見た目であり、花束という感情的な商品を選ぶ EC 体験が成立していない。また、日常業務を効率化する工夫（一括操作、ダッシュボード、視覚的アラート）が不足している。

### 良い点
- 業務フローに沿った画面遷移が自然（注文 → 受注 → 在庫確認 → 発注 → 入荷 → 結束 → 出荷）
- エラー処理が現実的（在庫不足時の代替日提案、購入単位の倍数切り上げ）
- 注文フローで「過去の届け先から選択」はリピーター顧客にとって便利
- ロール別ナビゲーションで不要な画面が見えないのは良い

### モダンデザイン準拠状況
- 日常的に使うモダンな EC サイト（Amazon、楽天など）と比較すると、商品画像なし・テーブル形式は大きく劣る
- 検索・フィルタ・ソートといった標準的な UI パターンが管理画面の一部にしかない
- ローディング表示、二重送信防止が未設計

### 改善提案
- 【高】商品画像を中心としたカード形式の商品一覧・詳細画面に変更
- 【高】注文確定ボタンの二重クリック防止とローディング表示
- 【高】得意先のスマートフォンからの注文を想定したレスポンシブ対応
- 【中】受注一覧の一括受付機能
- 【中】管理画面のダッシュボード画面
- 【中】在庫不足の視覚的アラート
- 【中】注文フローでの在庫リアルタイムチェック
- 【低】得意先マイページ（注文履歴、届け先管理）
- 【低】郵便番号 → 住所オートコンプリート
- 【低】商品一覧のソート機能

### 懸念事項
- 花屋のスタッフは朝の忙しい時間にシステムを使う。1 クリックでも減らせる設計が重要
- 得意先が自分の注文履歴を確認できない。「前に贈った花束と同じものを注文したい」に対応できない
- 注文途中のセッション切れでデータが消えるとユーザー離脱に直結

### スコープ外の発見
- 得意先の注文動機（記念日、感謝、お見舞いなど）に応じた商品カテゴリやフィルタがあると探しやすい
- 経営者ロールにビジネスインテリジェンス的な機能（売上推移、人気商品ランキング）があると価値が高い
