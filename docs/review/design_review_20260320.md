## 総合評価

全体として、10 件の設計ドキュメントはビジネス要件（インセプションデッキ・ユーザーストーリー）との整合性が高く、個人経営の花屋という事業規模に見合った現実的な設計判断がなされている。MVP の価値提案である「受注管理の効率化」「在庫推移の可視化による廃棄ロス削減」「リピーター体験向上」の 3 本柱に対し、設計が過不足なく対応している点を評価する。一方、段階的リリース（MVP / Phase 2 / Phase 3）との整合性においていくつかの不整合が確認された。

---

全体として、小規模 WEB ショップシステムとしてビジネス規模に見合った堅実な設計がなされています。ヘキサゴナルアーキテクチャの選定、ドメインモデルパターンの採用、ピラミッド形テスト戦略の適用は、「変更を楽に安全にできるソフトウェア」という目標に合致しています。一方で、ドキュメント間の整合性にいくつかの不一致があり、また一部の設計判断に技術的リスクが潜在しています。

---

全体として、ユーザーの業務フローをよく理解した上で設計されており、UI 設計の画面遷移やインタラクション設計はユースケース・ユーザーストーリーとの整合性が高いです。一方で、「ユーザーがこのシステムを使うとき、何を期待し、何に戸惑うか」という視点で深掘りすると、**注文フローにおける情報提示の不足**、**管理画面のナビゲーション一貫性の欠如**、**エラーからの回復導線の未設計箇所**がいくつか見つかりました。

---

全体として、テスト可能性を高く意識した設計がなされています。ヘキサゴナルアーキテクチャの採用、ドメインモデルパターンの選択、テストピラミッドの明示、TDD サイクルの定義など、テスタビリティの基盤は堅実です。一方で、受け入れ基準の具体性、エッジケースの網羅性、テスト戦略と各設計ドキュメント間の整合性について、いくつかの改善余地があります。

---

全体として、業務プロセスと利用者のニーズをよく理解した設計になっています。特に在庫推移の可視化、品質維持日数を考慮した発注支援、リピート顧客向けの届け先コピー機能など、現場の「痛み」に対応する設計が適切です。一方で、日常業務の「流れ」の中でいくつか引っかかるポイントがあり、実際に毎日使うスタッフの負担を軽減するための改善が必要です。

**プロダクトマネージャーの視点**

## 評価サマリー

全体として、10 件の設計ドキュメントはビジネス要件（インセプションデッキ・ユーザーストーリー）との整合性が高く、個人経営の花屋という事業規模に見合った現実的な設計判断がなされている。MVP の価値提案である「受注管理の効率化」「在庫推移の可視化による廃棄ロス削減」「リピーター体験向上」の 3 本柱に対し、設計が過不足なく対応している点を評価する。一方、段階的リリース（MVP / Phase 2 / Phase 3）との整合性においていくつかの不整合が確認された。

---

## 良い点

- **ビジネス規模に見合った技術選定**: マイクロサービスを避けたモノリシック構成、ECS Fargate の最小タスクサイズ（0.25 vCPU / 0.5 GB）からの開始、db.t3.micro の採用など、1〜2 名のチームで運用可能な「身の丈に合った」設計が徹底されている。YAGNI の原則に沿っており、過剰投資を避けている。

- **コアドメインの適切な識別**: 在庫推移計算、品質維持日数管理、受注状態遷移といったビジネスの競争優位性に直結するロジックがドメインモデルパターンで手厚く設計されている。特に `InventoryTransitionService` の計算式（`日別在庫予定数 = 前日在庫 + 当日入荷予定 - 当日受注引当 - 当日廃棄予定`）が明確に定義されており、ビジネス目標「廃棄ロス削減」を直接支える設計になっている。

- **「人間判断の尊重」の反映**: インセプションデッキのビジネスプリンシプルどおり、自動発注機能をスコープ外とし、在庫推移表示による「判断材料の提供」に徹する設計となっている。UI 設計の在庫推移画面（S-201）でも、内訳（入荷予定・受注引当・廃棄予定）を明示しており、仕入スタッフの発注判断を適切に支援している。

- **ロール別アクセス制御の充実**: 6 ロール（得意先・受注スタッフ・仕入スタッフ・フローリスト・配送スタッフ・経営者）のナビゲーション設計が業務実態と合致している。各ロールが必要な情報のみにアクセスできるよう RBAC が設計されている。

- **段階的リリースに配慮した UI フェーズ分割**: UI 設計の画面一覧で各画面にフェーズが明記されており（S-401 結束対象一覧は Phase 2、S-601 得意先一覧は Phase 3）、MVP では不要な画面を後回しにする判断ができている。

- **トレードオフの明示**: 非機能要件で品質特性間のトレードオフと優先順位が明確に定義されている。セキュリティ > 使用性 > 信頼性 > 保守性 > 性能 > 拡張性という順序は、「記念日に届ける」サービスの本質を捉えている。

## 改善提案

- **【重要度: 高】インセプションデッキのフェーズ定義とユーザーストーリーのフェーズ定義に不整合がある。** インセプションデッキでは「フェーズ 1（MVP）: 商品マスタ・受注・在庫推移表示」「フェーズ 2: 仕入管理・入荷管理・出荷管理」と定義されている。しかしユーザーストーリーでは US-010（発注）と US-011（入荷）が MVP に含まれており、出荷管理（US-012〜014）が Phase 2 になっている。このズレは開発チームと経営者の期待値のギャップにつながる。ユーザーストーリーの分類に合わせてインセプションデッキを更新するか、どちらを正とするかを明確にすべき。プロダクトマネージャーとしての判断: 在庫推移表示が MVP の中核であり、在庫推移の精度を確保するには発注・入荷も必須であるため、**ユーザーストーリー側の分類（発注・入荷を MVP に含む）が正しい**。インセプションデッキ側を修正すべき。

- **【重要度: 高】受注テーブルに数量カラムが存在しない。** データモデルの orders テーブルには `product_id`（商品 ID）はあるが、注文数量（quantity）がない。現在のモデルでは「1 注文 = 1 商品 1 個」が暗黙の前提となっている。これがビジネス要件として正しいのであれば、ドキュメントに明示すべき。花束は 1 回の注文で複数個を同じ届け先に送る場合がないのかを確認する必要がある。もしこれが意図的な設計であれば、ドメインモデルの不変条件に「1 受注 = 1 花束」を追加すること。

- **【重要度: 中】注文キャンセル時の在庫引当解除ロジックがドメインモデルに不足している。** ユーザーストーリー US-019 では「キャンセル時に在庫引当が解除される」が受入基準に含まれているが、ドメインモデルの Order 集約には `cancel()` メソッドのみで在庫引当解除のロジックが見当たらない。在庫推移計算サービスが受注引当を動的に計算しているため引当解除が不要な可能性もあるが、ここが曖昧だと実装時に混乱が生じる。在庫引当が「静的な割当」なのか「動的な推移計算」なのかをドメインモデル設計で明確化すべき。

- **【重要度: 中】バックエンドアーキテクチャに出荷ユースケースがあるが、MVP では不要。** `architecture_backend.md` のアプリケーション層に「出荷ユースケース」が既に含まれている。しかし出荷管理（結束・出荷処理）は Phase 2 のスコープである。MVP で実装不要な要素がアーキテクチャ図に含まれていると、チームが MVP のスコープを誤解するリスクがある。図にフェーズの注釈を追加するか、MVP 時点のアーキテクチャ図を別途用意すべき。

- **【重要度: 中】顧客向け画面の商品一覧がテーブル形式で、花屋のショップとしての訴求力が弱い。** UI 設計（S-003）の商品一覧はテーブル表示（商品名・価格・説明・操作）になっている。インセプションデッキのリスク #2 に「得意先の WEB リテラシーが低く、WEB 注文が定着しない」が挙げられているにもかかわらず、花束のビジュアルがない一覧では購買意欲が刺激されない。ビジネス目標「リピート率の向上」にも関わるため、MVP であっても商品画像のサムネイル表示は検討すべき。ただし、現在のデータモデルには products テーブルに画像カラムが存在しないため、対応する場合はデータモデルの変更も必要。

- **【重要度: 低】tech_stack.md の Node.js EOL 情報が矛盾している。** Node.js 20 LTS の EOL が「2026-04」、次回評価時期が「2025 Q4」と記載されている。現在は 2026-03-20 であり、評価時期が過去になっている。既に EOL が迫っている状態なので、Node.js 22 LTS への移行を技術スタックに反映すべき。

## 懸念事項

- **在庫推移計算のパフォーマンスリスク**: 在庫推移計算は複数テーブル（stocks, purchase_orders, orders, product_compositions）を跨いだ集計クエリが必要。非機能要件で API 応答時間 500ms を設定しているが、日付範囲が広い場合にこの目標を達成できるか不確実。在庫推移画面は仕入スタッフの発注判断の要であり、ビジネス価値の中核に直結するため、早期にプロトタイプレベルでパフォーマンス検証を行うべき。

- **廃棄在庫の自動ステータス更新トリガーが未定義**: stocks テーブルに `expiry_date` と `status`（available/degraded/expired）があるが、これをいつ・どのように更新するかがドメインモデルにも運用要件にも明記されていない。日次バッチなのか、在庫参照時にオンデマンドで計算するのかを決める必要がある。これは廃棄ロス削減のビジネス目標に直結する部分であり、曖昧なまま実装に進むべきではない。

- **JWT Access Token の有効期限 24 時間に Refresh Token がない**: 非機能要件では Access Token の有効期限が 24 時間と定義されているが、Refresh Token の仕組みが言及されていない。24 時間の有効期限は小規模システムとしては妥当だが、セキュリティ上は Access Token を短時間（15 分〜1 時間）にして Refresh Token で更新する方式が一般的。MVP ではこの方式でよいが、セキュリティ優先度 1 位の方針との整合性として、将来的な改善計画に含めるべき。

- **テスト戦略のミューテーションスコア目標がドキュメント間で不一致**: `test_strategy.md` ではミューテーションスコア目標が「60% 以上」、`non_functional.md` では「70% 以上（ドメイン層）」と記載されている。どちらを基準とするかを統一する必要がある。

---

## スコープ外の発見

- **インセプションデッキのフェーズ別ロードマップでスケジュール上の矛盾**: ガントチャートでは「分析フェーズ 2 週間 → フェーズ 1（MVP）5 週間 → フェーズ 2 3 週間 → フェーズ 3 3 週間」となっているが、インセプションデッキのフェーズ 2 の説明が「仕入管理・入荷管理・出荷管理」であるのに対し、ユーザーストーリーのフェーズ 2 は「出荷管理・届け日変更・キャンセル」となっている。スケジュール見積もりの前提が崩れている可能性がある。リリース計画ドキュメントの作成時にこの不整合を解消すべき。

- **仕入先マスタの管理画面が設計に含まれていない**: データモデルに suppliers テーブルが定義されており、発注時に仕入先情報を使用するが、仕入先を登録・編集する管理画面がUI設計に存在しない。初期データのセットアップ方法（DB 直接投入 or 管理画面）を決定する必要がある。MVP では DB 直接投入でもよいが、運用中の仕入先変更（連絡先変更等）を考慮すると、フェーズ 2 以降で管理画面を追加するか、または運用手順書に含める必要がある。

**アーキテクトの視点**

### 評価サマリー

全体として、小規模 WEB ショップシステムとしてビジネス規模に見合った堅実な設計がなされています。ヘキサゴナルアーキテクチャの選定、ドメインモデルパターンの採用、ピラミッド形テスト戦略の適用は、「変更を楽に安全にできるソフトウェア」という目標に合致しています。一方で、ドキュメント間の整合性にいくつかの不一致があり、また一部の設計判断に技術的リスクが潜在しています。

---

### 良い点

- **アーキテクチャ選定の妥当性**: ドメインモデルパターン + ヘキサゴナルアーキテクチャの選定理由が明確であり、ADR-001 で代替案の却下理由も含めて記録されています。「CQRS は不要」「マイクロサービスは運用複雑性が高すぎる」など、過剰設計を避ける判断が適切です。

- **ドメインモデルの集約設計**: 集約境界が明確に定義されており、不変条件が具体的に列挙されています。値オブジェクト（`QualityRetentionDays`, `PurchaseUnit`, `LeadTime` など）が業務概念をよく反映しており、ユビキタス言語も整理されています。

- **在庫推移計算の定式化**: 「日別在庫予定数 = 前日在庫 + 当日入荷予定 - 当日受注引当 - 当日廃棄予定」という計算式がドメインサービスとして明確に定義されており、テスト可能な仕様になっています。

- **テスト戦略の一貫性**: バックエンドのインサイドアウト TDD とフロントエンドのアウトサイドイン TDD を使い分ける方針が明確です。テストピラミッドの比率（70/20/10）も妥当です。

- **非機能要件の定量化**: ISO/IEC 25010 に基づき、応答時間・スループット・可用性のすべてに具体的な目標値と測定方法が定義されています。品質特性間のトレードオフも明示されています。

- **UI 設計の具体性**: PlantUML の Salt 記法で画面イメージが具体的に表現されており、インタラクション設計やエラー処理のシナリオも網羅的です。ユースケースとの対応関係も明確です。

- **運用設計の現実性**: 1〜2 名の運用体制に見合った日次・週次・月次の運用チェックリストが定義されており、障害レベル定義とエスカレーションフローも実用的です。

### 改善提案

- **【重要度: 高】受注テーブルに数量カラムが存在しない**
  - データモデル (`data-model.md`) の `orders` テーブルに注文数量のカラムがありません。ユースケース上は 1 注文 = 1 商品（花束）の前提に見えますが、これが明示的に記述されていません。もし同じ花束を複数個注文するケースがあるなら `quantity` カラムが必要です。明示的に「1 注文 = 1 花束」の制約をドメインモデルの不変条件として記述すべきです。

- **【重要度: 高】受注の「届け完了」への状態遷移が API に未定義**
  - データモデルで受注ステータスに `delivered`（届け完了）が定義されていますが、バックエンドの API エンドポイントに「届け完了」への更新 API がありません。ユースケースにも `deliver()` メソッドがドメインモデルに定義されていますが、それを呼び出す UC が欠落しています。配送完了をどう記録するのか（手動 / 自動）の設計が必要です。

- **【重要度: 高】入荷と在庫の連動ロジックが不明確**
  - `arrivals`（入荷）テーブルと `stocks`（在庫）テーブルの関係が設計上不明確です。入荷登録時に在庫レコードがどう作成されるのか（1 入荷 = 1 在庫ロットなのか）、`stocks.arrived_date` は `arrivals.arrived_at` から導出されるのかなど、ドメインサービスまたはアプリケーションサービスのレベルで明文化すべきです。

- **【重要度: 高】JWT の Refresh Token 戦略が未定義**
  - 非機能要件で「Access Token 有効期限 24 時間」と定義されていますが、Refresh Token の仕組みが言及されていません。24 時間は Access Token としては長く、トークン漏洩時のリスクが高まります。短い有効期限の Access Token + Refresh Token のパターンを採用するか、24 時間を正当化する理由（小規模で低リスク）を ADR として記録すべきです。また、非機能要件では「Token をメモリ保持」としていますが、SPA でページリロード時にログアウトされる問題への対処が未設計です。

- **【重要度: 中】ドメインモデルの `DeliveryDestination` の二重定義**
  - ドメインモデル上、`DeliveryDestination` が「得意先集約」内の値オブジェクトとして定義されている一方、`Order` 集約でも `deliveryDestination` フィールドとして使われています。しかし、データモデルでは `orders.delivery_destination_id` で FK 参照しています。値オブジェクトとしてコピーするのか、エンティティとして参照するのかが混在しています。集約間の参照には ID による参照を一貫させるべきです。

- **【重要度: 中】フロントエンドアーキテクチャに CloudFront が未記載**
  - 非機能要件 (`non_functional.md`) の対象システム構成に「CloudFront」が記載されていますが、インフラストラクチャ設計 (`architecture_infrastructure.md`) の構成図には CloudFront が含まれていません。SPA の配信戦略として S3 + CloudFront を使うのか、ECS Fargate から配信するのかが不明確です。整合性を合わせる必要があります。

- **【重要度: 中】stocks テーブルの同時実行制御が未設計**
  - 在庫の消費（結束時）や入荷登録が同時に行われた場合の排他制御が設計されていません。楽観ロック（バージョニング）や悲観ロックの方針をデータモデルまたはドメインモデルに明記すべきです。特に `Stock.consume()` 操作は競合リスクがあります。

- **【重要度: 中】バッチ処理の設計が未定義**
  - 非機能要件で「日次データ集計: < 30 秒」「月次レポート生成: < 60 秒」が定義されていますが、バッチ処理のアーキテクチャ（Spring Batch? スケジューラ? ECS Scheduled Task?）がどの設計ドキュメントにも記載されていません。運用要件の「廃棄予定在庫の通知」「在庫不足アラート」もバッチ的な処理が必要ですが、実装方式が未決定です。

- **【重要度: 中】Node.js の EOL 対応が遅れている**
  - 技術スタック (`tech_stack.md`) で Node.js 20 LTS の EOL が「2026-04」、次回評価時期が「2025 Q4」となっていますが、現在は 2026-03-20 であり、すでに評価時期を過ぎています。来月 EOL を迎えるため、Node.js 22 LTS への移行を早急に計画する必要があります。

- **【重要度: 低】テスト戦略の ミューテーションスコア目標値に不一致**
  - テスト戦略 (`test_strategy.md`) ではミューテーションスコア目標が「60% 以上」ですが、非機能要件 (`non_functional.md`) では「70% 以上（ドメイン層）」となっています。ドメイン層に限定した目標か全体の目標かの違いかもしれませんが、整合性を確認して統一すべきです。

- **【重要度: 低】商品画像の扱いが不明確**

### 懸念事項

- **在庫推移計算のパフォーマンスリスク**: 在庫推移計算は複数の集約（Stock, PurchaseOrder, Order, ProductComposition）にまたがるクエリが必要です。表示期間が長い場合や単品数が多い場合に、N+1 問題やクエリ数の増大が懸念されます。統合テストで早期にパフォーマンスを検証すべきです。

- **ECS Fargate の最小構成のリスク**: インフラ設計で「0.25 vCPU / 0.5 GB」の最小タスクサイズが指定されていますが、Spring Boot + JPA のアプリケーションにとって 0.5 GB メモリは厳しい可能性があります。JVM のヒープサイズ設定とコンテナメモリの関係を事前に検証すべきです。

- **ドメイン層の `@Entity` 依存**: ADR-001 で「ドメイン層のクラスが `@Entity` 以外の Spring / JPA アノテーションに依存していないことを検証する」としていますが、`@Entity` 自体はドメイン層に JPA の知識を持ち込んでいます。ヘキサゴナルアーキテクチャの純粋性を追求するならドメインモデルと JPA エンティティを分離すべきですが、小規模プロジェクトでは妥協として許容可能です。このトレードオフを ADR に明記すべきです。

- **ロール管理の拡張性**: `users.role` が VARCHAR(20) の単一値で設計されていますが、経営者が全機能にアクセスできることを考えると、将来的に複数ロールの付与が必要になる可能性があります。現状の単一ロール設計で十分かどうかを検討すべきです。

- **論理削除（is_active フラグ）の運用課題**: 商品・単品テーブルに `is_active` フラグがありますが、受注済みの商品が論理削除された場合の表示方針（受注詳細画面で何を表示するか）が未設計です。

### スコープ外の発見

- **ユースケースの `UC-011: 届け先をコピーする` が delivery_destinations テーブルの設計と不整合**: UC-011 では「過去の注文で使用した届け先を再利用する」ことが記述されていますが、データモデルでは `delivery_destinations` は `customers` に紐づいているため、注文からの逆引きではなく得意先の届け先一覧からの選択になります。ユースケースの記述とデータモデルの設計意図を合わせる必要があります。

- **監査ログのデータモデルが未定義**: 非機能要件と運用要件で「操作履歴 — データ変更操作を記録（誰が・いつ・何を）」「監査ログテーブルの検証」が定義されていますが、データモデルに監査ログ用のテーブルが存在しません。

---

- **Container / Presentational パターン**（`architecture_frontend.md`）: Presentational コンポーネントが Props のみで制御されるため、ユニットテストで外部依存なしにレンダリング結果を検証できます。

- **テストピラミッドの明示的な比率設定**（`test_strategy.md`）: 70:20:10 の比率が根拠とともに定義されており、チーム全体でテストレベルの判断基準を共有できます。

- **テストデータ戦略の定義**（`test_strategy.md`）: テストレベルごとにデータ管理方針が明確で、Builder パターンやファクトリメソッドの使用方針が示されています。

- **インサイドアウト / アウトサイドインの TDD アプローチの使い分け**（`test_strategy.md`）: バックエンドとフロントエンドでそれぞれ適切な TDD アプローチが選択されています。

- **非機能要件の定量的な目標値と測定方法**（`non_functional.md`）: すべての品質特性に対して具体的な目標値と測定方法が定義されており、テストの合否判定基準として使えます。

- **ステータス遷移の明示**（`data-model.md`, `domain_model.md`）: `OrderStatus` と `PurchaseOrderStatus` の遷移が定義されており、状態遷移テストの設計が可能です。

### 改善提案

#### 1. テスト戦略とドメインモデルの整合性

- **【重要度: 高】テスト戦略にドメインモデルの不変条件に対するテストケース一覧がない**。`domain_model.md` で定義された不変条件（6 集約分）は、テストケースとして明示的にマッピングされるべきです。例えば、「発注数量は購入単位の倍数」「入荷合計が発注数量を超えない」といった不変条件に対して、正常系・異常系・境界値のテストシナリオが `test_strategy.md` に必要です。

  **提案**: `test_strategy.md` に「不変条件テストマトリクス」セクションを追加し、各集約の不変条件ごとにテストケース（正常系、境界値、異常系）を列挙する。

- **【重要度: 高】在庫推移計算サービスのテストシナリオが不十分**。`domain_model.md` で「日別在庫予定数 = 前日在庫 + 当日入荷予定 - 当日受注引当 - 当日廃棄予定」と定義されていますが、`test_strategy.md` ではテスト対象例として1行で言及されているだけです。この計算は複数の集約にまたがるドメインサービスであり、テストの複雑性が高いため、以下のシナリオが必要です:
  - 入荷予定がない日の在庫推移
  - 複数の受注引当が重なる日の在庫推移
  - 廃棄予定と入荷予定が同日に発生する場合
  - 在庫がマイナスになるケース（不足検出）
  - 品質維持日数の境界値（期限当日、期限翌日）

#### 2. 受け入れ基準の明確化

- **【重要度: 高】UI 設計のインタラクション仕様にテスト可能な受け入れ基準が不足している**。`ui-design.md` のインタラクション設計セクションで、例えば「届け日を選択（カレンダーピッカー、翌日〜30 日後のみ選択可能）」とありますが、以下の受け入れ基準が曖昧です:
  - 「翌日」の起算はサーバー時刻かクライアント時刻か
  - 「30 日後」は営業日ベースか暦日ベースか
  - タイムゾーンの扱い（JST 前提？UTC?）
  - 日付変更のタイミング（深夜 0 時で切り替わるか）

  **提案**: 各画面の入力項目に対して、バリデーションルールを Given-When-Then 形式で明記する。

- **【重要度: 中】エラー処理の受け入れ基準が曖昧**。`ui-design.md` の「エラー処理・フィードバック」テーブルで「在庫不足（届け日指定時）→ 対応可能な最寄りの届け日を提案」とありますが、「最寄り」の定義が不明確です。何日先まで探索するのか、複数候補を提示するのか、在庫推移計算の結果をどう表示するのかが、テストケース設計に必要です。

#### 3. エッジケースの網羅性

- **【重要度: 高】OrderStatus の状態遷移図で不正遷移のテストケースが不足**。`domain_model.md` に `canTransitionTo(next): boolean` メソッドが定義されていますが、すべての不正遷移パターン（6 ステータス x 6 ステータス = 36 パターンのうち許可される遷移以外）を網羅するテストケースがどこにも列挙されていません。状態遷移テスト技法を適用し、全遷移パターンのマトリクスを作成すべきです。

- **【重要度: 中】データモデルの境界値が部分的にしか定義されていない**。`data-model.md` で `price` は「1〜999,999」、`quality_retention_days` は「1〜30」、`lead_time_days` は「1〜14」と定義されていますが、以下の境界値が未定義です:
  - `quantity`（商品構成、発注、入荷、在庫）: 最小値・最大値の制約が不明
  - `message`: VARCHAR(200) とあるが 0 文字（空）は許可されるか
  - `postal_code`: VARCHAR(10) のフォーマット制約（ハイフン有無等）
  - `email`: フォーマットバリデーションの具体的なルール
  - `phone`: フォーマットバリデーションの具体的なルール

  **提案**: `data-model.md` の各カラムに対してバリデーションルール列を追加する。

- **【重要度: 中】同時実行・競合状態のテストシナリオが欠落**。在庫の消費（`Stock.consume(qty)`）や受注のステータス更新など、複数ユーザーが同時操作した場合の排他制御に関するテストシナリオが設計ドキュメントのどこにも記述されていません。楽観的ロック（`@Version`）の使用有無、デッドロック防止策のテスト方針を明記すべきです。

- **【重要度: 低】ロール別アクセス制御の網羅的テストマトリクスがない**。`non_functional.md` と `ui-design.md` でロール別の権限が定義されていますが、6 ロール x 14 エンドポイント（`architecture_backend.md`）のアクセス制御マトリクスが明示されていません。

#### 4. テスト戦略の整合性

- **【重要度: 中】テストピラミッドの比率がドキュメント間で不一致**。`architecture_backend.md` では「ユニットテスト 70%、統合テスト 20%、E2E テスト 10%」、`test_strategy.md` でも同じ比率ですが、テスト戦略の「テスト種別定義」ではバックエンドとフロントエンドで別々のテスト種別（コンポーネントテスト等）が定義されており、フロントエンドの比率が明記されていません。フロントエンドのユニットテスト : コンポーネントテスト : E2E テストの比率を追記すべきです。

- **【重要度: 中】非機能要件のミューテーションテストスコア目標値が不一致**。`test_strategy.md` では「ミューテーションスコア 60% 以上」、`non_functional.md` では「ミューテーションテストスコア 70% 以上（ドメイン層）」と記載されており、不一致があります。統一が必要です。

- **【重要度: 低】E2E テストのスコープ定義が分散している**。`test_strategy.md` に「API レベル E2E」と「ブラウザ E2E」の2種類が定義されていますが、`architecture_infrastructure.md` の CI/CD パイプラインでは単に「E2E テスト」としか記載されておらず、ステージング環境で実行される E2E テストがどちらを指すのか不明確です。

#### 5. テスト自動化の実現可能性

- **【重要度: 中】ArchUnit テストの言及はあるが具体的なルールがない**。`non_functional.md` で「ArchUnit テスト」による層間依存の検証が述べられていますが、具体的にどのような依存ルールを検証するかが定義されていません。ヘキサゴナルアーキテクチャの依存方向ルール（ドメイン層はインフラ層に依存しない等）を ArchUnit のテストルールとして明示すべきです。

- **【重要度: 中】パフォーマンステストの自動化方針が不明確**。`non_functional.md` で「負荷テスト（k6 / JMeter）」が測定方法として記載されていますが、これが CI/CD パイプラインに組み込まれるのか、手動実行なのか、実行頻度はどうするのかが不明です。

- **【重要度: 低】アクセシビリティテストの自動化スコープが曖昧**。`non_functional.md` で axe-core による自動テスト、キーボード操作は手動テストと記載されていますが、axe-core をどのテストレベルで実行するか（コンポーネントテスト? E2E テスト?）が未定義です。

### 懸念事項

- **テスト実行時間の現実性**: `test_strategy.md` でユニットテスト全体 30 秒以内、統合テスト 3 分以内を目標としていますが、Testcontainers を使用した統合テストでは PostgreSQL コンテナの起動に 10〜30 秒かかります。テストの並列実行やコンテナの再利用戦略（`@Testcontainers` のシングルトンコンテナ）を検討しないと、テストケースが増えた際に目標時間の達成が困難になる可能性があります。

- **フロントエンド E2E テストの安定性リスク**: `test_strategy.md` のアウトサイドインアプローチでは E2E テストからフロントエンド開発を開始しますが、バックエンド API が未実装の段階での E2E テストの実行方法（MSW でモックするのか、API の契約テストを先に定義するのか）が不明確です。フロントエンドとバックエンドの開発順序とテスト戦略の整合性を明確にする必要があります。

- **テストデータの品質維持日数シナリオ**: 在庫の品質維持日数は時間の経過で状態が変わる（`AVAILABLE` → `DEGRADED` → `EXPIRED`）ため、テスト内で「現在日時」を制御する仕組み（Clock パターンやタイムフリーズ）が必要です。この方針が設計ドキュメントに含まれていません。テスト内で `java.time.Clock` を DI で差し替える設計をドメインモデルに組み込むべきです。

- **JWT Access Token の有効期限 24 時間**: `non_functional.md` で Access Token の有効期限が 24 時間と設定されていますが、Refresh Token の仕組みが記載されていません。セッション管理のテストシナリオ（トークン期限切れ時の振る舞い、トークンリフレッシュの動作）が不明確です。

- **監査ログのテスト方針が未定義**: `non_functional.md` の監査要件で「データ変更操作を記録（誰が・いつ・何を）」と定義されていますが、監査ログテーブルの存在がデータモデル（`data-model.md`）に含まれておらず、テスト方針も未定義です。

## 評価サマリー

全体として、業務プロセスと利用者のニーズをよく理解した設計になっています。特に在庫推移の可視化、品質維持日数を考慮した発注支援、リピート顧客向けの届け先コピー機能など、現場の「痛み」に対応する設計が適切です。一方で、日常業務の「流れ」の中でいくつか引っかかるポイントがあり、実際に毎日使うスタッフの負担を軽減するための改善が必要です。

## 良い点

- **在庫推移画面（S-201）の設計が秀逸です。** 前日在庫・入荷予定・受注引当・廃棄予定・在庫予定の内訳を日別で一覧表示できるのは、仕入スタッフが毎朝「今日はどの花材を発注すべきか」を判断するのに必要十分な情報です。類似システムではここが数字の羅列になりがちですが、この設計は判断に必要な情報が一画面に収まっています。

- **受注状態遷移の設計が業務実態に合っています。** 「出荷準備中」以降はキャンセル・届け日変更不可というルールは、花材を結束してしまった後に取り消すことが物理的に困難という現場の実態を正しく反映しています。

- **ドメインモデルのユビキタス言語が現場の用語と一致しています。** 「得意先」「結束」「品質維持日数」「購入単位」といった用語は、花屋の現場で実際に使う言葉そのものです。エンジニア用語に置き換えられていない点が良いです。

- **ロール別ナビゲーション設計が適切です。** 受注スタッフには受注管理だけ、仕入スタッフには在庫管理と発注管理だけ、というシンプルなメニュー構成は、各スタッフが「自分の仕事」に集中できます。

- **在庫推移画面から発注画面への直接遷移（S-201→S-301）は業務フローに沿っています。** 在庫不足を確認したらそのまま発注する、というのが日常の流れです。

- **1 注文 = 1 商品ルールの明確な定義は正しい判断です。** 花束は個別に結束・出荷するため、複数商品をまとめる必要がありません。

- **フェーズ分割の優先順位が適切です。** MVP で受注・在庫・発注の基盤を作り、Phase 2 で出荷管理を追加する判断は、「まずは注文を受けて花材を管理できること」が最優先という業務の現実に合っています。

## 改善提案

### UI 設計

- **【重要度: 高】 商品一覧画面（S-003）に商品画像の表示欄がありません。** 花束は「見た目」で選ぶ商品です。名前・価格・説明だけのテーブル表示では、得意先が「どの花束を贈ろうか」を判断できません。カード型レイアウトで商品画像を大きく表示し、その下に名前・価格を配置するのが自然です。競合の花屋の EC サイトはすべて画像中心のデザインです。データモデル（products テーブル）にも画像 URL カラムがないため、合わせて追加が必要です。

- **【重要度: 高】 注文画面（S-005）で数量指定ができません。** 同じ花束を複数個注文するケースがあります。例えば、会社の記念パーティーで来賓 3 名にそれぞれ花束を贈りたい場合、同じ花束を 3 回注文しなければなりません。「1 注文 = 1 商品」ルールは理解していますが、せめて数量を指定して一度に複数受注を作成できる仕組みがあると、得意先の手間が大幅に減ります。あるいは、1 注文に数量フィールドを追加して「1 注文 = 1 商品 x N 個」にする方が業務上自然です。

- **【重要度: 高】 得意先が自分の注文履歴を確認できる画面がありません。** 得意先が「先月注文した花束はいつ届いたかな」「今注文しているものの状態はどうなっているだろう」と確認したいのは自然な要求です。顧客向け画面に「マイページ」や「注文履歴」画面を追加してください。これがないと、問い合わせがすべて電話になり、受注スタッフの負担が増えます。

- **【重要度: 中】 受注一覧画面（S-101）で届け日が迫っている受注の視覚的な警告がありません。** 受注スタッフは毎朝、「今日受け付けなければ間に合わない注文」を最優先で処理します。届け日が翌日や翌々日の未処理受注には色や警告アイコンで目立たせてほしいです。類似システムでは、この「見落とし防止」がないために届け遅れが発生した経験があります。

- **【重要度: 中】 在庫推移画面（S-201）で在庫がマイナスになる日をアラート表示してほしいです。** 現在の設計では在庫予定数がボールドで表示されるだけですが、マイナスになる日は赤字やアラートアイコンで強調してほしいです。仕入スタッフが毎朝見る画面なので、「問題のある箇所」が一目でわかることが重要です。

- **【重要度: 中】 結束対象一覧画面（S-401）の「必要花材サマリ」は非常に良いのですが、花材の「在庫ロット」ごとの品質状態が見えるとさらに良いです。** フローリストは結束時に「鮮度の良い花材」を選びたいので、「在庫数 27 本」のうち「5 日前入荷: 10 本、2 日前入荷: 17 本」のようにロット情報があると助かります。ただし、これは Phase 2 以降の改善でも構いません。

- **【重要度: 低】 商品詳細画面（S-004）で花束の構成花材を見せるのは良いですが、得意先にとって「バラ（赤）3 本」より「赤いバラをメインに、かすみ草を添えた華やかなアレンジ」のような表現のほうが響きます。** 構成花材の一覧は残しつつ、花束のイメージ説明を充実させることを検討してください。

### データモデル

- **【重要度: 高】 orders テーブルに数量（quantity）カラムがありません。** 前述の通り、同じ花束を複数注文するケースへの対応が必要です。

- **【重要度: 高】 products テーブルに画像関連のカラム（image_url 等）がありません。** 花束の商品は画像なしでは販売できません。

- **【重要度: 中】 orders テーブルに注文キャンセル理由のカラムがありません。** キャンセル理由を記録しておくことで、経営者が「なぜキャンセルが発生したか」を分析し、サービス改善に活かせます。

- **【重要度: 低】 delivery_destinations テーブルに「メモ」フィールドがあると便利です。** 例えば「マンション 305 号室、オートロックの暗証番号は...」「在宅時間は午前中」といった配送に役立つ情報を記録できます。

### ドメインモデル

- **【重要度: 中】 受注の状態遷移で「届け完了」への遷移が配送スタッフの操作のみで行われる設計ですが、実際には配送業者に委託するケースもあり、「届け完了」の確認方法が不明確です。** 自社配送なのか外部委託なのかによって、この状態遷移の仕組みが変わります。業務上の判断が必要です。

- **【重要度: 中】 在庫の状態モデルに「引当済み」がありますが、データモデル（stocks テーブル）の status カラムの値定義には「available/degraded/expired」しかなく、「引当済み」（allocated）が含まれていません。** 要件定義では受注登録時に論理引当を行うと書かれているので、整合性を取る必要があります。

### バックエンドアーキテクチャ

- **【重要度: 中】 API エンドポイントに「得意先自身の注文履歴取得」がありません。** `GET /api/v1/orders`は受注スタッフ向けの受注一覧ですが、得意先が自分の注文一覧を取得する API（例: `GET /api/v1/my/orders`）が必要です。

### 非機能要件

- **【重要度: 中】 モバイル対応が「MVP では対応不要」となっていますが、得意先（個人顧客）がスマートフォンから注文するケースは非常に多いはずです。** 特に「贈り物の花束を注文する」というシーンでは、外出先や帰宅途中にスマートフォンで注文するのが自然です。管理画面はデスクトップ優先で問題ありませんが、顧客向け画面は早期にモバイル対応すべきです。Phase 2 での対応を提案します。

- **【重要度: 低】 Access Token の有効期限が 24 時間ですが、これは得意先にとっては短いかもしれません。** 1 週間に 1 回注文するリピーター客が、毎回ログインし直す必要があるのは手間です。得意先向けには Refresh Token による延長、またはパスワードレス認証（メールリンク等）の将来検討が望ましいです。

### 運用要件

- **【重要度: 中】 メンテナンスウィンドウが「日曜 2:00〜6:00」ですが、営業日が「月〜土」で、得意先の注文受付は 24 時間です。** 日曜深夜のメンテナンスは妥当ですが、お盆・年末年始・母の日の前などの繁忙期にはメンテナンスを避ける配慮が必要です。特に母の日（5 月第 2 日曜日）は花屋にとって最大の商戦期で、日曜にメンテナンスを入れるわけにはいきません。繁忙期カレンダーの定義と、繁忙期のメンテナンス凍結ルールが必要です。

## 懸念事項

- **商品画像が設計に一切含まれていないことが最大の懸念です。** 花束という商品の性質上、画像なしの EC サイトは成立しません。データモデル・UI 設計・インフラ（S3 の画像ストレージ設計）のすべてにおいて、商品画像の扱いを設計に追加する必要があります。これは MVP で対応必須です。

- **顧客向け画面のモバイル非対応は売上機会の損失につながる可能性があります。** 個人顧客がターゲットの花屋 EC で、スマートフォンから注文できないのは致命的になり得ます。「MVP では不要」の判断根拠を再確認すべきです。

- **得意先が自分の注文状況を確認できる手段がありません。** これがないと「注文は届きましたか？」「今どんな状態ですか？」という電話問い合わせが増え、受注スタッフの業務負荷が増大します。WEB ショップの目的が「手作業の限界の解消」であるなら、顧客自身がセルフサービスで確認できる仕組みは必須です。

- **在庫推移計算で「品質低下」（degraded）状態の在庫をどう扱うかの業務ルールが不明確です。** 要件定義の在庫状態モデルでは「品質低下」という状態がありますが、在庫推移計算のロジックでは「廃棄予定」（expiry_date が当日）のみ考慮されています。品質が低下した花材を結束に使うかどうかは、フローリストの判断になりますが、在庫推移画面でこの状態が見えないと判断材料になりません。

- **発注がシステム上の記録のみで、仕入先への実際の発注手段（FAX、メール、電話）との連携が考慮されていません。** 現実の花屋では、仕入先への発注は電話や FAX で行うことが多いです。システムで発注を登録した後、実際に仕入先に連絡する手段との関係を明確にすべきです。現時点では「システムは発注の記録管理のみ、実際の連絡は別途行う」という前提であれば問題ありませんが、UI 上でその旨を明示すべきです。
