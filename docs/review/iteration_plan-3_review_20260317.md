# コードレビュー結果 — イテレーション 3 計画

## レビュー対象

- `docs/development/iteration_plan-3.md`（新規作成）

## 総合評価

全 5 エージェントが計画の品質を高く評価しました。IT2 ふりかえりの反映、TDD による段階的実装戦略、ヘキサゴナルアーキテクチャの一貫した適用が優れています。一方、**設計ドキュメントとの不整合**（Supplier の leadTimeDays/purchaseUnit の配置先）、**在庫推移計算テストの具体性不足**、**在庫推移画面の内訳表示の欠如**が複数エージェントから重複して指摘されました。スケジュール面では木曜に S09 全タスク 9.5h を詰め込んでいる点がリスクです。

---

## 改善提案（重要度順）

### 高（計画修正推奨）

| # | 提案 | 箇所 | 指摘元 | 理由 |
|---|------|------|--------|------|
| H1 | Supplier テーブルへの leadTimeDays/purchaseUnit 追加を取りやめ、Item から取得する設計に変更（または ADR で決定） | タスク 0.4、Prisma スキーマ | architect | ドメインモデル設計書・データモデル設計書では Item の属性として定義済み。設計不整合 |
| H2 | StockForecastService のタスク 1.1（3h）を 4 つのベビーステップに分解 | タスク 1.1 | programmer | 3h は Red-Green-Refactor サイクルが大きすぎる。段階的実装の方針とタスク粒度を一致させる |
| H3 | 在庫推移計算の境界値テストケースをタスクに明示 | タスク 1.1 | tester | 在庫 0 境界、負の在庫、同日の入荷と出荷、品質維持日数 0 日等のテストシナリオが未定義 |
| H4 | トランザクション ADR（T7）をタスク一覧に追加 | タスク 0 | programmer | IT2 で P1（トランザクション管理漏れ）が深刻度「高」。タスクに含まれていない |
| H5 | arrivals テーブルを IT3 スコープから除外（YAGNI） | タスク 0.4 | programmer, architect | IT3 では入荷処理はスコープ外。未使用テーブルのマイグレーション不要 |
| H6 | 在庫推移画面（P07）に引当数・入荷予定数の内訳を表示（ツールチップ可） | P07 UI 設計 | user-rep | 仕入スタッフの発注判断には「なぜその数字か」の内訳が不可欠。V1.2 レビューからの継続課題 |
| H7 | リリース計画の Phase 割り当てとの不整合を解消（S09 は Phase 2 だが MVP 検証に含まれている） | ゴール、MVP 検証 | writer, user-rep | 読者の混乱を防ぐ。リリース計画を更新するか、MVP 検証を S08 のみに限定 |
| H8 | E2E テストを 2 件 → 3 件に増加（在庫推移→発注→確定→在庫推移反映の一連フロー追加） | タスク 3 | tester | S08→S09 の画面連携フローが E2E でカバーされていない |

### 中（対応推奨）

| # | 提案 | 箇所 | 指摘元 | 理由 |
|---|------|------|--------|------|
| M1 | 木曜の S09 全タスク 9.5h を水曜 PM から分散 | スケジュール | programmer, tester | 1 日の理想時間（5-6h）を大幅に超過。スケジュールリスク |
| M2 | PurchaseQuantityService を PurchaseOrder.createNew() のファクトリメソッドに吸収 | ドメインモデル | architect | 複数集約を横断しない計算はドメインサービスにする必然性なし。集約の凝集度向上 |
| M3 | StockForecastService の設計方針を選択 A（純粋ドメインサービス）/ B（Query Service）で明示 | 設計 | architect | ドメインサービスと StockForecastQuery の責務が重複。設計意図を ADR で記録 |
| M4 | StockForecast を domain/stock-forecast/ に独立配置 | ディレクトリ | architect | stock 集約の境界を保つ。StockForecastService は複数集約を横断する |
| M5 | PurchaseOrder の状態遷移に不正遷移テスト（入荷済み→発注済み）を明示 | タスク 2.1 | programmer, tester | IT2 で Order の不正遷移テスト漏れが問題になった教訓 |
| M6 | 在庫不足警告の閾値を固定値 5 ではなく「在庫予定数 <= 0（欠品日）」に変更 | P07 UI | user-rep | 単品ごとに適正在庫水準は異なる。欠品日なら全単品共通で使える |
| M7 | 発注画面（P08）に遷移元の在庫推移情報（現在庫・欠品予定日）を表示 | P08 UI | user-rep | 発注数量の判断に在庫推移情報が不可欠 |
| M8 | 発注確定後の遷移（在庫推移画面に戻り入荷予定反映を確認）をタスクに明示 | タスク 2.8 | user-rep | 逆方向の遷移と成功メッセージが未記載 |
| M9 | isExpiryWarning と「在庫不足警告」を区別して設計 | StockForecast | tester | 品質維持超過と在庫不足は異なる概念だが混在している |
| M10 | ADR セクションをテンプレートに合わせて追加 | 計画全体 | writer | T7 の ADR 作成予定を計画書に明記 |

### 低（改善の余地あり）

| # | 提案 | 箇所 | 指摘元 | 理由 |
|---|------|------|--------|------|
| L1 | T6（customerId 動的取得）を IT5 対応として注記 | IT2 ふりかえり反映表 | programmer | 網羅性の明確化 |
| L2 | 在庫不足警告閾値を定数として定義 | タスク 1.6 | programmer | マジックナンバー回避 |
| L3 | 期間フィルタのデフォルト値を「今日〜7 日後」に設定 | P07 UI | user-rep | 仕入スタッフの業務サイクルに合わせる |
| L4 | 関連ドキュメントに IT2 計画へのリンク追加 | 関連ドキュメント | writer | IT2 ふりかえり反映の参照性向上 |
| L5 | Supplier スキーマの差分追加であることを注記 | Prisma スキーマ | writer, architect | phone 列の欠落が混乱を招く |

## 矛盾事項

| # | 視点 A | 視点 B | 論点 | 推奨判断 |
|---|--------|--------|------|----------|
| 1 | architect: StockForecastService を純粋ドメインサービスに | tester: 計算ロジックのテスト容易性 | StockForecastQuery をインフラに置くか、ドメインサービスに集約するか | **選択肢 A（純粋ドメインサービス）** — テスト容易性とビジネスルールの集約化を優先 |
| 2 | architect: PurchaseQuantityService は集約に吸収可能 | programmer: ドメインサービスとして分離は単一責任に沿う | 計算ロジックの配置先 | **集約ファクトリに吸収** — 複数集約を横断しない計算はドメインサービスにする必然性なし |

IT1/IT2 の計画書フォーマットを踏襲した高品質な計画。TDD サイクルへの言及、ふりかえり反映、リスク分析が適切。

### 主要指摘
- タスク 1.1（3h）を 4 つのベビーステップに分解すべき
- トランザクション ADR（T7）がタスク一覧に含まれていない
- arrivals テーブルは YAGNI で IT3 スコープ外に
- 木曜の S09 全タスク 9.5h が 1 日に詰め込まれている
- App.tsx 分割の 2h 見積もりが過小（3h が現実的）
- StockForecast の itemId フィールドの必要性を再検討

### スコープ外の発見
- App.tsx で fetchApi と fetch が混在
- value-objects.ts に PurchaseUnit/SupplierId が既に定義済み
- DeliveryDate にバリデーションなし

</details>

<details>
<summary>xp-tester（高: 3 / 中: 2 / 低: 1）</summary>

### 評価サマリー
テスト戦略ドキュメントとの整合性はとれているが、在庫推移計算の複雑さに対してテスト設計の具体性が不足。E2E テスト 2 件では MVP 検証として不十分。

### 主要指摘
- StockForecastService の境界値テストケースを明示すべき（在庫 0 境界、負の在庫、同日の入荷と出荷等）
- PurchaseQuantityService の丸めルール（切り上げ/切り捨て）が曖昧
- E2E テストを 3 件に増加（在庫推移→発注→在庫推移反映フロー）
- PurchaseOrder の不正遷移テストを含める
- isExpiryWarning と在庫不足警告を区別

### スコープ外の発見
- Quantity 値オブジェクトが value <= 0 を拒否するが、在庫推移で負値が出うる
- テスト戦略ドキュメントの S08/S09 に E2E が含まれていない → 更新必要

</details>
