**レビュー日**: 2026-03-23
**レビュー対象**: IT6 実装フロントエンドコンポーネント

- ShipmentPage.tsx（出荷一覧画面 S-402）
- OrderDetailPage.tsx（受注詳細画面 S-102 — キャンセル・届け日変更機能）

## レビュー対象

- ShipmentPage.tsx: 届け日フィルタ + テーブル + 出荷確認ダイアログ + トースト通知
- OrderDetailPage.tsx: 注文情報表示 + 受付/キャンセル/届け日変更の操作 UI
- 関連 API クライアント: shipment-api.ts, order-admin-api.ts

## 総合評価

IT6 のフロントエンド実装は BundlingTargetsPage（IT5）で確立されたパターン（テーブル + 確認ダイアログ + トースト + グレーアウト）を忠実に踏襲しており、**インタラクションの一貫性が高い**。届け日変更の在庫チェック自動実行 + 代替日チップのクリック選択は、電話対応業務に最適化された優れた UX 設計。一方で、**受注詳細画面に内部 ID（顧客 ID / 商品 ID / 届け先 ID）が表示される致命的な問題**、**在庫チェック API エラー時のフィードバック欠如**、**ダイアログのアクセシビリティ未対応**（Escape キー / フォーカストラップ / aria 属性）が主要な改善点。

## エージェント別フィードバック詳細

<details>
<summary>xp-interaction-designer（高: 3 / 中: 4 / 低: 2）</summary>

### 評価サマリー
BundlingTargetsPage で確立されたパターンを忠実に踏襲し、3 画面の UI 一貫性は高い。セマンティックカラー（emerald/red/blue/gray）の使い分けが適切で、情報階層もタイポグラフィで表現されている。しかしダイアログのアクセシビリティ（フォーカストラップ、Escape キー、aria 属性）が完全に欠落しており、WCAG 2.1 AA 準拠に達していない。

### 良い点
- S-401 のパターン（テーブル + 確認ダイアログ + トースト + グレーアウト）を 3 画面で忠実に踏襲
- 在庫チェック結果のインラインバッジ表示（emerald/red）が即座のフィードバックを提供
- 代替日チップのクリック → 自動入力 → 再チェックの一連のフローが直感的
- ボタンの状態遷移（通常 → hover → disabled → isPending）が視覚的に明確
- dl/dt/dd の使用による注文情報の構造化がセマンティクスに適切

### モダンデザイン準拠状況
- カラーシステム: セマンティックカラーの使い分けが一貫（emerald=成功/red=エラー・デストラクティブ/blue=セカンダリ/amber=警告）
- コンポーネント一貫性: 高。3 画面でダイアログ・トースト・テーブルが統一スタイル
- スペーシング: Tailwind のユーティリティクラスで一貫したリズム
- マイクロインタラクション: hover/disabled 状態は定義済み。focus-visible 未定義
- Motion: トースト表示にアニメーションなし（fade-in/slide-up 推奨）
- レスポンシブ: テーブルが固定幅でモバイル未対応

### 改善提案
- 【高】ダイアログの Escape キー / フォーカストラップ / aria 属性追加（WCAG 2.1 AA）
- 【高】在庫チェック API エラー時のフィードバック
- 【高】背景スクロールの防止（ダイアログ open 時）
- 【中】ステータスバッジの色差別化
- 【中】テーブルのレスポンシブ対応
- 【中】トースト通知のスタック対応 + 手動閉じ
- 【中】トースト表示のアニメーション追加（fade-in / slide-up）
- 【低】代替日チップに曜日表示
- 【低】focus-visible 状態の追加（ボタン、リンク）

### 懸念事項
- ダイアログが `div` ベースで `role="dialog"` なし → スクリーンリーダーがダイアログとして認識しない
- `title` 属性によるツールチップはモバイルで表示されず、スクリーンリーダー対応も不完全
- z-50 のハードコードが複数箇所。z-index の体系化が必要

### スコープ外の発見
- shipment-api.ts と order-admin-api.ts の両方に shipOrder 関数が定義されている（DRY 違反）
- AppLayout のナビゲーション内での出荷管理メニューの配置が未確認
</details>

### 評価サマリー
3 機能の業務フローとの整合性は高い。代替日チップのクリック選択は電話対応に最適化された優れた設計。出荷確認ダイアログの届け先情報表示による誤操作防止も実用的。しかし受注詳細画面の内部 ID 表示は業務的に致命的であり、在庫チェックエラー時のフィードバック欠如は電話対応業務に支障をきたす。

- キャンセル確認の「在庫引当が解除されます」警告が後工程への影響を明示
- 代替日チップのクリック → 自動入力 → 再チェックが電話対応に最適
- PREPARING 以降の disabled + ツールチップに「店長にご相談ください」のエスカレーション案内

### モダンデザイン準拠状況
- トースト通知: SaaS 標準パターンに沿うが手動閉じ手段なし
- 確認ダイアログ: モーダルオーバーレイは一般的だが Escape キー未対応
- リアルタイム検証: 日付入力→自動在庫チェックはモダンなインライン検証パターン
- ローディング状態: スピナー + disabled + テキスト変更（「処理中...」）は適切

### 改善提案
- 【高】受注詳細画面で内部 ID の代わりに名前を表示
- 【高】在庫チェック API エラー時のフィードバック表示
- 【高】出荷済み判定をサーバーレスポンスベースに変更
- 【中】出荷一覧にサマリー表示（出荷対象/出荷済み/残り件数）
- 【中】操作ボタンの横並びアクションバー化
- 【中】日付・日時の表示フォーマットを日本語化 + 曜日表示
- 【低】エラーメッセージのステータスコード別分岐
- 【低】CANCELLABLE_STATUSES の canReschedule からの分離

### 懸念事項
- 電話対応中の操作効率: 受注詳細画面の情報量とボタン配置がスクロールを要求
- 出荷一覧の一括処理: 件数増加時に 1 件ずつ処理は非効率（Phase 3 で検討）
- タブレット利用時のテーブル横幅不足（出荷一覧 6 列、特に住所列）

### スコープ外の発見

- acceptMutation.isPending と isAccepting の二重管理（エッジケースで UI 不整合）
- トーストの重なり問題（連続操作時）
