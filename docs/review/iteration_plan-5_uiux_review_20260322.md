# UI/UX レビュー結果: IT5 実装画面（S-302 / S-401）

## レビュー対象

- S-302: 入荷登録画面（docs/design/ui-design.md L666-695）
- S-401: 結束対象一覧画面（docs/design/ui-design.md L697-726）
- IT5 計画の UI セクション（docs/development/iteration_plan-5.md）

## 総合評価

S-302 と S-401 は既存の管理画面パターン（ナビゲーション、テーブル形式、戻りリンク）と高い一貫性を持ち、ドメインモデルのユビキタス言語がそのまま UI ラベルに反映された堅実な設計。ただし、**入荷登録の連続処理効率**（仕入スタッフが朝の忙しい時間に複数発注を処理するシナリオ）と**花材不足時の行動導線**（フローリストが在庫不足を検知した後のアクション）が不足しており、業務の実態に対する配慮に改善の余地がある。

## モダンデザイン準拠サマリー

| 評価項目 | 状態 | 備考 |
|---|---|---|
| カラーシステム | 要改善 | S-401 過不足マイナスの色指定が未定義 |
| タイポグラフィ | OK | 既存 Type Scale（Title / Body Small）に収まる |
| Elevation & Surface | 要改善 | S-302 の情報テーブルとフォームの階層が未定義 |
| コンポーネント一貫性 | OK | CTA = Primary Button、戻りリンク = テキストリンク |
| スペーシング | 要改善 | テーブル・フォーム間のスペーシング未指定 |
| レスポンシブ / Adaptive | 要改善 | IT5 新規画面の具体的な対応指示がない |
| ダークモード | 未対応 | 既存画面と同様、未対応 |
| 状態デザイン（空 / Loading / Error） | 要改善 | S-401 空状態のみ定義済。S-302 のローディング・エラー状態が未定義 |

## 改善提案（重要度順）

### 高（リリース前に対応すべき）

| # | 提案 | 箇所 | 指摘元 | 理由 |
|---|------|------|--------|------|
| H-1 | 入荷登録後の連続作業導線を追加（成功トースト + S-301 自動遷移、または「次の発注を入荷登録」ボタン） | S-302 | Designer, User | 朝の荷受けで複数発注を連続処理するシナリオへの対応不足。IT4 レビューでも「発注成功時トースト通知欠落」が指摘済み |
| H-2 | 過不足マイナス値に視覚アラート追加（error カラー + 「不足」バッジ + アイコン + aria-label）。在庫不足時の行動指示バナーも追加 | S-401 必要花材サマリ | Designer, User | フローリストにとって最も重要な情報。色のみでは WCAG 1.4.1 抵触 |
| H-3 | 入荷登録画面に入荷日フィールド追加（デフォルト「本日」、変更可能） | S-302 | User | IT4 UI レビュー M-2 からの継続課題。前日入荷分のまとめ登録に非対応。品質維持日数の計算起点に影響 |
| H-4 | 入荷数量フィールドの仕様明確化（デフォルト値=残数量、min=1、max=残数量、0/負数バリデーション） | S-302 | Designer | ワイヤーフレームのデフォルト「20」が意図的か不明。入力制約が未定義 |

### 中（対応推奨）

| # | 提案 | 箇所 | 指摘元 | 理由 |
|---|------|------|--------|------|
| M-1 | 受注テーブルの行を展開可能にし各受注の花材構成を表示（アコーディオン行） | S-401 受注テーブル | Designer, User | フローリストが結束作業で「この花束にどの花材が何本必要か」を確認したい |
| M-2 | 結束完了の取り消し手段を定義（30 秒以内の undo ボタン or 管理者権限での取消） | S-401 | User | 不可逆操作（在庫消費）の誤操作リカバリが未定義 |
| M-3 | 結束完了後の花材サマリ自動更新（invalidateQueries で再取得 + スケルトン表示） | S-401 | Designer | 連続結束時に残り在庫を正確に把握するため |
| M-4 | 情報テーブルに「発注日」「希望納品日」「ステータス」を追加 | S-302 | Designer, User | 遅延入荷の把握 + 入荷済み発注への誤遷移防止 |
| M-5 | 両画面のレスポンシブ対応指示を追記（Mobile: カード形式 / Desktop: テーブル形式） | S-302, S-401 | Designer | 既存のレスポンシブ方針テーブルに IT5 新規画面の記載がない |
| M-6 | S-302 のローディング状態定義（コンテンツスケルトン + 発注 ID 無効時のエラー状態） | S-302 | Designer | 既存ガイドラインのパターン適用が未記述 |

### 低（改善の余地あり）

| # | 提案 | 箇所 | 指摘元 | 理由 |
|---|------|------|--------|------|
| L-1 | 受注テーブルのソート機能（最低限「商品名」） | S-401 | User | 同一商品の連続結束で作業効率向上 |
| L-2 | 見出しを「結束対象一覧 - 出荷日: 2026-05-26（準備日: 2026-05-25）」に変更 | S-401 | Designer | 「本日」だけでは届け日との関係が曖昧 |
| L-3 | S-302 の発注ステータスが「入荷済み」の場合、入荷フォームを非表示にし「すべて入荷済みです」表示 | S-302 | Designer | 入荷済み発注への誤操作防止 |

## 矛盾事項

| # | 視点 A | 視点 B | 論点 | 推奨判断 |
|---|--------|--------|------|----------|
| 1 | Designer: 入荷数量のデフォルトは残数量をプリセット | User: 空欄にして意識的に入力させるべき | 全量入荷の効率 vs 誤入力防止 | デフォルト残数量 + フィールド選択時の全選択（ワンタッチで上書き可能） |
| 2 | Designer: 在庫不足時に結束完了ボタンを disabled にすべき | User: 不足でも強制結束が必要な場合がある | 厳格な在庫制御 vs 業務の柔軟性 | 確認ダイアログで強い警告を表示し、結束は許可する。ただし US-013 の受入条件で方針を明確化すべき |

<summary>xp-interaction-designer（高: 3 / 中: 4 / 低: 2）</summary>

### 評価サマリー
既存の管理画面パターンとの一貫性が高く、ユビキタス言語が UI ラベルに反映された堅実な設計。MUI コンポーネントへの具体的な対応指示と、状態デザインの網羅性に改善が必要。

### 高優先度
1. S-401 過不足マイナスの視覚アラート + WCAG 1.4.1 準拠（色 + バッジ + アイコン + aria-label）
2. S-302 入荷数量フィールドの仕様明確化（デフォルト値、min/max、0/負数バリデーション）
3. S-302 成功後フロー未定義（トースト通知 + S-301 自動遷移）

### 中優先度
- S-401 結束完了後の花材サマリ自動更新方針
- S-401 受注テーブルの花材構成展開
- 両画面のレスポンシブ対応指示
- S-302 ローディング・エラー状態定義

### 低優先度
- S-302 発注ステータス表示
- S-401 見出しの日付表記改善

### 懸念事項
- 在庫不足時の結束完了ボタンの扱いが未定義
- IT4 レビュー指摘の Focus/Active 状態が未解消（WCAG 2.4.7）
- MUI と Tailwind CSS の併用方針不明確

### スコープ外
- S-301「入荷登録」ボタンの表示条件（status !== 'RECEIVED'）が未明記
- S-401 → S-402 への遷移導線が IT5 時点で未定義

</details>

<details>
<summary>xp-user-representative（高: 3 / 中: 3 / 低: 1）</summary>

### 評価サマリー
両画面とも業務の流れに対して自然な構成。ただし入荷登録の連続処理効率と、結束対象一覧の花材不足時の行動導線が不足。

### 高優先度
1. 入荷登録後の連続作業導線がない（朝の荷受けで複数発注を処理）
2. 結束対象一覧で花材不足時の行動導線がない（アラート + 行動指示バナー）
3. 入荷登録画面に入荷日フィールドがない（IT4 レビューからの継続課題 2 回目）

### 中優先度
- 各受注の花材内訳が表示されない（個別結束作業時に必要）
- 結束完了の取り消し（undo）ができない
- 入荷情報テーブルに発注日・希望納品日がない

### 低優先度
- 受注テーブルのソート機能

### 懸念事項
- 「本日の結束対象」の日付切り替え不可（繁忙期の前倒し結束に対応できない）
- フローリストのタッチターゲットサイズ未定義（手が汚れた状態での操作配慮不足）

### スコープ外
- 一括入荷登録機能（仕入先単位のチェックリスト形式）
- 結束作業の優先順位表示（配送ルート連携）
- 入荷品質チェック（受入拒否・検品結果の記録）

</details>
