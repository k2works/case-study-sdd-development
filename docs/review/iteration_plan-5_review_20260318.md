# コードレビュー結果 — イテレーション 5 計画

## レビュー対象

- `docs/development/iteration_plan-5.md`（新規作成）

## 総合評価

5 エージェント全員が計画の構成・タスク分解・TDD アプローチの継続性を高く評価している。一方、**S05（届け日変更）の在庫引当解除→再引当の設計が不十分**という指摘が全エージェントから共通して挙がった。具体的には StockLot.deallocate() メソッドの欠落、トランザクション制御パターンの未決定、受入条件の曖昧さが主要な問題点である。また **8 SP がベロシティ上限ギリギリ**であることへの懸念も全員共通。

## 改善提案（重要度順）

| # | 提案 | 箇所 | 指摘元 | 理由 |
|---|------|------|--------|------|
| H1 | StockLot.deallocate() タスクを追加する | タスク 3.1 の前提 | Architect, Tester, Programmer | 現在の StockLot に引当解除メソッドがなく、S05 実装の前提が欠落 |
| H2 | S05 の受入条件を詳細化する（状態制約・日付制約・在庫制約） | S05 受入条件 | Tester, User Rep | 「変更不可」の具体的ルールがないとテストケース設計不可 |
| H3 | 届け日変更のトランザクション設計を明確化する（ADR 作成推奨） | タスク 3.3 | Architect | 引当解除→再引当失敗時のロールバック方針が未定義 |
| H4 | マイグレーション戦略を事前確定する（FK optional vs シードデータ先行） | タスク 0.1 | Architect, Programmer | 既存 Order データに Customer レコードがなく、FK 追加で失敗するリスク |
| H5 | Destination エンティティと既存 DestinationSnapshot の関係を明記する | タスク 1.2 | Programmer | 2 つの届け先概念（マスター vs スナップショット）の変換パスが未整理 |

## 矛盾事項

| # | 視点 A | 視点 B | 論点 | 推奨判断 |
|---|--------|--------|------|----------|

| 2 | Programmer: DestinationRepository は不要、CustomerRepository に含める | Architect: Destination の二重管理（マスター vs 注文履歴）を整理すべき | 届け先の管理方針 | CustomerRepository に含める方針で、order-destinations API は MVP では不要（destinations テーブルのみ使用） |

## 対応方針

### 即時対応（計画修正）

1. **H1**: タスク 3.1 の前に「StockLot.deallocate() メソッド追加 + テスト」を 0.5h のタスクとして追加
2. **H2**: S05 の受入条件に状態制約・日付制約・在庫制約の 6 パターンを明記
3. **H4**: マイグレーション戦略は「FK optional（customerId Int?）」を採用。将来的に必須化
4. **H6**: S05 は「変更依頼 + 基本的な可否判定（状態チェック + 日付チェック）」、S06 は「在庫ベースの高度な可否判定」として境界を設定
5. **H7**: 変更不可時に理由を表示する受入条件を追加
6. **M4**: 成功基準を「登録・編集・一覧表示」に修正

### 開発着手時に対応

7. **H3**: 届け日変更のトランザクション方針を ADR-003 として記録
8. **H5**: タスク 1.2 に DestinationSnapshot への変換メソッドを含める
9. **M1**: E2E テスト 4.3 に変更不可シナリオを追加
10. **M5**: PUT /api/orders/:id/delivery-date のリクエスト/レスポンス型を定義

### 検討事項（IT5 中に判断）

11. **矛盾 1**: S05 の引当再計算スコープ — MVP は引当解除のみ、再引当なし。木曜終了時点でスコープ判断
12. **矛盾 2**: order-destinations API — MVP では destinations テーブルのみ使用。注文履歴からの導出は IT6 以降
