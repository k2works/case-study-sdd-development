# IT3 開発レビュー

## 指摘事項

### 1. 高: 発注候補の生成が花材名ベースの結合に依存しており、安定識別子を失っています

- 対象: `apps/backend/src/inventory.service.ts`, `apps/backend/src/purchase-order.service.ts`
- 内容:
  - 在庫推移 API は `materialId` をレスポンスから落とし、 `materialName` だけを返しています。
  - 発注候補生成では、その `materialName` を使って花材マスタへ再結合しています。
- 問題:
  - 発注候補の下流機能が可変な表示名に依存しており、花材名変更や将来の重名、表示名補正で結合が壊れます。
  - `US-00B` の受け入れ基準 3 は「保存後に在庫推移と発注候補へ反映される」ですが、その反映経路に安定識別子がありません。
- 対応案:
  - 在庫推移レスポンスに `materialId` を含め、発注候補生成も `materialId` で結合する。

### 2. 高: 花材保存と発注確定で `fetch` 例外を未捕捉のため、通信断で画面が落ちます

- 対象: `apps/frontend/app/admin/page.tsx`
- 内容:
  - `handleSaveMaterial()` と `handleConfirmPurchaseOrder()` は `await fetch(...)` を直接呼んでおり、 `try/catch` がありません。
- 問題:
  - バックエンド停止や一時的なネットワーク断で `TypeError: Failed to fetch` がそのまま UI 例外になります。
  - 同じ画面内の `受注一覧` と `在庫推移` は既に例外処理済みで、今回追加した `IT3` 導線だけ障害耐性が欠けています。
- 対応案:
  - 両ハンドラを `try/catch` で包み、失敗時は画面を維持したままエラーメッセージを表示する。
  - 通信例外のフロントテストを追加する。

### 3. 中: 推奨発注数が不足量を見ずに常に 1 購入単位で、欠品量を満たせない場合があります

- 対象: `apps/backend/src/purchase-order.service.ts`
- 内容:
  - 発注候補は最初の不足日のみを拾い、 `suggestedQuantity` を常に `purchaseUnit` に固定しています。
- 問題:
  - 不足量が購入単位を超えるケースや、複数日に連続して不足するケースで、提案数量が必要量を満たしません。
  - `US-05` の「必要な花材を必要なタイミングで確保したい」というストーリー意図に対して過少発注になります。
- 対応案:
  - 不足量と購入単位から切り上げ数量を算出する。
  - 必要なら対象期間内の不足を集約して提案ロジックを定義する。

### 4. 中: 花材変更が発注候補へ反映される回帰テストが不足しています

- 対象: `apps/backend/test/material.spec.ts`, `apps/backend/test/purchase-order.spec.ts`
- 内容:
  - テストでは「花材更新後に在庫推移へ反映される」ことは確認していますが、同じ変更が発注候補へ反映されることは確認していません。
- 問題:
  - 今回の `US-00B` 受け入れ基準 3 は在庫推移と発注候補の両方を対象にしていますが、後者の回帰安全網がありません。
  - 実装と計画書の実績メモの間に、テスト保証の穴が残っています。
- 対応案:
  - 花材更新後に `GET /admin/purchase-orders/candidates` を叩き、名称や購入単位の反映まで検証する結合テストを追加する。

## 総評

- `IT3` の縦切りは一通り成立しており、画面と API の最短導線は確認できます。
- 一方で、発注候補ロジックの識別子設計と数量提案ロジックは、次のデータ拡張で破綻しやすい状態です。
- マージ前に少なくとも指摘 1 と 2 は修正した方が安全です。
