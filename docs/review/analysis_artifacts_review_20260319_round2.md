## 総合評価

前回の高重要度指摘は概ね反映されており、要件、設計、テスト戦略、非機能、運用までのつながりはかなり強くなっています。特に、届け先スナップショット、`partner API` 分離、監視 / バックアップ / 復旧の具体化により、実装前の不確実性は大きく減っています。

一方で、今回の再レビューでは `出荷準備 / 結束` の中間工程が `BUC -> UC -> US -> UI -> テスト戦略 -> データモデル` に一貫して落ちていない点が最重要論点として残りました。加えて、状態名の不統一、リピーター識別ルールの厳しさ、障害時 UX の不足、`partner API` 契約未定義、Node.js バージョン不整合が実装前に整理すべき課題です。

## 矛盾事項

明確な視点対立はありません。各エージェントの指摘は、主に次の 3 点に収束しています。

- `結束` と `出荷確定` の間の業務境界がトレーサブルに定義されていない
- 状態名と識別ルールが文書ごとに揺れている
- 障害時や例外時の UX / テスト条件が未完成である

### 評価サマリー
全体として、要件から設計、テスト、運用までのトレーサビリティはかなり強く、MVP の注文導線と在庫確認はビジネス価値に沿って整理されています。一方で、花束の結束・出荷準備の実作業と、リピーター識別の条件にまだ抜けがあり、重要な価値の回収を少し落としています。

### 良い点
- MVP を注文・受注確認・在庫確認に絞り、Phase 2/3 にマスタ保守と再注文 / 日付変更を分離しているのは妥当です。
- 受注スナップショット、在庫推移の参照モデル、RTO / RPO、監視と復旧演習が一貫しており、業務と運用の整合が高いです。
- `BUC -> UC -> US` の対応関係が概ね通っていて、仕様の追跡性が高いです。

### 改善提案
- 【重要度: 高】`BUC-08` は `UC-08` の「確認」と `UC-09` の「出荷確定」だけでは実現できません。花束を結束して出荷準備完了にする中間工程と、誰が完了責任を持つかを追加してください。
- 【重要度: 中】再利用条件がメールアドレスと電話番号の完全一致に寄りすぎています。検索キーのルールか救済導線を明文化した方がよいです。

### 懸念事項
- `BUC-09` / `US-10` は出荷確定を受注スタッフ責務にしていますが、実際に花束準備を完了するのが誰かが曖昧です。
- `US-07` の識別条件が厳しいままだと、再注文機能が「あるが当たらない」機能になりやすいです。

### スコープ外の発見
- `docs/design/tech_stack.md` の Node.js `24 LTS` は、プロジェクト基準の Node.js `22.x` と不整合です。

全体としては、ヘキサゴナルアーキテクチャ、Feature 単位のフロントエンド、AWS マネージドサービス中心の構成が一貫しており、受注から出荷までの流れも概ね自然です。いまの主要リスクは、出荷準備の中間工程のトレーサビリティ不足と、状態名・識別ルールの未統一による実装時の解釈ぶれです。

- `orders` の届け先スナップショットと `inventory_projections` の再生成可能な投影を分けており、履歴保全と参照最適化の切り分けができています。
- `test_strategy` で、スナップショット保存や一意性、主要 E2E までトレーサビリティが取れています。
- `operation` で、監視、バックアップ、監査、復旧の運用線がかなり具体化されています。

### 改善提案
- 【重要度: 高】`BUC-08` に対応する `UC` と画面がありません。独立ユースケースとして追加するか、1 つに統合してトレーサビリティを回復してください。
- 【重要度: 高】出荷ライフサイクルの状態名が `準備中` / `結束済み` / `出荷準備完了` / `未出荷` に分散しています。1 体系へ統一してください。
- 【重要度: 中】顧客照合ルールをデータモデルへ明記してください。
- 【重要度: 中】`partner API` の認証方式、idempotency、retry、エラーモデルを固定してください。

### 懸念事項
- `出荷対象確認`、`花束の結束`、`出荷確定` が別粒度で書かれていて、権限と監査ログの所有者がずれる可能性があります。

### スコープ外の発見
- `.specify/memory/constitution.md` の Node.js `22.x` と `tech_stack` の Node.js `24 LTS` が不整合です。

## 推奨アクション

1. `結束` と `出荷確定` の関係を先に決め、`BUC-08` を `UC / US / UI / test_strategy / data-model` まで通してください。
2. 出荷状態と発注状態の許可値、遷移、失敗時状態を 1 つの状態表として要件 / データ / ドメインで揃えてください。
3. リピーター識別の救済導線、障害時 UX、`partner API` 最低契約、Node.js バージョン不整合を順に解消してください。
