# イテレーション 6 計画レビュー結果

## レビュー対象

- `docs/development/iteration_plan-6.md`（イテレーション 6 計画）

## 総合評価

最終イテレーションとして堅実な計画であり、ベロシティの余裕（6 SP / 平均 7.6 SP）を品質確保に充てる判断は全エージェントから好評価を受けた。一方で、S06 の在庫引当再計算フローにおける **deallocate のタイミング方針が混在** している点、**キャンセル後の画面遷移が未定義** である点、**S06 の受入条件が user_story.md と乖離** している点が複数エージェントから高優先度で指摘された。これらを実装着手前に解決すれば、完成版リリースに十分な品質を確保できる。

## 改善提案（重要度順）

### 高

| # | 提案 | 指摘元 | 理由 |
|---|------|--------|------|
| H1 | deallocate のタイミングを「在庫チェック後」に統一する。シーケンス図では deallocate 先行型だが、リスク表では「チェック先行」も並記。「読み取り → 在庫チェック → $transaction 内で deallocate → allocate → order 更新」に統一 | Architect, Designer, Tester, User Rep | ロールバック失敗時に在庫データ不整合のリスク。チェック先行ならロールバック不要 |
| H2 | キャンセル後の画面遷移を定義する。推奨: 受注詳細画面に留まり状態が「キャンセル」に更新、キャンセル/届け日変更ボタンは非表示化 | Designer, User Rep | キャンセル操作の結果フィードバックが未定義。ユーザーが操作完了を確認できない |
| H3 | S06 の受入条件を user_story.md と整合させる。計画で 4 項目に拡張されたが原本は 2 項目。原本側を更新し「判断結果を記録できる」の扱いを明確化 | PM, User Rep | 受入条件の二重管理はリリース判定時に混乱を招く |
| H4 | S06 受入条件に「在庫チェック失敗時に既存引当が変更前と完全一致すること」を追加 | Tester | ロールバック/不変性の検証が受入条件になく、テスト漏れのリスク |
| H5 | Repository インターフェースに `tx?` パラメータ追加タスクを追加（見積もり 1-1.5h）。ADR-003 で必要と明記されているが計画のタスクに含まれていない | Architect | prisma.$transaction のコールバック方式に対応するインターフェース拡張が必須 |
| H6 | キャンセル不可時のエラーメッセージに代替手段を案内（「お電話にてお問い合わせください」等） | User Rep | 出荷準備中でシステム上キャンセル不可でも、現場では電話対応でキャンセルを受け付けるケースがある |
| H7 | 在庫不足時のエラーメッセージに次のアクションを提示（在庫推移画面へのリンク、または代替可能日の表示） | Designer, User Rep | 「不足しています」だけではスタッフが次に何をすべきかわからない |

### 中

| # | 提案 | 指摘元 | 理由 |
|---|------|--------|------|
| M1 | S15 のアクターをストーリー本文・シーケンス図で「受注スタッフ」に修正（UI がスタッフ向け受注詳細画面であるため） | Designer, User Rep | ストーリーのペルソナと UI 設計の不一致 |
| M2 | リリース条件（Release 1.2）に非機能要件（テストカバレッジ目標、SonarQube Quality Gate PASS）を追加 | PM | Definition of Done には含まれるがリリース条件に含まれていない |
| M3 | E2E テスト 3.3（全体フロー）で在庫数量の定量的な検証を含める（各ステップで有効在庫・引当済み数を確認） | Tester | 数量検証なしでは在庫引当の正確性をテストできない |
| M4 | デモ項目 4 を修正:「出荷準備中でキャンセル不可」→「出荷準備中の受注にキャンセルボタンが非表示であることを確認」（UI でボタン非表示設計のため） | Designer | ワイヤーフレーム設計とデモ項目が矛盾 |
| M5 | キャンセル済み受注の受注一覧（P05）での表示方法を定義（グレーアウト、フィルタ対応等） | Designer | キャンセル状態の表示ルールが未定義 |
| M6 | validateWithStock() の責務を明確化 — Validator は状態・日付検証に留め、在庫引当可否判断はユースケース内で行う | Architect | Validator にロット知識を持ち込むと凝集度が低下 |
| M7 | DeliveryDateChangeValidator の `new Date()` 直接呼び出しを解消（Clock インターフェース等による DI） | Architect, Tester | テスト時の日付依存・タイムゾーン依存リスク |

### 低

| # | 提案 | 指摘元 | 理由 |
|---|------|--------|------|
| L1 | IT4 Try T4（出荷記録の E2E テスト追加）の扱いを明記（IT6 に含めるか、完成版で許容するか） | PM, Tester | retrospective-5 で「持ち越し → IT6」だが計画に未記載 |
| L2 | PUT /api/orders/:id/cancel → POST の検討（REST 慣例。既存 PUT との一貫性を優先するなら PUT のままで可） | PM, Architect, Designer | 冪等でない状態遷移操作は POST が一般的 |
| L3 | 確認ダイアログに「この操作は取り消せません」を追加 | Designer | キャンセルの Undo が不可能であることをユーザーに明示 |
| L4 | IT6 完了時に ui-design.md を更新するタスクを追加 | Designer | キャンセル機能追加が画面遷移図に未反映 |

## 矛盾事項

| # | 視点 A | 視点 B | 論点 | 推奨判断 |
|---|--------|--------|------|----------|
| 1 | Architect: validateWithStock() は凝集度が低下するため不要 | 計画: ドメインモデル図に validateWithStock() を記載 | Validator に在庫チェックを持たせるか、UseCase に閉じるか | **UseCase に閉じる**（Architect 推奨）。Validator は状態・日付チェックに留め、在庫チェックは UseCase のオーケストレーション責務とする |
| 2 | PM: キャンセル理由の記録機能を将来候補に | User Rep: 任意入力のキャンセル理由フィールドを追加 | IT6 スコープに含めるか | **スコープ外**。将来の改善候補として記録のみ |
