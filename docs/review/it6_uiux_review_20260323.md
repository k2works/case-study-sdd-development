# IT6 UI/UX 設計 マルチパースペクティブレビュー

**レビュー日**: 2026-03-23
**レビュー対象**: IT6 対象画面（S-402 出荷一覧、S-102 キャンセル・届け日変更追加）

---

## 総合評価

IT6 の UI/UX 設計は S-401 結束対象一覧の確立されたパターンを踏襲しており、ダイアログ・フィードバック・ステータス制御の一貫性が高い。届け日変更の代替日クリッカブルチップは電話対応の業務フローに適した優れた設計。しかし、**UI 設計ドキュメント（ui-design.md）本体が IT6 仕様で未更新**であり、計画書に詳細仕様が散在している状態は実装時の解釈ブレを招く。また、**キャンセル不可時のエスカレーションガイダンスが 3 回連続で未対応**であり、業務上の必須要件として確実に対応すべき。

## 改善提案（重要度順）

### 高（リリース前に対応すべき）

| # | 提案 | 箇所 | 指摘元 | 理由 |
|---|------|------|--------|------|
| H-1 | ui-design.md に IT6 仕様（出荷確認ダイアログ、キャンセル UI、届け日変更フォーム）を反映する | docs/design/ui-design.md | UX Designer | 計画書に散在する仕様をデザインドキュメントに集約しないと、実装時の解釈ブレが発生する |
| H-2 | キャンセル不可時のツールチップにエスカレーション案内を追加（「出荷準備中のためキャンセルできません。対応が必要な場合は店長にご相談ください」） | S-102 キャンセルボタン | User-Rep | **3 回目の指摘**。ツールチップに理由だけでは受注スタッフが得意先に「ではどうすれば」と聞かれて答えられない |
| H-3 | 出荷一覧のフィルタ条件を UI 上で明示（「届け日: YYYY-MM-DD の出荷対象」表示 + 日付切り替え機能） | S-402 ヘッダー | User-Rep | 「本日」表示のみでは何の日付で絞っているか不明。翌日分の確認もできない |
| H-4 | 出荷一覧・出荷ボタンのタッチターゲットを最低 48px 確保 + 行間の誤タップ防止余白 | S-402 テーブル行 | User-Rep | 配送スタッフは手袋着用でスマホ/タブレット操作。IT5 レビューからの継続指摘 |

### 中（対応推奨）

| # | 提案 | 箇所 | 指摘元 | 理由 |
|---|------|------|--------|------|
| M-1 | 届け日変更フォームに変更前の届け日を表示（「現在の届け日: 2026-04-01 → 新しい届け日: ___」の比較形式） | S-102 届け日変更 | User-Rep | 電話対応中に「今いつで、いつに変更するか」の比較が見えないと不便 |
| M-2 | 出荷一覧を届け先住所の昇順でソート（配送ルート効率化） | S-402 テーブル | User-Rep | 同一日に複数出荷がある場合、配送順の参考になる |
| M-3 | 出荷確認ダイアログの内容を具体化（受注番号・商品名・届け先氏名・住所・「出荷済みに変更されます」） | S-402 出荷ダイアログ | User-Rep | 誤操作防止のため、操作対象の特定と結果の説明が必要 |
| M-4 | ダイアログのフォーカストラップと Escape キーでの閉じ操作を定義 | 全ダイアログ | UX Designer | WCAG 2.1 AA 準拠。キーボードナビゲーション対応が必要 |
| M-5 | S-402 のレスポンシブ対応（Mobile: テーブル→カード形式変換） | S-402 | UX Designer | 配送スタッフがモバイル端末で使用する可能性 |

2. 在庫チェック結果のインラインバッジ表示が即座のフィードバックを提供
3. 代替日クリッカブルチップが再チェックを自動実行する設計
4. キャンセル確認ダイアログの副作用明示（「在庫引当が解除されます」）

- タイポグラフィ: 情報階層は概ね良好だが住所列の truncation 未定義

- 状態デザイン: 空状態は定義済み。ローディング状態が欠落
