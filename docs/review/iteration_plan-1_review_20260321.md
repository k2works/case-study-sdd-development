## 総合評価

ヘキサゴナルアーキテクチャに沿ったレイヤー別タスク分解、リスク駆動のスケジューリング（Week 1 に認証基盤を集中）、保守的なベロシティ設定（計画 13 SP の 85%）など、初回イテレーションとして堅実な計画である。一方、**アーキテクチャ設計書との不整合**（API バージョニング、データモデル）、**TDD サイクルの明示不足**、**セキュリティテストの計画欠落**、**受入条件の詳細化不足**（アカウントロック仕様、パスワード要件）が主な改善点として指摘された。

## 改善提案（重要度順）

### 高（開発着手前に対応すべき）

| # | 提案 | 箇所 | 指摘元 | 理由 |
|---|------|------|--------|------|
| 1 | API パスに `/api/v1/` バージョニングプレフィックスを追加 | API 設計セクション | programmer, architect | `architecture_backend.md` では `/api/v1/` を定義。IT1 で不一致を放置すると全 API 呼び出し箇所の修正が必要 |
| 2 | TDD サイクルをタスク名に明示（例: 「ドメインモデル TDD」） | タスク分解セクション | programmer, tester | 現状のタスク名では「先に全部実装してからテスト」に陥りやすい。TDD の意図を明確化すべき |
| 3 | セキュリティテストのタスクを追加（JWT 検証、認可、パスワードハッシュ） | US-017 タスク | tester | テスト戦略の品質ゲート「セキュリティスキャン Critical/High 0 件」を満たすために必須 |
| 4 | アカウントロック仕様の詳細化（ロック期間、リセット条件、ロック中の挙動） | US-017 受入条件 | tester, user-rep | 境界値テストとエッジケースが定義できず、実装時に判断が曖昧になる |
| 5 | users テーブル DDL をデータモデル設計書と整合させる | データモデルセクション | architect | `first_name`/`last_name` vs `name`、`role` の VARCHAR 長など不一致あり |
| 6 | items テーブルの `supplier` カラムの方針を明記 | データモデルセクション | architect, programmer | `domain_model.md` では `supplierId: SupplierId` だが IT1 では文字列。暫定対応ならその旨を注記 |

### 中（対応推奨）

| # | 提案 | 箇所 | 指摘元 | 理由 |
|---|------|------|--------|------|
| 7 | パスワードバリデーション要件を受入条件に追加（最小文字数、確認一致） | US-018 受入条件 | tester, user-rep | UI 設計に「パスワード確認」フィールドがあるのに受入条件に未記載 |
| 8 | 認証ドメインモデルを `domain_model.md` に追記 | ドメインモデルセクション | architect, programmer | 認証コンテキストが設計書に存在せず、IT1 計画と乖離 |
| 9 | プロジェクト基盤整備タスクを追加（ArchUnit 初期設定、Flyway、Checkstyle、パッケージ構造） | タスク分解セクション | programmer | 見積もり 48h に含まれていない初期セットアップ工数（4-6h） |
| 10 | `purchase_unit` の型を INTEGER に統一 | データモデルセクション | architect | ドメインモデル・data-model.md は INTEGER だが IT1 DDL は VARCHAR |
| 11 | US-003 の境界値を受入条件に追加（品質維持日数・リードタイムの下限値） | US-003 受入条件 | tester | バリデーションタスクはあるが、具体的な境界値が未定義 |
| 12 | ロール別の初期画面方針を明確化 | US-017 受入条件 | user-rep | 「ロールに応じたトップ画面」の具体的な振り分けが未定義 |

### 低（改善の余地あり）

| # | 提案 | 箇所 | 指摘元 | 理由 |
|---|------|------|--------|------|
| 13 | エラーレスポンス形式（RFC 7807）を API 設計に明記 | API 設計セクション | architect | パターン確立の一環として統一すべき |
| 14 | Day 5 のタスク密度を調整（統合テストを Day 6 に移動） | スケジュールセクション | programmer | 3 タスク 6h が詰め込まれている |
| 15 | `locked_until` カラムを `TIMESTAMPTZ` に変更検討 | データモデルセクション | tester | TZ なし TIMESTAMP はテスト環境と本番で挙動が異なるリスク |
| 16 | フロントエンドのテスト方針を明記（Vitest/React Testing Library） | 完了条件セクション | programmer, tester | カバレッジ 80% 目標に対するフロントエンドのテスト計画が不明 |

## 開発着手前の確認事項

以下 3 点を決定してから開発に着手すること：

1. **アカウントロック期間**: 何分間ロックするか（推奨: 30 分）
2. **パスワード要件**: 最小文字数と必須文字種（推奨: 8 文字以上、英数字必須）
3. **ロール別初期画面**: IT1 では全ロール共通ダッシュボードとするか
