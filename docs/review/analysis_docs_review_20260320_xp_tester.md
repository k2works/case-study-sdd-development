### 評価サマリー
分析成果物は、業務フロー・状態遷移・モジュール境界が比較的そろっており、受注から在庫 projection までの主要シナリオを統合テストへ落とし込みやすい構成です。一方で、受け入れ判定に必要な業務ルールと非機能要件の多くが定性的なままで、システムテストの合否を機械的に判定できない箇所が残っています。

### 良い点
- バックエンド設計で `domain` / `application` / `presentation` ごとのテスト方針が明示されており、テストピラミッドを維持しやすいです。
- 要件定義に業務フロー、イベントモデル、状態遷移があり、注文、届け日変更、発注、入荷、出荷の主要シナリオを Given-When-Then に分解しやすいです。
- 在庫 projection を業務上のストレスポイントとして明示し、初期は同期更新を採用しているため、結合テストで期待値を固定しやすいです。
- フロントエンドが REST API と ViewModel 変換を前提にしており、API 契約テストと画面結合テストの責務分離がしやすいです。
- UI 設計で顧客導線とスタッフ導線が分かれており、主要 E2E シナリオの起点が明確です。

### 改善提案
- 【重要度: 高】要件を受け入れ可能な形に数値化してください。`迷わず注文できる`、`迅速に対応したい`、`俯瞰できる`、`確実に届けてほしい` は現状では自動受け入れテストの期待結果にできません。例として、注文完了までの最大ステップ数、届け日変更可否判定の応答時間、在庫推移画面の更新時間、注文 API 成功率などを明記すると、システムテストと非機能テストの合否判定が可能になります。
- 【重要度: 高】届け日 / 出荷日ルールの境界条件を明文化してください。現状は `出荷日は届け日の前日` の 1 ルールのみで、休業日、締め時刻、当日注文、配送不可日、タイムゾーン、祝日・繁忙期の扱いが未定です。ここが曖昧だと、`ShipmentDateCalculator` と `OrderDateChangePolicy` の境界値テストを設計できません。
- 【重要度: 高】届け日変更機能の受け入れ条件を詳細化してください。誰が、いつまで、どの状態の注文に対して、何回まで変更できるか、変更不可時にどの代替候補を返すかが不足しています。状態遷移図には `変更受付中` と `キャンセル` がある一方で、UI / API / 要件一覧では変更期限や取消条件がそろっておらず、システムテストで仕様の期待値を固定できません。
- 【重要度: 高】在庫 projection の期待値算出ルールを examples 付きで追加してください。`available_quantity = opening + inbound - reserved` はありますが、品質維持日数超過、部分入荷、不良品、入荷超過、複数注文同時確定、届け日変更の差分反映時の優先順位が文書化されていません。少なくとも decision table か worked example を用意し、結合テストの oracle を作れる状態にすべきです。
- 【重要度: 高】認証 / 認可の前提を早めに確定してください。届け先再利用と届け日変更は顧客の本人性確認に依存しますが、顧客認証方式とスタッフ認可粒度が複数ドキュメントで TBD のままです。これでは E2E テストで事前条件を定義できず、権限境界の結合テストも組めません。
- 【重要度: 中】エラー契約を標準化してください。バックエンドでは `業務コード付き`、フロントエンドでは `業務エラーとシステムエラーを区別` とありますが、エラーコード体系、HTTP status の対応、画面表示メッセージ、再試行可否が未定です。API 契約テストと UI の異常系テストを安定化するため、代表エラー payload を追加することを推奨します。
- 【重要度: 中】非機能要件をテスト可能な SLO / SLI に変換してください。現状のインフラ設計には監視対象があるものの、可用性、性能、バックアップ、リストア、ログ保全、監査証跡、アクセシビリティの達成基準が不足しています。例として、RTO / RPO、注文 API の p95 latency、在庫 projection 更新の freshness、WCAG 適合レベルを定義すると検証計画を立てやすくなります。
- 【重要度: 中】バッチ / scheduler のテスタビリティを設計に入れてください。`in-process / cron` は運用上は簡潔ですが、時刻依存、再実行、二重実行防止、手動起動、ドライランの仕組みがないと結合テストが不安定になります。Clock 注入、job idempotency、手動トリガー API / CLI のどれを使うかを決めるとよいです。
- 【重要度: 中】テストデータ戦略を追加してください。注文、再注文、日付変更、部分入荷、廃棄リスクをまたぐシナリオはデータ準備が複雑です。seed、factory、固定日付、代表商品構成セットを定義しておくと、E2E と integration test の保守性が上がります。
- 【重要度: 低】UI 設計のアクセシビリティ要件を操作ベースで補強してください。ラベル付与やキーボード操作の方針はありますが、Date Picker、テーブル、Toast、色ラベル併用の具体的な検証観点が不足しています。主要画面ごとに keyboard path と screen reader の期待動作を 1 行ずつ追加すると、受け入れテストへ落とし込みやすくなります。

### 懸念事項
- `1 受注 1 商品` は明記されていますが、同一商品を複数束購入できるのかが要件・UI・データモデルで不明です。`orders` に数量がなく、現状は 1 注文 1 束に読めるため、ビジネス意図を確認すべきです。
- `quality_status` を持つ一方で、不良入荷や一部受入拒否が在庫 projection にどう反映されるかが不明です。入荷異常系の回帰テストが不足しやすいです。
- 在庫 projection を同期更新から将来非同期化する計画ですが、整合性確認用の read-your-write 保証や freshness の許容値が未定です。後で非同期化したときに既存 E2E が不安定になるリスクがあります。
- 経営者向け KPI 確認は要件に存在しますが、UI 設計では専用画面が見当たりません。受け入れ対象に含めるのか、将来スコープなのかを整理した方がよいです。

### スコープ外の発見
- 要件定義の状態遷移には `キャンセル` が含まれていますが、UI 設計、API 例、データモデル詳細ではキャンセル機能が実質未定義です。今回の主要スコープ外なら、状態モデルから外すか将来要件として明示した方が混乱を防げます。
- フロントエンドアーキテクチャでは `商品一覧 -> ログイン -> 受注一覧` の遷移がありますが、UI 設計の全体遷移にはログイン画面がありません。認証導線の不一致は後続の E2E 設計に影響します。
