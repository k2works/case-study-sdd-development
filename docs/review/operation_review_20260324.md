## 総合評価

**B+（小規模チームの制約を意識した堅実な設計。コスト見積もりの精緻化と運用手順の具体化が必要）**

## 改善提案（重要度順）

### 高

| # | 提案 | 指摘元 | 理由 |
| :--- | :--- | :--- | :--- |
| 1 | RDS 監視に FreeableMemory を追加 | Arch | CPU のみ監視でメモリ不足を見逃すリスク |
| 2 | ECS タスクを 0.5vCPU/1GB に引き上げ検討 | Arch | 0.25vCPU/0.5GB では CTE クエリ実行中にブロック |
| 3 | 障害対応ランブック作成（L3 シナリオ） | PM | 1 名体制での深夜障害対応手順が未定義 |
| 4 | バッチ処理の失敗通知・リカバリ手順を定義 | PM, Tester | 品質維持期限チェック失敗は業務クリティカル |
| 5 | RDS 保存時暗号化（encryption at rest）を有効化 | Arch | 個人情報保護法上の要件 |
| 6 | 日次バッチのテスト戦略を追加 | Tester | バッチ処理のテストがテスト戦略に未記載 |

### 中

| # | 提案 | 指摘元 | 理由 |
| :--- | :--- | :--- | :--- |
| 7 | NAT Gateway コストの考慮（VPC Endpoint 検討） | PM | 隠れコスト $30+/月 |
| 8 | ステージング環境のコスト見積もり追加 | PM | 総運用コストが不透明 |
| 9 | 負荷テストツール（Locust 等）をテスト戦略に追加 | Arch, Tester | レスポンスタイム目標の検証手段がない |
| 10 | Django Admin の IP 制限 | Arch | 全テーブル CRUD 権限のため不正アクセスリスク大 |
| 11 | DB マイグレーションの拡張・縮退パターンを方針化 | Arch | スナップショット復元依存を回避 |
| 12 | pre-commit をLint のみに変更検討 | Arch, PM | テスト増加時のコミット待ち時間 |

### 低

| # | 提案 | 指摘元 | 理由 |
| :--- | :--- | :--- | :--- |
| 13 | アクセシビリティテストを CI に組み込み | Arch | WCAG 2.1 AA 要件の自動検証 |
| 14 | 監視アラートのチューニング方針追加 | PM | アラート疲れの防止 |
| 15 | 秘匿情報管理方針（Secrets Manager 等）を追記 | PM | docker-compose に平文パスワード |

## 対応方針

### 即時修正

1. RDS FreeableMemory 監視を追加
2. ECS タスクサイズ見直し（0.5vCPU/1GB）
3. バッチ処理の失敗通知・リカバリ手順を追加
4. RDS 暗号化を明記
5. 日次バッチのテスト戦略を追加
6. NAT Gateway / VPC Endpoint の方針を追記

### 開発フェーズで対応

7. 障害対応ランブック作成
8. 負荷テスト計画
9. Django Admin IP 制限の実装
