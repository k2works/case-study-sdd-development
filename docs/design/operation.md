# 運用要件

本書は、フラワーショップ「フレール・メモワール」 WEB ショップシステムの運用要件を定義します。少人数運用を前提に、監視、バックアップ、障害対応、変更管理を自動化中心で設計し、`99.5%` の可用性、`RTO 4 時間`、`RPO 15 分` を満たすことを目標とします。

## 前提

- インフラは AWS のマネージドサービス中心構成とする
- 顧客向け経路、スタッフ向け管理画面、partner API を分離して運用する
- 監視、デプロイ、バックアップ、監査ログ収集はできる限り IaC と CI/CD で自動化する
- 運用対象環境は Local、Development、Staging、Production とする

## 運用体制

### 役割

| 役割 | 主責務 |
| :--- | :--- |
| 開発担当 | アプリケーション障害の一次調査、恒久対応、リリース実施 |
| 運用担当 | 監視確認、アラート初動、バックアップ確認、復旧手順実行 |
| 店舗責任者 | 業務影響判断、顧客対応判断、緊急変更承認 |
| 管理者 | 権限棚卸し、監査ログ確認、月次レビュー承認 |

### エスカレーション方針

| 重大度 | 例 | 初動目標 | 通知先 |
| :--- | :--- | :--- | :--- |
| `Sev1` | 注文不可、出荷確定不可、DB 障害 | 15 分以内 | 運用担当、開発担当、店舗責任者 |
| `Sev2` | 一部画面性能劣化、batch 停滞、partner API 障害 | 30 分以内 | 運用担当、開発担当 |
| `Sev3` | 一時的エラー増加、警告閾値超過 | 当日中 | 運用担当 |

## 定常運用

### 日次運用

- 監視ダッシュボードで API 応答時間、HTTP `5xx`、ジョブ失敗、DB 接続数を確認する
- 前日分のバックアップ成功可否と PITR 状態を確認する
- 監査ログ収集の欠損有無を確認する
- 在庫推移再計算ジョブの失敗件数と処理時間を確認する
- ECS タスク再起動回数、WAF ブロック件数、partner API 認証失敗件数を確認する

### 週次運用

- アラート履歴をレビューし、不要アラートと閾値調整候補を整理する
- Development / Staging の未適用設定差分を確認する
- 商品マスタ / 花材マスタ更新の監査ログを抜き取り確認する
- ディスク使用量、DB サイズ、オブジェクトストレージ使用量の増加傾向を確認する

### 月次運用

- OS / コンテナベースイメージ / ミドルウェアのセキュリティ更新を評価し、定例メンテナンスで適用する
- 権限棚卸しを行い、退職者、異動者、不要権限を削除する
- SLA / SLO 実績、重大障害件数、リリース件数、ロールバック件数をレビューする
- バックアップからのリストア演習を Staging で少なくとも 1 回行う

### 年次運用

- DR 訓練を実施し、`RTO 4 時間`、`RPO 15 分` の達成可否を記録する
- 監査ログ保持、暗号化設定、証明書更新計画を見直す
- サポート期限切れが近いランタイム、フレームワーク、DB の更新計画を策定する

## 監視設計

### 監視対象

| 区分 | 監視項目 | 目的 |
| :--- | :--- | :--- |
| 可用性 | HTTP ヘルスチェック、ECS タスク稼働数、ALB ターゲット正常数 | サービス停止の早期検知 |
| 性能 | 商品一覧、注文確認、受注一覧、在庫推移、注文確定 API、出荷確定 API の応答時間 | 非機能要件の維持 |
| 品質 | HTTP `5xx` 率、アプリ例外件数、ジョブ失敗件数 | 障害兆候の検知 |
| データ | DB CPU、接続数、ストレージ残量、スロークエリ、レプリケーション / PITR 状態 | データ層の劣化検知 |
| セキュリティ | WAF ブロック件数、認証失敗件数、権限変更、partner API 署名 / 認証失敗 | 不正アクセスの検知 |
| ビジネス | 注文件数、注文失敗件数、在庫推移再計算時間、出荷未確定件数 | 業務影響の可視化 |

### アラート閾値

| 監視項目 | Warning | Critical |
| :--- | :--- | :--- |
| 商品一覧 / 受注一覧 p95 | 2.0 秒超が 5 分継続 | 3.0 秒超が 5 分継続 |
| 注文確定 API p95 | 3.0 秒超が 5 分継続 | 5.0 秒超が 5 分継続 |
| HTTP `5xx` 率 | 1% 超が 5 分継続 | 3% 超が 5 分継続 |
| ECS タスク稼働数 | 期待値未満が 3 分継続 | サービス不能状態を即時検知 |
| DB CPU 使用率 | 70% 超が 10 分継続 | 85% 超が 10 分継続 |
| DB ストレージ使用率 | 75% 超 | 90% 超 |
| 在庫推移再計算時間 | 4 分超 | 5 分超 |
| ジョブ失敗件数 | 同一ジョブ 3 回連続失敗 | 同一ジョブ 5 回連続失敗 |
| partner API 認証失敗 | 15 分間で 10 件超 | 15 分間で 30 件超 |

### 通知経路

- `Critical` は即時にチャット通知し、当番担当へエスカレーションする
- `Warning` は運用ダッシュボードと日次レビューで確認する
- 同一事象の重複通知は抑制し、復旧通知とセットで送る
- 顧客影響がある `Sev1` は店舗責任者へ 15 分以内に連絡する

## ログ / 監査

### ログ分類

| 種別 | 内容 | 保持期間 |
| :--- | :--- | :--- |
| アプリケーションログ | API 呼び出し、例外、ジョブ実行結果 | 90 日 |
| アクセスログ | CDN、WAF、ALB アクセス | 180 日 |
| 監査ログ | 受注変更、届け日変更、発注確定、出荷確定、マスタ更新、権限変更 | 3 年 |
| デプロイログ | CI/CD 実行履歴、承認履歴、ロールバック履歴 | 1 年 |

### ログ運用方針

- すべてのアプリケーションログは JSON 形式で構造化出力する
- 相関 ID を顧客画面、管理画面、partner API、batch で共通利用する
- 個人情報はログへ平文出力しない。住所、電話番号、メールアドレスはマスクする
- 監査ログは改ざん耐性のある保存先へ転送し、アプリ担当者でも削除不能にする

## バックアップ / リストア設計

### バックアップ対象

| 対象 | 方式 | 頻度 | 保存期間 |
| :--- | :--- | :--- | :--- |
| PostgreSQL | 自動バックアップ + PITR | 継続取得、日次スナップショット | 日次 35 日、月次 12 か月 |
| オブジェクトストレージ | バージョニング + ライフサイクル管理 | 即時 | 90 日以上 |
| IaC / アプリ定義 | Git 管理 + CI 生成物保管 | 変更時 | リポジトリ履歴準拠 |
| 監査ログ | 専用ストレージ保管 | 日次集約 | 3 年 |
| Secrets / 設定 | マネージドサービス履歴 + IaC | 変更時 | 1 年以上 |

### リストア方針

- `Sev1` 障害時は DB の PITR により `RPO 15 分` 以内の時点へ復旧する
- Staging で月次リストア演習を行い、注文、出荷、在庫推移、監査ログ整合性を確認する
- リストア後はアプリケーションヘルスチェック、主要画面の疎通確認、代表ユースケースのスモークテストを行う

### 復旧手順

1. 障害の影響範囲と復旧対象時点を確定する。
2. Production への書き込みを停止し、必要に応じてメンテナンス表示へ切り替える。
3. DB を指定時点へ復旧し、接続先を切り替える。
4. 受注件数、出荷件数、在庫推移、監査ログ件数を復旧前後で照合する。
5. スモークテスト完了後に業務再開を宣言する。

## 障害対応

### 障害分類

| 分類 | 代表例 | 一次対応 |
| :--- | :--- | :--- |
| アプリケーション障害 | 例外多発、注文確定失敗 | 直近デプロイ確認、ログ確認、必要に応じてロールバック |
| データベース障害 | 接続不可、性能劣化、容量逼迫 | フェイルオーバー確認、接続数制御、復旧手順起動 |
| 外部境界障害 | partner API 認証失敗、外部配送連携停止 | 経路切り分け、署名 / 鍵確認、必要に応じて手動運用へ切替 |
| セキュリティ障害 | 不正アクセス疑い、権限誤設定 | 該当経路遮断、鍵 / セッション無効化、監査ログ保全 |

### インシデント対応フロー

1. アラート受信後に重大度を判定する。
2. 顧客影響と業務影響を 15 分以内に整理する。
3. 暫定復旧を優先し、恒久対応は切り分け後に別 issue 化する。
4. `Sev1` は当日中に障害記録を残し、1 営業日以内にポストモーテムを作成する。

### 手動運用へのフォールバック

- partner API 停止時は仕入先連携を一時停止し、受注 / 出荷業務は管理画面で継続する
- 在庫推移再計算が失敗した場合は、直前正常スナップショットを参照しつつ再実行を優先する
- 顧客注文導線に重大障害がある場合は、トップで障害告知を表示し、店舗電話受付へ誘導する

### partner API 運用契約

- 認証失敗は即時に `4xx` として記録し、同一仕入先で 15 分間にしきい値超過時は鍵 / 署名設定を確認する
- `Idempotency-Key` が欠落した書き込み要求は受け付けない
- `5xx` またはタイムアウト時は、同一キーでの再送履歴を確認してから再実行する
- partner API の障害時は手動運用へ切り替えた時刻、対象仕入先、復旧時刻を障害記録へ残す

## セキュリティ運用

### アカウント / 権限管理

- スタッフ権限は RBAC で管理し、管理者だけがロール変更できる
- 退職、異動、委託終了時は当日中にアカウントを停止する
- 特権アカウント利用は監査対象とし、月次で棚卸しする

### 脆弱性 / パッチ管理

- 緊急度 `高` の脆弱性は 7 日以内、`中` は 30 日以内を目標に対応する
- コンテナイメージ、依存ライブラリ、IaC の脆弱性スキャン結果を月次レビューに含める
- WAF ルール変更、Secrets 更新、証明書更新は変更管理フローに従う

## 変更管理

### 標準変更

- コード変更は Pull Request、CI 通過、レビュー承認を経て反映する
- Development は main 反映後に自動デプロイ、Staging / Production は承認付きデプロイとする
- リリース前に DB 変更有無、feature flag、ロールバック可否をチェックリストで確認する

### 緊急変更

- `Sev1` 収束のための緊急変更は開発担当と店舗責任者の 2 者承認で実施する
- 緊急変更後は 1 営業日以内に通常の Pull Request と ADR / 障害記録へ反映する

### ロールバック方針

- アプリケーションは直前安定版コンテナイメージへ切り戻せる状態を維持する
- DB 変更を伴うリリースは後方互換を持つ段階リリースを原則とする
- ロールバック判断は `5xx` 急増、主要ユースケース失敗、重大なデータ不整合を基準に行う

## 運用受け入れ基準

- `Sev1`、`Sev2` の監視項目、通知先、初動目標がダッシュボードと手順書に反映されている
- 月次リストア演習で `RTO 4 時間`、`RPO 15 分` の達成を検証できる
- 顧客画面、管理画面、partner API、batch のログが相関 ID で追跡できる
- 標準変更と緊急変更の承認フロー、ロールバック手順が CI/CD 手順書と整合している
- 年次 DR 訓練と月次権限棚卸しの実施責任者が明確になっている
