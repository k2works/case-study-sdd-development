# 運用要件

## 1. 文書の目的

本書は、フレール・メモワール WEB ショップシステムを継続的に安全運用するために必要な運用要件を定義するものです。

非機能要件で定義した SLA / SLO、RTO / RPO、セキュリティ、監視基準を満たすために、日常運用、監視、バックアップ、障害対応、変更管理の標準を明確化します。

## 2. 前提と適用範囲

### 2.1 前提

- システムは `local`、`staging`、`production` の 3 環境で運用します
- `web`、`api`、`postgres` を中核とするコンテナ + managed PostgreSQL 構成を前提とします
- 小規模な XP チームが開発と運用を兼務し、平日日中を通常運用時間とします
- `stock_projection` は同期更新を基本とし、障害時は再計算 job で復旧します
- 顧客向け注文導線は営業時間外も利用されるため、critical 障害は営業時間外でも検知 / 一次対応できる体制を前提とします

### 2.2 適用対象

- 顧客向け Web 画面
- スタッフ向け Web 画面
- REST API
- scheduler / background job
- PostgreSQL とバックアップ
- GitHub Actions を用いた CI/CD
- ログ、メトリクス、アラート、監査ログ

### 2.3 運用原則

- 自動化優先。手動作業は runbook に限定し、同じ操作を 2 回以上行うなら自動化候補とします
- 監視は「検知して対応するもの」だけを alert にし、観測用途は dashboard に留めます
- 変更にはロールバック手順を必須とし、切り戻し不能な変更は本番投入しません
- バックアップは取得だけでなく、定期リストア検証まで完了して初めて有効と見なします
- 障害対応では顧客影響の縮小を優先し、恒久対策は事後に issue 化して追跡します

## 3. 運用体制

### 3.1 役割分担

| 役割 | 主責務 | 補足 |
| :--- | :--- | :--- |
| 運用責任者 | 本番運用方針、承認、重大障害の指揮 | 小規模体制では tech lead 兼務可 |
| 当番開発者 | alert 一次受け、障害切り分け、暫定復旧 | 平日持ち回りを基本 |
| 事業責任者 | 顧客影響判断、告知判断、業務優先順位決定 | owner 相当 |
| 開発チーム | 恒久対策、監視改善、runbook 更新 | ふりかえりで改善を継続 |

### 3.2 対応時間

| 区分 | 対応方針 |
| :--- | :--- |
| 営業時間内 | warning / critical ともに当番開発者が確認する |
| 営業時間外 | critical のみ即時通知し、warning は翌営業日に確認する |
| planned maintenance | 24 時間前までに社内共有し、顧客影響がある場合は告知する |

## 4. 定常運用フロー

### 4.1 日次運用

| 項目 | 実施タイミング | 内容 | 完了条件 |
| :--- | :--- | :--- | :--- |
| ヘルスチェック確認 | 営業開始前 | Web / API / DB / job の dashboard を確認 | critical alert なし |
| 失敗 job 確認 | 毎営業日 | `stock_projection` 再計算、通知再試行、集計 job の失敗を確認 | 連続失敗 0 件 |
| バックアップ確認 | 毎営業日 | 前日バックアップ成功と PITR 状態を確認 | 失敗なし |
| エラーログ確認 | 毎営業日 | 5xx、認証失敗急増、例外ログを確認 | 未対応の重大エラーなし |
| 顧客影響確認 | 毎営業日 | 注文失敗、届け日変更失敗、在庫更新遅延を確認 | SLA 逸脱なし |

### 4.2 週次運用

| 項目 | 内容 |
| :--- | :--- |
| エラーバジェット確認 | 注文 API 成功率、主要画面 p95、`stock_projection` freshness を確認する |
| 容量確認 | DB ストレージ、接続数、画像保存量、ログ量を確認する |
| 監視ノイズ棚卸し | 不要 alert、重複 alert、runbook 不足を見直す |
| 変更レビュー | 本番反映された変更と未完了の follow-up を確認する |

### 4.3 月次運用

| 項目 | 内容 | 成果物 |
| :--- | :--- | :--- |
| セキュリティパッチ適用 | OS / runtime / 依存ライブラリの patch 状況を確認し適用する | patch 実施記録 |
| 権限棚卸し | スタッフ権限と secrets アクセス権を見直す | 権限レビュー記録 |
| バックアップ復旧演習 | staging で DB リストア手順を検証する | 復旧演習記録 |
| 容量 / コスト確認 | コンテナ利用量、DB サイズ、監視コストを確認する | 月次運用レポート |

### 4.4 四半期 / 年次運用

| 項目 | 頻度 | 内容 |
| :--- | :--- | :--- |
| production 相当 restore 訓練 | 半年ごと | 本番バックアップから staging へ復旧し、RTO / RPO を確認する |
| DR / 障害訓練 | 四半期ごと | API 停止、DB 障害、projection 遅延を想定した訓練を実施する |
| 監査ログレビュー | 四半期ごと | 重要操作ログと権限設定の整合を監査する |
| 証明書 / 契約更新確認 | 年次 | ドメイン、証明書、外部サービス契約更新を確認する |

## 5. 監視設計

### 5.1 監視レイヤー

| レイヤー | 目的 | 主な指標 |
| :--- | :--- | :--- |
| 可用性監視 | 顧客影響の即時検知 | health check、HTTP 5xx、synthetic check |
| 性能監視 | SLO 逸脱の早期検知 | p95 / p99 応答時間、slow query、job duration |
| 業務監視 | 業務停止やデータ遅延の検知 | 注文成功率、届け日変更失敗率、`stock_projection` freshness |
| セキュリティ監視 | 不正利用 / 設定逸脱の検知 | 認証失敗、権限変更、脆弱性検知 |

### 5.2 監視項目と閾値

| 対象 | 指標 | warning | critical | 対応 runbook |
| :--- | :--- | :--- | :--- | :--- |
| Web | p95 応答時間 | 2 秒超が 15 分継続 | 3 秒超が 5 分継続 | Web 遅延調査 |
| API | 5xx 率 | 1% 超が 5 分継続 | 3% 超が 5 分継続 | API 障害切り分け |
| 注文 API | 成功率 | 99% 未満が 15 分継続 | 97% 未満が 5 分継続 | 注文停止対応 |
| DB | CPU 使用率 | 80% 超が 15 分継続 | 90% 超が 10 分継続 | DB 負荷対応 |
| DB | 接続数 | 80 超 | 100 超 | 接続枯渇対応 |
| Job | 連続失敗回数 | 2 回連続失敗 | 3 回連続失敗 | job 再実行 |
| `stock_projection` | 更新遅延 | 60 秒超 | 5 分超 | projection 再計算 |
| セキュリティ | 認証失敗件数 | 10 分で通常比 2 倍 | 10 分で通常比 5 倍 | 認証攻撃調査 |
| バックアップ | 日次実行結果 | 失敗 1 回 | 失敗 2 回連続 | バックアップ復旧 |

### 5.3 監視実装要件

- 顧客向け主要導線に対して 5 分間隔の synthetic monitoring を設定すること
- Web / API は構造化ログ、メトリクス、ヘルスエンドポイントを提供すること
- 監視 dashboard には `requestId` からログへ辿れる導線を持たせること
- すべての critical alert には runbook URL と直近デプロイ情報を添付すること
- `stock_projection` freshness は command 成功時刻と projection 更新時刻の差分で算出すること

### 5.4 エスカレーション

| 重要度 | 条件 | 通知先 | 初動目標 |
| :--- | :--- | :--- | :--- |
| P1 | 注文不能、全体停止、データ消失疑い | 当番開発者、運用責任者、事業責任者 | 15 分以内に一次報告 |
| P2 | 一部機能停止、業務遅延、SLO 大幅逸脱 | 当番開発者、運用責任者 | 30 分以内に暫定対処方針提示 |
| P3 | 軽微な性能劣化、warning 継続 | 当番開発者 | 翌営業日までに対応計画化 |

## 6. バックアップ / リストア設計

### 6.1 バックアップ対象

| 対象 | 方式 | 頻度 | 保持 | 備考 |
| :--- | :--- | :--- | :--- | :--- |
| PostgreSQL | 自動 snapshot + PITR | 日次 snapshot / 継続アーカイブ | 30 日 | RPO 15 分以内を満たす |
| 商品画像など object storage | versioning + lifecycle | 更新時随時 / 日次整合確認 | 30 日以上 | 導入時に有効化 |
| 監査ログ | 集約保存 | 日次転送 | 1 年以上 | 重要操作は改ざん耐性を考慮 |
| IaC / 設定 | Git 管理 | 変更時 | 永続 | secrets の平文 export はしない |

### 6.2 リストア要件

| シナリオ | 目標 | 手順概要 |
| :--- | :--- | :--- |
| DB 誤更新 | 60 分以内に復旧 | 影響範囲確認 → PITR で staging 復旧 → 差分確認 → production 反映 |
| DB 全損 | 60 分以内に代替 DB へ復旧 | 最新 snapshot 復旧 → アプリ接続切替 → smoke test |
| `stock_projection` 破損 | 30 分以内に再構築開始 | command 正本確認 → projection truncate / rebuild → freshness 回復確認 |
| object storage 誤削除 | 120 分以内に復旧 | versioning から復元 → 参照確認 |

### 6.3 リストア演習

- staging で四半期ごとに `orders` と `stock_projection` を含む復旧演習を行うこと
- production バックアップを使った restore 確認を半年ごとに行うこと
- 演習では開始時刻、復旧完了時刻、失敗点、手順差分を記録し、runbook を更新すること

## 7. 障害対応設計

### 7.1 代表的な障害パターン

| 障害パターン | 検知方法 | 暫定対応 | 恒久対策候補 |
| :--- | :--- | :--- | :--- |
| 注文 API 障害 | synthetic check、5xx alert | 直近デプロイ切り戻し、依存先確認 | 回帰 test 追加、負荷対策 |
| DB 負荷 / 接続枯渇 | DB CPU / 接続 alert | 高負荷 query 停止、接続 pool 調整 | index 最適化、上限見直し |
| `stock_projection` 遅延 | freshness alert、job failure | 再計算 job 実行、対象期間限定 rebuild | 更新処理改善、監視強化 |
| 認証障害 | login failure 急増、401 / 403 異常 | 認証基盤 / secret / cookie 設定確認 | auth test 拡充、設定監査 |
| バックアップ失敗 | backup alert | 再試行、managed service 状態確認 | backup 設定改善、容量確保 |

### 7.2 初動手順

1. alert 受信後 5 分以内に顧客影響の有無を判定する。
2. 影響が注文不能またはデータ破損疑いの場合は P1 としてエスカレーションする。
3. 直近 24 時間の変更、deploy、migration、feature flag 変更を確認する。
4. 暫定復旧を優先し、必要なら deploy rollback または feature flag 無効化を実施する。
5. 復旧後 1 営業日以内に incident 記録を残し、再発防止策を backlog 化する。

### 7.3 連絡と報告

| タイミング | 内容 | 宛先 |
| :--- | :--- | :--- |
| 検知後 15 分以内 | 一次報告、影響範囲、暫定対応方針 | 運用責任者、事業責任者 |
| 復旧時 | 復旧完了、残課題、顧客影響有無 | 関係者全体 |
| 1 営業日以内 | incident report、再発防止策 | 開発チーム、事業責任者 |

## 8. 変更管理設計

### 8.1 標準リリースフロー

| 手順 | 必須条件 |
| :--- | :--- |
| PR 作成 | 変更目的、影響範囲、テスト結果、必要なら runbook 更新を記載する |
| CI 実行 | format / lint / unit / integration / build / docs build が成功する |
| staging 反映 | 自動 deploy と smoke test が成功する |
| production 承認 | 運用責任者または担当者が rollback 手順を確認する |
| production 反映 | Rolling Update 後に health check と主要導線確認を行う |

### 8.2 ロールバック要件

| 変更種別 | ロールバック方法 | 事前条件 |
| :--- | :--- | :--- |
| アプリケーション | 直前イメージへ再 deploy | 前バージョン image が保持されていること |
| 設定変更 | Git 管理された設定へ戻す | 変更がコード化されていること |
| migration | 原則 forward fix。必要時のみ事前定義した rollback script を使う | 破壊的変更は expand / contract で実施 |
| feature flag | flag を無効化 | 切替条件が明文化されていること |

### 8.3 緊急変更

- P1 障害対応のための緊急変更は、当番開発者と運用責任者の合意で実施可能とすること
- 緊急変更後は次営業日までに通常 PR 形式で履歴と恒久対策を残すこと
- 口頭 / チャットのみの変更完了扱いを禁止し、必ず issue または PR に記録すること

## 9. セキュリティ運用設計

### 9.1 定常セキュリティ運用

| 項目 | 頻度 | 要件 |
| :--- | :--- | :--- |
| 依存脆弱性確認 | PR ごと + 日次 | critical / high は production 反映前に解消する |
| secrets 棚卸し | 月次 | 不要 secret を削除し、利用者を確認する |
| 権限レビュー | 月次 | RBAC と運用権限を見直す |
| 監査ログ確認 | 週次 | 権限変更、認証失敗急増、重要操作を確認する |

### 9.2 セキュリティインシデント対応

- 不正アクセス疑い時は対象アカウント / token を即時失効できること
- 監査ログと access log を `requestId`、`actorId`、`targetId` で相互参照できること
- 影響する secret は 24 時間以内に rotate すること

## 10. 運用成果物と見直し

### 10.1 必須成果物

- alert 一覧と閾値表
- P1 / P2 runbook
- backup / restore 手順
- release / rollback 手順
- incident report テンプレート
- 月次運用レポート

### 10.2 見直しトリガー

- 月間 SLA 99.5% を下回った場合
- 注文 API 成功率 SLO 99.9% を下回った場合
- 同一種類の P1 / P2 障害が四半期内に 2 回以上発生した場合
- DB 容量、接続数、job 実行時間が想定上限の 80% を超えた場合
- 認証方式や deployment topology を変更する ADR が承認された場合

## 11. 未確定事項

- スタッフ認証基盤の詳細実装方式は別途確定が必要です
- object storage 導入後の画像バックアップ詳細は実装フェーズで補完します
- 24/7 オンコールを正式に置くかは受注量拡大時に再判断します
