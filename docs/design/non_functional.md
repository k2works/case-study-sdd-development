# 非機能要件定義 - フレール・メモワール WEB ショップシステム

## 性能要件

### レスポンスタイム

| 操作 | p50 | p95 | p99 | 備考 |
| :--- | :--- | :--- | :--- | :--- |
| 商品一覧表示 | 200ms | 500ms | 1s | 静的データのため高速 |
| 注文登録 | 300ms | 800ms | 2s | 在庫引当処理を含む |
| 在庫推移表示 | 500ms | 1s | 3s | 日別計算のため重め |
| 受注一覧表示 | 200ms | 500ms | 1s | ページネーション適用 |
| 出荷登録 | 300ms | 800ms | 2s | ステータス更新を含む |

### スループット・同時接続

| 指標 | 目標値 | 備考 |
| :--- | :--- | :--- |
| 同時接続ユーザー数（ピーク） | 50 ユーザー | 小規模 WEB ショップ想定 |
| 1 日あたり受注件数 | 最大 200 件 | 将来的な拡張を考慮 |
| API スループット | 100 req/s | ピーク時 |

## セキュリティ要件

### 認証・認可

| 対象 | 方式 | 詳細 |
| :--- | :--- | :--- |
| 顧客向け画面 | セッション認証 | メールアドレス + パスワード |
| 管理画面 | セッション認証 | スタッフ ID + パスワード |
| 認可モデル | RBAC | 顧客 / 受注スタッフ / 仕入スタッフ / フローリスト |

### ロール定義

| ロール | アクセス可能な機能 |
| :--- | :--- |
| 顧客 | 商品一覧・注文登録・届け日変更 |
| 受注スタッフ | 受注管理・出荷管理・商品マスタ・得意先管理 |
| 仕入スタッフ | 在庫推移・発注管理・入荷管理・単品マスタ |
| フローリスト | 出荷対象一覧（参照のみ） |

### データ保護

| 対象 | 要件 |
| :--- | :--- |
| 通信経路 | HTTPS（TLS 1.2 以上）を必須とする |
| パスワード | bcrypt でハッシュ化して保存する |
| 個人情報（氏名・住所・電話番号） | DB 保存時は平文可（通信は HTTPS で保護） |
| セッション | HttpOnly + Secure Cookie、有効期限 24 時間 |

### セキュリティ対策

| 脅威 | 対策 |
| :--- | :--- |
| SQL インジェクション | Prisma の Prepared Statement を使用 |
| XSS | React の自動エスケープ + CSP ヘッダー設定 |
| CSRF | SameSite Cookie + CSRF トークン |
| 不正アクセス | ログイン失敗 5 回でアカウントロック（30 分） |
| 過剰リクエスト | レートリミット（100 req/分/IP） |

### 監査ログ

以下の操作を監査ログとして記録する：

- 受注登録・届け日変更・キャンセル
- 発注登録・入荷登録・出荷登録
- マスタデータの変更（商品・単品・得意先）
- 管理画面へのログイン・ログアウト

記録項目: 操作日時 / 操作者 ID / 操作種別 / 対象リソース ID / 変更前後の値

## 可用性要件

### SLA

| 環境 | 稼働率目標 | 許容ダウンタイム（月） |
| :--- | :--- | :--- |
| 本番環境 | 99.5% | 約 3.6 時間 |
| ステージング環境 | 95% 以上 | - |

### 障害復旧目標

| 指標 | 目標値 | 備考 |
| :--- | :--- | :--- |
| RTO（復旧時間目標） | 4 時間以内 | 障害発生から復旧まで |
| RPO（復旧時点目標） | 24 時間以内 | データ損失の許容範囲 |

### メンテナンス方針

- 定期メンテナンスは深夜帯（0:00〜6:00）に実施する
- メンテナンス時はメンテナンスページを表示する
- 計画メンテナンスは 3 日前に告知する

## 保守性要件

### ログ出力

| ログレベル | 出力条件 | 保持期間 |
| :--- | :--- | :--- |
| ERROR | 例外・システムエラー | 90 日 |
| WARN | バリデーションエラー・在庫不足 | 30 日 |
| INFO | 受注登録・出荷登録等の業務イベント | 30 日 |
| DEBUG | 開発環境のみ | 7 日 |

ログ形式: JSON（timestamp / level / message / requestId / userId）

### 監視項目

| 監視対象 | 指標 | 閾値（警告） | 閾値（緊急） |
| :--- | :--- | :--- | :--- |
| API レスポンスタイム | p95 | > 1s | > 3s |
| エラーレート | 5xx 率 | > 1% | > 5% |
| DB 接続数 | 使用率 | > 70% | > 90% |
| ディスク使用率 | 使用率 | > 70% | > 90% |
| メモリ使用率 | 使用率 | > 80% | > 95% |

### アラート設定

| 重要度 | 通知先 | 対応時間目標 |
| :--- | :--- | :--- |
| 緊急（P1） | 担当者へ即時通知 | 30 分以内に対応開始 |
| 警告（P2） | 担当者へ通知 | 4 時間以内に確認 |
| 情報（P3） | ログ記録のみ | 翌営業日に確認 |

## 拡張性要件

### スケーラビリティ方針

| 対象 | 方針 |
| :--- | :--- |
| バックエンド | 水平スケーリング可能な設計（ステートレス API） |
| DB | 垂直スケーリングを基本とし、必要に応じてリードレプリカを追加 |
| フロントエンド | CDN による静的ファイル配信でスケール |

### 将来の拡張シナリオ

| シナリオ | 対応方針 |
| :--- | :--- |
| 受注件数が 10 倍に増加 | バックエンドの水平スケーリング + DB リードレプリカ追加 |
| 法人顧客対応 | Customer エンティティに顧客種別を追加 |
| 複数店舗対応 | テナント ID をデータモデルに追加（マルチテナント化） |
| 自動発注機能 | StockTransitionService を拡張して発注推奨ロジックを追加 |

## 制約条件

| 制約 | 内容 |
| :--- | :--- |
| 対応ブラウザ | Chrome / Firefox / Safari の最新 2 バージョン |
| 対応デバイス | PC（1024px 以上）を主対象。スマートフォン対応は将来検討 |
| データ保持期間 | 受注データ: 5 年 / ログ: 90 日 / 監査ログ: 1 年 |
