# 非機能要件

## 1. 文書の目的

本書は、フレール・メモワール WEB ショップシステムにおける非機能要件を、測定可能な形で定義するものです。

機能要件とアーキテクチャ設計を前提に、性能、セキュリティ、可用性、保守性、拡張性、使用性の観点から、設計・実装・運用の判断基準を明確化します。

## 2. 前提と適用範囲

### 2.1 前提

- システムは顧客向け注文導線とスタッフ向け業務画面を持つ単一 Web システムです
- バックエンドはモジュラーモノリス + 4 層レイヤードを採用します
- 正本データは PostgreSQL に集約し、`stock_projection` は query model として管理します
- 小規模な XP チームが `local`、`staging`、`production` の 3 環境で継続運用します

### 2.2 適用対象

- 顧客向け Web 画面
- スタッフ向け Web 画面
- REST API
- PostgreSQL
- scheduler / background job
- CI/CD と監視運用

## 3. サービスレベル目標

### 3.1 SLA / SLO

| 項目 | 目標 | 備考 |
| :--- | :--- | :--- |
| 月間可用性 SLA | 99.5% 以上 | planned maintenance を除く |
| 顧客向け注文 API 成功率 SLO | 99.9% 以上 | `POST /api/orders`、月間 |
| スタッフ向け主要 API 成功率 SLO | 99.5% 以上 | 在庫推移、発注、入荷、届け日変更 |
| `stock_projection` 更新 freshness | 95% 以上が 60 秒以内 | 同期更新を基本、再計算 job 含む |
| critical 障害の一次検知 | 5 分以内 | 監視アラート発火まで |

### 3.2 想定負荷

| 指標 | 通常時 | ピーク時 |
| :--- | :--- | :--- |
| 同時アクティブ顧客ユーザー | 20 | 50 |
| 同時アクティブスタッフユーザー | 5 | 15 |
| 1 日あたり注文件数 | 100 | 300 |
| 1 日あたり届け日変更件数 | 10 | 30 |
| 1 日あたり発注 / 入荷更新件数 | 20 | 80 |

### 4.3 バッチ / scheduler 要件

| 処理 | 目標 |
| :--- | :--- |
| 廃棄リスク再計算 | 1 回 5 分以内 |
| 出荷対象集計 | 対象日 1 日分を 2 分以内 |
| 通知再試行処理 | 1 ジョブ 1 分以内 |
| 再計算 job の再実行開始 | 障害検知から 10 分以内 |

### 4.4 容量要件

- 3 年分の注文、変更履歴、発注、入荷、出荷、在庫 projection を保持しても、通常検索性能を維持できること
- `orders` は 50 万件、`stock_projections` は 1,000 万件を初期の上限想定とし、p95 を維持できること
- DB 接続数は通常時 50 以下、ピーク時 100 以下に抑えること

## 5. セキュリティ要件

### 5.1 守るべき資産

- 顧客の氏名、住所、電話番号、メールアドレス
- 注文情報、届け日変更履歴、出荷対象情報
- スタッフの操作履歴
- 発注 / 入荷データ

### 5.2 認証・認可要件

| 項目 | 要件 |
| :--- | :--- |
| スタッフ認証 | 必須。セッションベースまたは OIDC 連携可能な構成にする |
| スタッフ認可 | RBAC を採用し、少なくとも `order_staff`、`purchase_staff`、`florist`、`owner` を分離する |
| 顧客導線 | 注文はゲスト利用可。ただし届け先再利用と届け日変更は本人確認済みセッションまたはワンタイム認証 URL を必須とする |
| セッション管理 | HttpOnly、Secure、SameSite=Lax 以上の Cookie を利用する |
| 権限エラー | 未認証は 401、権限不足は 403 で返す |

### 5.3 データ保護要件

- 通信はすべて TLS 1.2 以上とする
- 保存データはマネージドサービス標準の暗号化 at rest を有効化する
- 秘密情報は Secrets 管理に置き、リポジトリに保存しない
- ログには住所全文、電話番号全文、メールアドレス全文を出力しない
- バックアップデータにも暗号化を適用する

### 5.4 監査・追跡要件

| 対象操作 | 監査ログ必須 | 保持期間 |
| :--- | :--- | :--- |
| 注文確定 | 必須 | 1 年以上 |
| 届け日変更判定 / 確定 | 必須 | 1 年以上 |
| 発注登録 / 入荷登録 | 必須 | 1 年以上 |
| スタッフ権限が必要な参照操作 | 必須 | 180 日以上 |
| 認証成功 / 失敗 | 必須 | 180 日以上 |

監査ログには少なくとも `timestamp`、`requestId`、`actorType`、`actorId`、`action`、`targetId`、`result` を含めます。

### 5.5 脆弱性管理要件

- 依存パッケージの脆弱性スキャンを PR と定期実行で行うこと
- critical / high 脆弱性は production リリース前に解消すること
- セキュリティパッチは公開から 14 日以内の適用を目標とすること

## 6. 可用性・信頼性要件

### 6.1 稼働目標

| 項目 | 目標 |
| :--- | :--- |
| 月間可用性 | 99.5% 以上 |
| planned maintenance 通知 | 24 時間前まで |
| critical API 継続失敗の自動検知 | 5 分以内 |

### 6.2 障害復旧目標

| 項目 | 目標 |
| :--- | :--- |
| RTO | 60 分以内 |
| RPO | 15 分以内 |
| staging での復旧手順検証 | 四半期ごと 1 回以上 |
| production バックアップリストア確認 | 半年ごと 1 回以上 |

### 6.3 障害許容要件

- Web / API のどちらか一方の再起動で全データが失われないこと
- `stock_projection` 更新失敗時はコマンドをロールバックし、再計算 job で復旧可能であること
- DB 障害時は自動バックアップから復旧できること

## 7. 保守性・運用性要件

### 7.1 ログ要件

| 項目 | 要件 |
| :--- | :--- |
| ログ形式 | JSON の構造化ログ |
| 必須キー | `timestamp`、`level`、`requestId`、`orderId`、`customerId`、`shipDate` |
| アプリログ保持期間 | 30 日以上 |
| 監査ログ保持期間 | 180 日以上、重要操作は 1 年以上 |
| PII マスキング | 必須 |

### 7.2 監視要件

| 対象 | 指標 | 閾値 / 基準 |
| :--- | :--- | :--- |
| Web | p95 応答時間 | 2 秒超で warning、3 秒超で critical |
| API | 5xx 率 | 5 分平均 1% 超で warning、3% 超で critical |
| 注文 API | 成功率 | 15 分平均 99% 未満で warning |
| DB | CPU 使用率 | 80% 超が 15 分継続で warning |
| DB | 接続数 | 80 接続超で warning、100 超で critical |
| Job | 失敗回数 | 同一 job 3 回連続失敗で critical |
| `stock_projection` freshness | 更新遅延 | 60 秒超で warning、5 分超で critical |

### 7.3 アラート要件

- critical は即時通知、warning は営業時間内通知とする
- 通知先は少なくとも開発 / 運用責任者 1 名以上を含むこと
- アラートには runbook または復旧手順リンクを含めること

### 7.4 変更容易性要件

- モジュール間の循環依存を禁止すること
- 新しい業務ルール変更時に、対応するテスト追加なしでは merge しないこと
- PR から本番反映までの標準 lead time を 3 営業日以内に保つことを目標とすること

## 8. 使用性・アクセシビリティ要件

### 8.1 使用性

| 項目 | 目標 |
| :--- | :--- |
| 顧客向け注文完了までのステップ数 | 5 ステップ以内 |
| 注文フォーム主要エラーの即時表示 | 入力後 500ms 以内 |
| スタッフ向け一覧の主要操作 | マウスまたはキーボードのみで完結可能 |

### 8.2 アクセシビリティ

- 顧客向け主要導線は WCAG 2.1 AA 相当を目標とする
- すべてのフォームにラベルとエラーメッセージを持つこと
- 状態表示は色だけに依存しないこと
- 主要操作の成功 / 失敗は Toast だけでなく画面上にも残すこと

## 9. 拡張性要件

### 9.1 スケーラビリティ方針

| 項目 | 方針 |
| :--- | :--- |
| Web | 水平スケール前提 |
| API | 水平スケール前提 |
| DB | 初期は垂直スケール、必要時に read 負荷分散を検討 |
| Job | 初期は API 同居、失敗率や実行時間増大時に分離 |

### 9.2 将来拡張シナリオ

- 顧客アカウント本格導入
- 配送業者連携
- 商品画像配信基盤の追加
- KPI ダッシュボード強化
- 店舗数 / スタッフ数増加

### 9.3 拡張時の制約

- ADR なしでアーキテクチャ境界を変更しないこと
- `stock_projection` を非同期化する場合は整合性戦略を再定義すること
- `items.supplier_id` の単一仕入先前提を崩す場合はデータモデルを見直すこと

## 11. リスクと見直しポイント

| リスク | 影響 | 対策 |
| :--- | :--- | :--- |
| 顧客本人確認方式が未確定 | セキュリティ要件が揺れる | 認証設計を次段で確定し、本書を更新する |
| 在庫 projection の負荷増大 | p95 劣化、更新遅延 | 遅延監視、差分更新、job 分離を段階的に導入する |
| DB 単一障害点 | サービス停止 | managed DB、バックアップ、復旧訓練で補う |
