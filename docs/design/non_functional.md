# 非機能要件

本書は、フラワーショップ「フレール・メモワール」 WEB ショップシステムの非機能要件を定義します。少人数運用と将来拡張の両立を前提に、性能、可用性、セキュリティ、保守性、拡張性、使用性を測定可能な形で整理します。

## 前提

- 初期リリースは個人顧客向け WEB ショップと店舗スタッフ向け管理画面を対象とする
- インフラは AWS のマネージドサービス中心構成を前提とする
- MVP ではシンプルな運用を優先しつつ、Phase 2 以降の拡張余地を残す

## 性能要件

### 画面 / API 応答時間

| 対象 | p50 | p95 | p99 |
| :--- | :--- | :--- | :--- |
| 商品一覧表示 | 1.0 秒以内 | 2.0 秒以内 | 3.0 秒以内 |
| 注文確認表示 | 1.0 秒以内 | 2.0 秒以内 | 3.0 秒以内 |
| 注文確定 API | 1.5 秒以内 | 3.0 秒以内 | 5.0 秒以内 |
| 受注一覧検索 | 1.0 秒以内 | 2.0 秒以内 | 3.0 秒以内 |
| 在庫推移表示 | 1.5 秒以内 | 3.0 秒以内 | 5.0 秒以内 |
| 出荷確定 API | 1.0 秒以内 | 2.0 秒以内 | 3.0 秒以内 |

### スループット / 容量

| 指標 | 目標 |
| :--- | :--- |
| 同時アクティブ顧客数 | 100 ユーザー以上 |
| 同時アクティブスタッフ数 | 20 ユーザー以上 |
| 注文確定処理 | 1 分あたり 30 件以上 |
| 在庫推移参照 | 1 分あたり 120 リクエスト以上 |
| 保持受注件数 | 5 年分をオンライン参照可能 |

### バッチ / 再計算

| 対象 | 目標 |
| :--- | :--- |
| 在庫推移再計算 | 5 分以内に完了 |
| マスタ変更反映 | 10 分以内に表示へ反映 |

## 可用性 / 信頼性要件

### SLA / SLO

| 指標 | 目標 |
| :--- | :--- |
| 月間稼働率 | 99.5% 以上 |
| 計画停止 | 月 2 時間以内 |
| 重大障害件数 | 四半期 1 件以下 |

### 障害復旧

| 指標 | 目標 |
| :--- | :--- |
| RTO | 4 時間以内 |
| RPO | 15 分以内 |
| 主要データバックアップ | 1 日 1 回以上 + 自動スナップショット |

### データ保全

- 受注、出荷、発注、入荷、監査ログはバックアップ対象とする
- 受注の届け先スナップショットは履歴不変データとして扱う
- 重要テーブルの論理削除または履歴保持方針を運用要件で明確化する

## セキュリティ要件

### 認証 / 認可

| 項目 | 要件 |
| :--- | :--- |
| 顧客認証 | メールアドレスと電話番号による履歴照合を MVP で利用し、将来は会員認証へ拡張可能にする |
| スタッフ認証 | 管理画面はスタッフ専用認証を必須とする |
| 認可モデル | ロールベースアクセス制御（RBAC）を採用する |
| ロール | 受注スタッフ、仕入スタッフ、フローリスト、管理者 |
| partner API | 顧客 / スタッフ向け経路とは別認証方式を適用し、署名検証、冪等性キー、再送方針を持つ |

### 通信 / 保存データ保護

| 項目 | 要件 |
| :--- | :--- |
| 通信暗号化 | 外部通信は TLS 1.2 以上を必須とする |
| 保存データ暗号化 | DB、オブジェクトストレージ、バックアップは暗号化する |
| シークレット管理 | Secrets Manager 等で一元管理し、リポジトリへ平文保存しない |

### 監査 / 追跡

| 項目 | 要件 |
| :--- | :--- |
| 監査対象 | 受注変更、届け日変更、発注確定、入荷登録、出荷確定、マスタ更新 |
| 保持項目 | 誰が、いつ、何を、変更前後どうしたか |
| 監査ログ保持期間 | 3 年以上 |

## 保守性要件

### ログ / 監視

| 項目 | 要件 |
| :--- | :--- |
| アプリケーションログ | JSON 形式で構造化出力する |
| ログレベル | `error` `warn` `info` を本番で標準使用する |
| ログ保持期間 | アプリ 90 日、監査ログ 3 年 |
| 監視対象 | API 応答時間、HTTP 5xx、DB 接続、ジョブ失敗、ECS タスク異常 |
| アラート初動 | 重大度 `高` は 15 分以内に確認開始 |

### 変更容易性

| 項目 | 要件 |
| :--- | :--- |
| 依存方向 | ドメイン層はフレームワークへ直接依存しない |
| テスト | バックエンド行カバレッジ 85% 以上、フロント Feature 80% 以上 |
| ADR | 重要設計判断は ADR に記録する |

## 拡張性要件

### スケーラビリティ

| 項目 | 方針 |
| :--- | :--- |
| アプリケーション | ECS タスクの水平スケールで対応する |
| DB | 初期は垂直スケール中心、負荷増加時に read replica を検討する |
| 在庫推移再計算 | 同期処理で限界が見えた時点で Redis / BullMQ を段階導入する |

### 将来拡張

- 会員認証の追加
- 配送状況通知の追加
- partner API の非同期連携化
- 発注支援ロジックの高度化

## 使用性 / アクセシビリティ要件

| 項目 | 要件 |
| :--- | :--- |
| 顧客注文導線 | 初回利用者が 5 ステップ以内で注文完了できる構成にする |
| 入力エラー表示 | 項目単位と画面上部要約の両方で表示する |
| キーボード操作 | 主要フォームはキーボードのみで完結できる |
| 視認性 | 色だけに依存しない状態表示を行う |
| 対応デバイス | 顧客画面はスマートフォン / PC、管理画面は PC を主対象とする |
| フォーカス制御 | エラー時は要約または最初の不正項目へフォーカス移動する |
| 操作対象サイズ | 主要操作は 44px 四方以上を目安にする |

## 互換性 / 移植性要件

| 項目 | 要件 |
| :--- | :--- |
| 顧客画面対応ブラウザ | 主要モダンブラウザの最新版と 1 世代前を対象 |
| 管理画面対応ブラウザ | Chrome 最新版を標準対象 |
| 開発環境再現性 | Docker ベースでローカル再現可能にする |

## 制約

- MVP では 99.9% 以上の高可用性は目標としない
- Redis / BullMQ は初期導入必須とせず、性能要件を超過した時点で導入判断する
- partner API の詳細認証方式とエラーモデルは運用要件で具体化する

## 受け入れ基準

- 性能テストで主要 API が p95 目標を満たす
- バックアップと復元手順が RTO / RPO を満たすことを Staging で検証する
- 監査対象操作がすべてログに残る
- テスト戦略に定義したセキュリティ / 性能観点の検証が CI または Staging で実施される
