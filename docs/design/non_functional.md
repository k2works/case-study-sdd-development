# 非機能要件定義

## 前提

- インフラ: Heroku PaaS（Web Dyno x1〜2、Heroku Postgres）
- アーキテクチャ: Rails モノリシック MVC + ActiveRecord
- 利用者: 個人顧客（得意先）+ スタッフ（数名）
- 規模: 小規模 EC（日次受注数十件オーダー）

## 性能要件

### レスポンスタイム

| 操作カテゴリ | p50 | p95 | p99 | 備考 |
|------------|-----|-----|-----|------|
| 画面表示（一覧系） | 300ms | 1,000ms | 2,000ms | 商品カタログ、受注一覧、出荷一覧 |
| 画面表示（詳細系） | 200ms | 500ms | 1,000ms | 受注詳細、商品詳細 |
| データ登録・更新 | 300ms | 1,000ms | 2,000ms | 注文確定、出荷処理、発注登録 |
| 在庫推移計算 | 500ms | 2,000ms | 3,000ms | StockForecastService の計算 |
| ログイン | 500ms | 1,500ms | 3,000ms | Devise 認証処理 |

### スループット

| 指標 | 目標値 | 備考 |
|------|--------|------|
| 同時接続数 | 10（初期）〜 50（成長期） | Heroku Web Dyno のスレッド数に依存 |
| 日次受注件数 | 100 件/日 | ピーク時（母の日等）は 3 倍を想定 |
| ページビュー | 1,000 PV/日 | 商品カタログの閲覧を含む |

### パフォーマンス対策

| 対策 | 対象 | 実装方針 |
|------|------|---------|
| N+1 クエリ防止 | 全一覧系画面 | `includes` / `preload` を徹底 |
| ページネーション | 一覧画面（受注、発注、出荷） | Kaminari gem、1 ページ 20 件 |
| DB インデックス | 検索頻度の高いカラム | data-model.md のインデックス設計に準拠 |
| 在庫推移キャッシュ | StockForecastService | 初期はキャッシュなし。必要に応じて導入 |

## セキュリティ要件

### 認証・認可

| 項目 | 方式 | 備考 |
|------|------|------|
| 認証方式 | メールアドレス + パスワード（Devise） | 得意先・スタッフ共通 |
| パスワード要件 | 最低 8 文字、英数混合 | Devise の設定で制御 |
| セッション管理 | Cookie ベース（Rails 標準） | セッションタイムアウト: 30 分 |
| 認可方式 | ロールベース（customer / staff） | Controller の before_action で制御 |
| リソース認可 | スコープ制御 | 得意先は自分の注文のみアクセス可能 |

### 通信・データ保護

| 項目 | 方式 | 備考 |
|------|------|------|
| 通信暗号化 | HTTPS（TLS 1.2 以上） | Heroku の自動 SSL |
| パスワード暗号化 | bcrypt | Devise 標準 |
| 環境変数 | Heroku Config Vars | コードに含めない |
| CSRF 対策 | Rails 標準の authenticity_token | 全フォームに自動適用 |
| SQL インジェクション | ActiveRecord Parameterized Query | 生 SQL を使わない |
| XSS 対策 | Rails の自動エスケープ | `html_safe` の使用を禁止 |

### セキュリティスキャン

| ツール | 実行タイミング | 対象 |
|--------|--------------|------|
| Brakeman | CI パイプライン（全 Push） | Rails セキュリティ脆弱性 |
| bundler-audit | CI パイプライン（全 Push） | gem の既知の脆弱性 |
| rack-attack | 実行時（常時） | ブルートフォース・DDoS 軽減 |

### レートリミット

| 対象 | 制限 | 備考 |
|------|------|------|
| ログイン試行 | 5 回/分/IP | rack-attack で制御 |
| API リクエスト | 60 回/分/IP | rack-attack で制御 |
| 会員登録 | 3 回/時/IP | rack-attack で制御 |

## 可用性要件

### 稼働率目標

| 項目 | 目標値 | 備考 |
|------|--------|------|
| 月間稼働率 | 99.5%（約 3.6 時間/月のダウンタイム許容） | Heroku の SLA に準拠 |
| 計画メンテナンス | 月 1 回、深夜帯に実施 | デプロイ時の瞬断は許容 |

### 障害復旧

| 項目 | 目標値 | 備考 |
|------|--------|------|
| RTO（復旧目標時間） | 4 時間 | Heroku の自動復旧 + 手動対応 |
| RPO（復旧目標時点） | 24 時間 | 日次バックアップからの復旧 |

### バックアップ

| 対象 | 方式 | 頻度 | 保持期間 |
|------|------|------|---------|
| データベース | Heroku PGBackups | 日次自動 | 直近 5 世代 |
| コード | GitHub リポジトリ | Push ごと | 無期限 |

## 保守性要件

### ログ

| 項目 | 方式 | 備考 |
|------|------|------|
| アプリケーションログ | Rails.logger → STDOUT | Heroku Logplex で集約 |
| ログレベル | production: info、staging: debug | 環境変数で制御 |
| ログ保持期間 | 直近 7 日（Heroku 標準） | 長期保存が必要な場合はアドオン追加 |
| リクエストログ | Heroku Router Log | レスポンスタイム含む |

### 監視

| 監視対象 | 指標 | 閾値 | 備考 |
|---------|------|------|------|
| レスポンスタイム | p95 | 2,000ms 超で警告 | Heroku Metrics |
| メモリ使用量 | Dyno メモリ | 512MB の 80% で警告 | Heroku Metrics |
| エラー率 | 5xx レスポンス率 | 1% 超で警告 | Heroku Metrics |
| DB 接続数 | アクティブ接続 | 上限の 80% で警告 | Heroku Postgres Metrics |

### コード品質

| 項目 | 基準 | ツール |
|------|------|--------|
| テストカバレッジ | 85% 以上 | SimpleCov |
| コードスタイル | RuboCop 違反 0 | RuboCop |
| セキュリティ | Brakeman 警告 0（Critical） | Brakeman |

## 拡張性要件

### スケーリング方針

| フェーズ | 構成 | トリガー |
|---------|------|---------|
| 初期（MVP） | Web Dyno x1、Postgres Mini | - |
| 成長期 | Web Dyno x2、Postgres Basic | 同時接続 10 超、レスポンスタイム劣化 |
| 将来 | AWS ECS への移行検討 | 同時接続 50 超、Heroku コスト増 |

### 将来の拡張シナリオ

| シナリオ | 影響範囲 | 対応方針 |
|---------|---------|---------|
| 商品画像の追加 | Product モデル + ストレージ | Active Storage + S3 |
| メール通知 | 注文確認、出荷通知 | Action Mailer + SendGrid |
| 決済連携 | 注文フロー | 外部決済 API（Stripe 等） |
| 複数商品注文（カート） | Order モデル + UI | Order → OrderItem の 1:N 化 |
| 配送追跡 | Shipment モデル + UI | 配送業者 API 連携 |

## 非機能要件サマリー

| カテゴリ | 最重要指標 | 目標値 |
|---------|----------|--------|
| 性能 | 画面表示 p95 | 1,000ms 以内 |
| セキュリティ | Brakeman Critical | 0 件 |
| 可用性 | 月間稼働率 | 99.5% |
| 保守性 | テストカバレッジ | 85% 以上 |
| 拡張性 | スケーリング | Dyno 追加で対応可能 |
