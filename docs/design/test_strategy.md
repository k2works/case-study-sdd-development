# テスト戦略

## 1. 文書の目的

本書は、フレール・メモワール WEB ショップシステムに対して、アーキテクチャに適合したテスト戦略を定義するものです。

小規模な XP チームが継続的に変更しても品質を維持できるよう、テスト形状、テストレベルごとの責務、カバレッジ目標、ツール、CI 連携、要件トレーサビリティを定義します。

## 2. テスト戦略の基本方針

### 2.1 採用方針

**システム全体としてはピラミッド型を基本としつつ、フロントエンドでは Feature テストを厚めに置くハイブリッド戦略**を採用します。

- バックエンドは `domain` に業務ルールが厚く載るため、ユニットテスト中心で保護します
- フロントエンドは顧客導線とスタッフ導線の品質が重要なため、Feature テストと主要導線 E2E を明示的に置きます
- 在庫 projection はアーキテクチャ上のストレスポイントのため、シナリオテストを独立して厚くします
- すべての実装は TDD を基本とし、受け入れ観点は BDD 形式で整理します

### 2.2 テスト形状を選ぶ理由

本システムのバックエンドは、モジュラーモノリス + 4 層レイヤードであり、以下の性質を持ちます。

- 届け日変更、在庫引当、品質維持日数、出荷日算出などの業務ルールが複雑です
- 単一 DB 内で整合性を保つ設計であり、マイクロサービスのような外部結合点は多くありません
- `stock_projection` だけは読み取り最適化された query model として、統合シナリオの重点保護が必要です

このため、全体の主形状は **ピラミッド型** が最も適しています。ダイヤモンド型ほど統合テストを主役にする必要はなく、逆ピラミッド型のように E2E を増やしすぎると、小規模チームに対して保守コストが高くなります。

## 3. テスト形状

### 3.1 全体形状

| テストレベル | 目安比率 | 主対象 |
| :--- | :--- | :--- |
| ユニットテスト | 65 〜 75% | バックエンド `domain`、フロントエンドの pure function / mapper / hook |
| 統合テスト | 20 〜 25% | バックエンド `application`、DB 結合、フロントエンド Feature |
| E2E / 受け入れテスト | 5 〜 10% | 顧客向け主要導線、スタッフ向け重要業務導線 |

### 3.2 サブシステム別の重点

| 対象 | 重点 | 理由 |
| :--- | :--- | :--- |
| バックエンド | ユニットテスト重視 | ドメインルールが厚く、変更頻度も高いため |
| フロントエンド | Feature / コンポーネントテスト重視 | UI の組み合わせと状態変化を高速に検証したいため |
| 在庫 projection | シナリオテスト重視 | 複数イベントの相互作用で破綻しやすいため |
| 顧客導線 | E2E 重視 | 注文完了までの体験品質が事業価値に直結するため |

## 4. テストレベルの定義

### 4.1 ユニットテスト

**責務**

- 単一の関数、値オブジェクト、エンティティ、ドメインサービスの振る舞いを検証する
- 境界値、異常系、業務ルールの不変条件を保護する
- 外部依存はモックまたは fake に置き換える

**主対象**

- `ShipmentDateCalculator`
- `OrderDateChangePolicy`
- `RequiredItemExpander`
- `StockAvailabilityPolicy`
- `DestinationSnapshot` 生成ロジック
- ViewModel mapper、URL state 変換、入力バリデーション

### 4.2 統合テスト

**責務**

- コンポーネント間連携、トランザクション境界、DB 結合を検証する
- バックエンドでは `application` と repository 実装の協調を検証する
- フロントエンドでは Feature 単位の UI / API / state の連携を検証する

**主対象**

- 注文確定ユースケース
- 届け日変更の事前判定と確定
- 発注登録と入荷記録
- `stock_projection` 更新
- `OrderFormFeature`
- `StockProjectionFeature`
- `PurchaseOrderFeature`

### 4.3 API 契約テスト

**責務**

- `presentation` 層の HTTP 契約を検証する
- request / response DTO、status code、業務エラーコード、認可境界を保護する
- フロントエンドとの接続面を壊さないことを確認する

**主対象**

- `POST /api/orders`
- `POST /api/orders/{orderId}/date-change-check`
- `POST /api/orders/{orderId}/date-change`
- `GET /api/stock/projections`
- `POST /api/purchase-orders`
- `POST /api/purchase-orders/{id}/arrivals`
- `GET /api/shipments`

### 4.4 シナリオテスト

**責務**

- 複数の業務イベントを通した整合性を検証する
- 特に `stock_projection` の更新順序、差分反映、競合耐性を保護する

**主対象**

- 同日多件注文
- 部分入荷
- 届け日変更による旧日付 / 新日付差分反映
- 発注 → 入荷 → 出荷準備の一連整合

### 4.5 E2E テスト

**責務**

- ユーザー視点で主要シナリオをシステム全体として検証する
- 画面遷移、主要入力、成功 / 失敗表示、認証済み導線を保護する

**主対象**

- 顧客が商品選択から注文完了まで進める
- 過去の届け先を再利用して再注文する
- 届け日変更可否を確認して変更を確定する
- スタッフが在庫推移を見て発注登録する

### 4.6 受け入れテスト

**責務**

- 要件と受入条件を Given-When-Then で確認する
- イテレーション完了時の Done 判定に使う

**方針**

- 顧客価値に直結するストーリーを優先して定義します
- Playwright E2E または API / Feature 統合テストで自動化可能なものから自動化します

## 5. バックエンド戦略

### 5.1 レイヤー別戦略

| レイヤー / 要素 | 主要テスト | 目的 |
| :--- | :--- | :--- |
| `domain` | ユニットテスト | 業務ルール、不変条件、境界値の保護 |
| `application` | 統合テスト | ユースケース、トランザクション、Repository 協調の検証 |
| `presentation` | API 契約テスト | HTTP 契約とエラー契約の保護 |
| `stock_projection` | シナリオテスト | 複数イベント横断の整合性検証 |

### 5.2 重点シナリオ

- 注文確定により必要花材が正しく引き当てられること
- 届け先再利用時も注文には届け先スナップショットが保存されること
- 届け日変更で旧日付と新日付の在庫差分が正しく反映されること
- 発注登録と入荷記録で `stock_projection` が破綻しないこと
- 同日多件注文でも出荷一覧と `stock_projection` が矛盾しないこと

## 6. フロントエンド戦略

### 6.1 テストの置き方

| 対象 | 主要テスト | 目的 |
| :--- | :--- | :--- |
| Feature | コンポーネント / 統合テスト | UI と state、API 連携の保護 |
| 顧客導線 | E2E テスト | 注文体験の保護 |
| スタッフ画面 | Feature / 統合テスト | 一覧操作、再読込、絞り込みの保護 |
| 共通 UI | ユニット / アクセシビリティテスト | 見た目と基本操作の保護 |

### 6.2 重点シナリオ

- 顧客が商品選択から注文完了まで迷わず進めること
- 過去の届け先を使って再注文できること
- 届け日変更時に変更可否が適切に表示されること
- 在庫推移画面で日付変更、絞り込み、再読込が破綻しないこと
- スタッフ画面で色だけに依存しない状態表示が維持されること

## 7. カバレッジ目標

### 7.1 数値目標

| 区分 | 目標 |
| :--- | :--- |
| バックエンド `domain` statement coverage | 85% 以上 |
| バックエンド `application` statement coverage | 70% 以上 |
| バックエンド API 契約対象 endpoint | 主要 endpoint 100% |
| フロントエンド Feature coverage | 主要 Feature 80% 以上 |
| 顧客向け主要導線 E2E | 注文、再注文、届け日変更の 100% |
| `stock_projection` シナリオ | 重点シナリオ 100% |

### 7.2 カバレッジ運用ルール

- 数値だけでなく、重要シナリオの有無を重視します
- 新しい業務ルールを追加する場合は、対応するユニットテストまたはシナリオテストを必須にします
- `stock_projection` と届け日変更は、カバレッジよりも代表シナリオの網羅を優先します

## 8. テストツール方針

### 8.1 推奨ツール

| 対象 | 推奨ツール | 用途 |
| :--- | :--- | :--- |
| TypeScript unit test | Vitest | 高速なユニットテスト実行 |
| Frontend component / Feature test | React Testing Library + Vitest | UI 振る舞いの検証 |
| API 契約テスト | Supertest + Vitest | REST API 契約の検証 |
| DB 統合テスト | Testcontainers | PostgreSQL を含む統合検証 |
| E2E テスト | Playwright | 顧客導線と主要業務導線の検証 |
| Test data / mocking | Factory helper / MSW | UI と API の独立検証 |
| Coverage 計測 | V8 coverage / Istanbul 互換 | CI 上での可視化 |

### 8.2 ツール選定理由

- TypeScript 中心の技術スタックと相性がよいこと
- 小規模チームでもメンテナンスしやすいこと
- component / API / E2E を役割分担しやすいこと

## 9. CI/CD 連携

### 9.1 実行タイミング

| タイミング | 実行内容 |
| :--- | :--- |
| ローカル | 変更対象のユニット / Feature テスト |
| Pull Request | lint、unit test、integration test、API 契約テスト、frontend build、backend build |
| `main` マージ後 | 上記に加えて E2E smoke test、docs build |
| リリース前 | 主要導線 E2E、`stock_projection` シナリオ、回帰確認 |

### 9.2 失敗時の扱い

- PR の品質ゲートで失敗した場合は merge 不可とします
- E2E 失敗時は flaky を疑う前に、まず API 契約と Feature テストの失敗有無を確認します
- `stock_projection` シナリオ失敗はリリースブロッカーとして扱います

## 10. TDD / BDD 方針

### 10.1 TDD

- バックエンドは `domain` → `application` の順に Red-Green-Refactor で進めます
- フロントエンドは外側の Feature 振る舞いからテストを書き、必要な UI 部品へ分解します

### 10.2 BDD

- 顧客価値に直結するシナリオは Given-When-Then で記述します
- 受け入れテストの文章と E2E / 統合テストの名前を対応づけます

## 11. トレーサビリティ

### 11.1 要件とテストの対応

| 要件 / 価値 | 主な自動テスト |
| :--- | :--- |
| 花束を簡単に注文できる | 商品一覧 Feature テスト、注文フォーム Feature テスト、注文完了 E2E |
| 記念日に確実に届ける | `ShipmentDateCalculator` unit、注文 API 契約、届け日変更シナリオ、出荷対象抽出統合 |
| 届け先を再利用して再注文する | `customers` / `orders` 統合、届け先選択 Feature、再注文 E2E |
| 受注内容と変更履歴を一元管理する | 受注一覧 / 詳細 Feature、注文詳細 API 契約、届け日変更統合 |
| 在庫推移を見ながら発注判断する | `stock_projection` シナリオ、在庫推移 Feature、発注登録統合 |
| 廃棄リスクと受注量を把握する | 在庫 projection query テスト、在庫推移画面統合、KPI query テスト |

### 11.2 重点受け入れシナリオ

- Given 顧客が商品を選んでいる、When 届け先と届け日を入力して確定する、Then 注文完了し出荷日が前日で確定する
- Given 顧客が過去の届け先を持っている、When 届け先再利用で再注文する、Then 再利用元を参照しつつ注文にはスナップショットが保存される
- Given 既存注文がある、When 届け日変更を判定する、Then 旧日付と新日付の在庫差分に基づいて可否が返る
- Given 仕入スタッフが在庫推移画面を見ている、When 発注登録と入荷登録を行う、Then 在庫 projection が更新される

## 12. リスクと対策

| リスク | 影響 | 対策 |
| :--- | :--- | :--- |
| E2E を増やしすぎる | 実行時間と保守コストが増える | 顧客主要導線と smoke に限定する |
| `stock_projection` をユニットだけで守ろうとする | イベント横断の破綻を見逃す | シナリオテストを独立して置く |
| UI と API の責務が重複する | 同じ失敗を複数層で検証して遅くなる | mapper / 契約 / Feature の責務を分離する |
| flaky E2E が増える | CI 信頼性が下がる | Feature / API 契約テストを厚くし、E2E は最小限にする |

## 13. 段階的実施

1. バックエンド `domain` のユニットテスト基盤を整える
2. 注文確定、届け日変更、発注登録の統合テストを作る
3. `stock_projection` シナリオテストを追加する
4. フロントエンド Feature テストを注文 / 在庫推移から追加する
5. 顧客向け主要導線の Playwright E2E を追加する
