# テスト戦略

本書は、フラワーショップ「フレール・メモワール」 WEB ショップシステムのテスト戦略を定義します。ヘキサゴナルアーキテクチャと Feature 単位 UI を前提に、ユニットテストを土台とするピラミッド型テストを採用します。

## テスト形状

### 採用形状

- ピラミッド型テスト

### 採用理由

- バックエンドはドメインロジックと判定処理が中心であり、ユニットテストの投資対効果が高い
- 在庫推移、届け日変更可否、出荷確定などの不変条件を高速に検証したい
- UI は重要だが、主要ユーザー導線は限定されているため、E2E は絞って運用できる
- 外部連携は将来増えうるが、現時点では統合テストと契約テストで十分に抑えられる

### 構成比の目安

| テストレベル | 目安比率 |
| :--- | :--- |
| ユニットテスト | 70% |
| 統合テスト | 20% |
| E2E / 受け入れテスト | 10% |

## テストレベルの責務

### ユニットテスト

対象:

- ドメインモデル
- 値オブジェクト
- ドメインサービス
- フロントエンドの画面ローカルロジック
- バリデーション

責務:

- ビジネスルールと不変条件の検証
- 境界値、異常値、状態遷移の検証
- 依存先をモックしたユースケース単体の検証

主な対象例:

- `届け日変更可否判定サービス`
- `在庫推移算出サービス`
- `受注` の状態遷移
- `花束構成` の数量制約
- 注文入力フォームの必須チェック

### 統合テスト

対象:

- API 層とアプリケーション層の連携
- Repository と DB の連携
- 在庫推移算出と永続化
- partner API を含む外部境界

責務:

- トランザクション境界の検証
- DB 制約、一意制約、スナップショット保存の検証
- API 入出力と永続化結果の整合検証

主な対象例:

- 注文確定時に届け先スナップショットが保存される
- `inventory_projections` の一意性が維持される
- 出荷確定で受注状態と出荷実績が同時更新される
- 発注作成で仕入先別に正しいデータが保存される

### E2E / 受け入れテスト

対象:

- 顧客注文導線
- 受注一覧と届け日変更
- 在庫推移確認
- 出荷対象確認と出荷確定
- マスタ保守の主要導線

責務:

- ユーザーストーリーの受け入れ基準を満たすことの確認
- 画面遷移、入力、エラー表示、完了表示の確認
- 主要業務フローが実際の操作で破綻しないことの確認

## テストツール

| テストレベル | ツール | 用途 |
| :--- | :--- | :--- |
| ユニット | Vitest | ドメイン、ユースケース、フロントロジックの高速テスト |
| API 統合 | Vitest + Supertest | NestJS API と HTTP レベルの検証 |
| DB 統合 | Prisma + PostgreSQL テスト DB | Repository と制約検証 |
| E2E | Playwright | 顧客導線、管理画面導線のブラウザテスト |
| 補助 | Test Fixtures / Factory | テストデータ生成の共通化 |

## TDD / BDD 方針

- バックエンド:
  - Red → Green → Refactor をユースケース単位で回す
  - ドメインルールはユニットテストを先に書く
- フロントエンド:
  - 顧客導線は受け入れテスト起点で外側から進める
  - 複雑なフォームロジックはユニットテストも併用する
- 受け入れテスト:
  - ユーザーストーリーの受け入れ基準をそのまま Playwright シナリオへ落とす

## カバレッジ目標

| 対象 | 目標 |
| :--- | :--- |
| バックエンドドメイン / アプリケーション | 行カバレッジ 85% 以上 |
| フロントエンド Feature ロジック | 行カバレッジ 80% 以上 |
| 主要ユースケース統合テスト | MVP / フェーズ 2 の対象 API を 100% カバー |
| E2E | MVP の顧客注文導線と管理画面主要導線を 100% カバー |

## 非機能観点のテスト

- 性能:
  - 在庫推移表示のレスポンス時間
  - 受注一覧検索の応答時間
- セキュリティ:
  - 認可境界
  - partner API 境界
  - 個人情報表示制御
- 回復性:
  - DB 制約違反時のエラー応答
  - 外部連携失敗時の再試行方針

## CI/CD 連携

### Pull Request

- ESLint / Prettier チェック
- ユニットテスト
- API 統合テスト
- 必要最小限の Playwright スモークテスト

### Main ブランチ

- 全ユニットテスト
- 全統合テスト
- 主要 E2E シナリオ
- カバレッジ計測

### Staging 前

- リリース対象ストーリーに対応する受け入れテスト一式
- マイグレーションを含む DB 統合確認

## テストデータ方針

- 商品、花材、仕入条件、届け先履歴は Factory で生成する
- 受注スナップショットの検証用に、履歴更新前後のテストデータを用意する
- 在庫推移テストでは、同一花材・同一日付の重複投影を作らないことを明示的に確認する

## 今後の拡張

- Phase 2 で Redis / BullMQ を導入した場合は、ジョブ統合テストを追加する
- partner API の連携仕様確定後は、契約テストを追加する
- 非機能要件確定後に性能テストしきい値を具体化する
