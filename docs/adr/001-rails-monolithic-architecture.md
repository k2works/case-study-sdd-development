# ADR-001: Rails モノリシック MVC アーキテクチャの採用

## ステータス

承認

## コンテキスト

フレール・メモワール WEB ショップシステムのアーキテクチャパターンを選定する必要がある。以下の背景がある。

- チーム規模: 1-2 名のフルスタック開発者
- ドメインの複雑さ: 中程度（在庫推移計算が最も複雑）
- 初期開発速度: MVP を素早くリリースしフィードバックを得たい
- XP プラクティス: TDD、小さなリリース、シンプルな設計

## 検討した選択肢

### A: Rails モノリシック MVC + ActiveRecord

- 利点: 開発速度が高い、Convention over Configuration、フルスタック統合
- 欠点: 大規模化時のモジュール分離が困難

### B: Rails API + React SPA

- 利点: フロントエンド/バックエンドの独立性、リッチな UI
- 欠点: チーム規模に対して複雑さが過剰、2 つの技術スタックの保守

### C: Node.js (Express/NestJS) + React

- 利点: JavaScript 統一、フロントエンドとの親和性
- 欠点: Rails ほどの生産性フレームワークがない、チームの習熟度

## 決定

**A: Rails モノリシック MVC + ActiveRecord** を採用する。

フロントエンドは Rails の ERB テンプレート + Hotwire（Turbo/Stimulus）で構築し、SPA フレームワークは採用しない。

## 理由

1. **チーム規模に最適**: 1-2 名で最も生産性が高いのは Rails のフルスタック開発
2. **ドメインの複雑さに適合**: ActiveRecord パターンは中程度の複雑さに適する。複雑な業務ロジックは Service Object で分離
3. **開発速度の最大化**: Rails の scaffold、Devise、ActiveRecord により、MVP 開発を高速化
4. **Hotwire で SPA 的 UX**: SPA を導入せずとも Turbo Drive/Frames で十分な UX を実現
5. **将来の拡張性**: モノリシックから始めて、必要に応じてモジュール分割可能

## 影響

- テスト戦略: ダイヤモンド形（統合テスト重視）を採用
- インフラ: Heroku PaaS で簡易にデプロイ可能
- フロントエンド専任は不要
- 将来的に複雑化した場合はモジュラーモノリスへの移行を検討
