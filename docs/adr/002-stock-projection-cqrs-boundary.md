# ADR-002: 在庫推移は限定的 CQRS と同期 projection 更新で開始する

在庫推移は単一 DB を正本としたまま、限定的 CQRS と同期 projection 更新で開始します。

日付: 2026-03-20

## ステータス

承認済み

## コンテキスト

在庫推移画面は読み取り負荷と計算量が高く、注文、届け日変更、発注、入荷、出荷が同じ在庫見通しへ影響します。

- ただし、初期段階では別サービスや別データストアを持つほどの規模ではありません
- 二重引当や projection 崩れを防ぐため、更新整合性を先に固定する必要があります
- 小規模チームでの運用を考えると、まずは正しさを優先し、必要になってから非同期化したいです

## 決定

**全面的な CQRS は採用せず、在庫推移に限って Command / Query のコード上の分離と読み取り用 projection を導入します。初期フェーズでは、影響を受ける `item_id × projection_date` 範囲を同一 PostgreSQL トランザクション内で同期再計算します。**

### 変更箇所

- 正本データは PostgreSQL の取引系テーブルに置く
- `stock_projections` は query model として扱う
- 注文、届け日変更、発注、入荷、出荷確定時に同期的に projection を更新する
- 更新時は対象となる `item_id × projection_date` 範囲を一意キーとロック方針で直列化する
- 更新失敗時はコマンドをロールバックし、必要に応じて再計算ジョブで対象範囲を修復する

### 代替案

- フル CQRS
  - 別サービス、別ストア、イベント基盤の運用負荷が高く、初期フェーズでは過剰なため却下しました
- projection を持たず都度計算
  - スタッフ向け在庫推移画面と判定 API の応答性能が不安定になるため却下しました
- 初期から非同期 job 更新
  - eventual consistency の運用コストが先に増え、業務上重要な在庫判定の説明性が下がるため却下しました

## 影響

### ポジティブ

- 正本と projection の責務が分かれ、読み取り最適化しやすくなります
- 初期段階で在庫整合性を説明しやすく、障害調査もしやすくなります
- 将来的に差分更新や非同期化へ段階的に移行できます

### ネガティブ

- 同期再計算のため、書き込み処理が長くなる可能性があります
- ロック範囲の設計を誤ると、競合やスループット低下の原因になります

## コンプライアンス

- `stock_projections` に `item_id + projection_date` の一意制約があること
- projection 更新が単一 DB トランザクション内で完結していること
- 非同期化する場合は、新しい ADR で整合性戦略を更新すること
