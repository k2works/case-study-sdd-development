# ADR-001: ヘキサゴナルアーキテクチャを採用し、初期段階では CQRS を採用しない

受注、在庫、発注、出荷の中核業務を変更しやすく保つため、バックエンドはヘキサゴナルアーキテクチャを採用し、初期段階では CQRS を導入しません。

日付: 2026-03-19

## ステータス

承認済み

## コンテキスト

- 本システムは、受注、在庫推移、発注、出荷の業務ルールが密接に関係する
- 永続化方式や Web フレームワークに業務ロジックを引きずられない構造が必要
- 将来的に通知や在庫推移再計算などの非同期処理はあり得るが、現時点のユースケースは同期的な CRUD と判定処理が中心
- 初期リリースで CQRS を採用すると、読み書き分離の設計・運用コストが先行する

## 決定

バックエンドアーキテクチャとして、ポートとアダプターによるヘキサゴナルアーキテクチャを採用する。初期段階では CQRS を採用せず、単一のトランザクションモデルで受注、在庫、発注、出荷を扱う。

### 代替案

- レイヤードアーキテクチャ
  - 却下理由: ドメインルールが増えると、アプリケーション層と永続化層の責務が曖昧になりやすい
- 初期から CQRS を採用する
  - 却下理由: 在庫推移の読み取り最適化には有効だが、MVP 時点では複雑性が高すぎる

## 影響

### ポジティブ

- ドメインロジックをフレームワークや DB 実装から分離できる
- テストしやすい構造を維持しやすい
- 将来、必要になった時点で読み取りモデルや非同期処理を追加しやすい

### ネガティブ

- 層やポートの設計が増えるため、初期実装量はやや増える
- チームにヘキサゴナル構成の理解が必要

## コンプライアンス

- ドメイン層が Web フレームワークや ORM に直接依存していないこと
- ユースケースごとのトランザクション境界がアプリケーション層に整理されていること
- 読み取り専用最適化が必要になった時のみ、別 ADR で CQRS 採用を検討すること
