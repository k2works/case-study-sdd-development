# ADR-001: バックエンドはモジュラーモノリス + 4 層レイヤードを採用する

小規模な XP チームでも変更しやすさを保つため、バックエンドはモジュラーモノリス + 4 層レイヤードを採用します。

日付: 2026-03-20

## ステータス

承認済み

## コンテキスト

フレール・メモワール WEB ショップシステムでは、単純な CRUD だけでなく、届け日から出荷日を導出する判断、花束構成から必要花材を展開する計算、注文・発注・入荷・出荷に応じた在庫推移の再計算を扱います。

- 在庫推移、届け日変更、出荷対象抽出など、複数の業務ルールが密接に関係します
- 一方で、開発チームは 1 〜 2 名規模であり、過剰な分散構成や抽象化は運用負荷になりやすいです
- 将来の変更は `orders`、`stock`、`purchasing`、`shipping` を中心に局所化したいです

## 決定

**バックエンドは 1 つの Node.js API アプリケーション上に、業務モジュールごとのモジュラーモノリスとして構成し、各モジュール内部は `presentation` / `application` / `domain` / `infrastructure` の 4 層で整理します。**

### 変更箇所

- `catalog`、`orders`、`customers`、`stock`、`purchasing`、`shipping` を主要モジュールとする
- `domain` は他レイヤーへ依存しない
- `presentation -> application -> domain` を基本依存方向とする
- 永続化、外部通知、定時処理など変更されやすい外部境界のみポート / アダプターで抽象化する

### 代替案

- 3 層レイヤード
  - 実装は速いが、在庫推移や届け日変更判定のような業務ロジックが Service に集中しやすいため却下しました
- フルヘキサゴナル / Clean Architecture
  - 境界は明確になるが、小規模チームに対して抽象の数が増えすぎるため却下しました
- マイクロサービス
  - デプロイや運用の複雑性が先に立ち、現段階のシステム規模に対して過剰なため却下しました

## 影響

### ポジティブ

- 業務ルールを `domain` に寄せて保護しやすくなります
- モジュール境界ごとに変更影響を閉じ込めやすくなります
- unit / integration / API contract へ責務に沿ってテストを配置しやすくなります

### ネガティブ

- モジュール境界が曖昧なまま実装すると、モノリスの中で密結合が進むリスクがあります
- 一部の外部境界だけをポート化するため、どこまで抽象化するかの判断を継続的に行う必要があります

## コンプライアンス

- `apps/api` 配下が業務モジュールごとに分割されていること
- モジュール内部が 4 層で整理され、循環依存がないこと
- `domain` がインフラ詳細や HTTP 入出力に依存していないこと
