# ADR-001: アーキテクチャパターン選定（ドメインモデル + ポートとアダプター）

バックエンドのビジネスロジックパターンにドメインモデルパターン、アーキテクチャスタイルにポートとアダプター（ヘキサゴナルアーキテクチャ）を採用する。

日付: 2026-03-20

## ステータス

2026-03-20 承認済み

## コンテキスト

フレール・メモワール WEB ショップシステムのバックエンドアーキテクチャを選定する必要がある。

- **業務領域**: 受注管理・在庫管理は中核の業務領域であり、品質維持日数を考慮した在庫推移計算、受注状態遷移管理など複雑なビジネスルールが存在する
- **データ構造**: 10 以上のエンティティ、複数の状態遷移、在庫推移の日別計算が必要
- **チーム規模**: 1〜2 名のフルスタック開発者
- **テスト要件**: TDD を採用し、ビジネスロジックの高いテスタビリティが求められる

## 決定

**ビジネスロジックパターン**: ドメインモデルパターン

**アーキテクチャスタイル**: ポートとアダプター（ヘキサゴナルアーキテクチャ）

**テスト戦略**: ピラミッド形テスト（ユニット 70%、統合 20%、E2E 10%）

### 変更箇所

- バックエンドのパッケージ構造をドメイン層・アプリケーション層・インフラストラクチャ層・プレゼンテーション層に分割
- リポジトリインターフェース（ポート）をドメイン層に配置し、JPA 実装（アダプター）をインフラストラクチャ層に配置

### 代替案

| 代替案 | 却下理由 |
|:---|:---|
| トランザクションスクリプト | ビジネスロジックが複雑すぎる。在庫推移計算・状態遷移をスクリプトで管理すると保守性が急速に低下する |
| テーブルモジュール | ドメインの概念（花束構成、品質維持日数）がテーブル構造と 1:1 にならない |
| CQRS | 永続化モデルは RDB 1 つで十分。読み書き分離による複雑さは不要 |
| クリーンアーキテクチャ | ヘキサゴナルアーキテクチャと本質的に同じ。Spring Boot エコシステムとの相性でポートとアダプターを選択 |
| レイヤードアーキテクチャ | ドメイン層がインフラストラクチャ層に依存してしまい、テスタビリティが低下する |

## 影響

### ポジティブ

- ドメインロジックが外部技術（DB、Web フレームワーク）から独立し、ユニットテストが容易
- ビジネスルールの変更がドメイン層に局所化される
- 将来の技術変更（DB 変更、API フレームワーク変更）に対する柔軟性が高い
- TDD のインサイドアウトアプローチと自然に整合する

### ネガティブ

- トランザクションスクリプトと比較してコード量が増加する
- ドメインモデルと永続化モデルのマッピングが必要
- チームメンバーにドメイン駆動設計の知識が求められる

## コンプライアンス

- ドメイン層のクラスが `@Entity` 以外の Spring / JPA アノテーションに依存していないことをアーキテクチャテスト（ArchUnit）で検証する
- 依存方向が外側→内側（ドメイン層に向かう単方向）であることを CI で自動チェックする
- テストピラミッドの比率（ユニット 70%、統合 20%、E2E 10%）をカバレッジレポートで確認する
