# ADR-003: インフラストラクチャ選定（AWS ECS Fargate）

インフラストラクチャに AWS を採用し、ECS Fargate によるコンテナベースのモノリシックデプロイメントを行う。

日付: 2026-03-20

## ステータス

2026-03-20 承認済み

## コンテキスト

フレール・メモワール WEB ショップシステムのインフラストラクチャを選定する必要がある。

- **システム規模**: 小〜中規模（ユーザー数は限定的。個人顧客 + スタッフ数名）
- **チーム規模**: 1〜2 名のフルスタック開発者（専任のインフラエンジニアなし）
- **トラフィック**: 低〜中程度（花束注文のため急激なスパイクは想定しにくい）
- **可用性要件**: 中程度（注文を逃さない程度の可用性は必要）
- **コスト**: 個人経営の花屋のため、コスト効率を重視

## 決定

**クラウドプロバイダ**: AWS

**デプロイメントパターン**: コンテナベースのモノリシックデプロイメント（ECS Fargate）

**データベース**: Amazon RDS PostgreSQL（マルチ AZ）

**ネットワーク**: VPC + ALB（パブリック/プライベートサブネット分離）

**IaC**: Terraform

### 変更箇所

- フロントエンド・バックエンドを Docker コンテナ化し、ECS Fargate で実行
- ALB でパスベースルーティング（`/api/*` → バックエンド、その他 → フロントエンド）
- RDS PostgreSQL をプライベートサブネットに配置

### 代替案

| 代替案 | 却下理由 |
|:---|:---|
| マイクロサービス | 1〜2 名のチームで運用するには複雑すぎる。モノリスで十分な規模 |
| AWS Lambda（サーバーレス） | Spring Boot のコールドスタート問題。ドメインモデルパターンとの相性が悪い |
| EC2 | サーバー管理の運用負荷が高い。Fargate の方がインフラ管理が少ない |
| GCP / Azure | AWS のエコシステム（ECS、RDS、CloudWatch の統合）が最も成熟している |
| Kubernetes (EKS) | 小規模システムには運用複雑性が過剰。ECS Fargate で十分 |
| Heroku / Railway | カスタマイズ性と将来のスケーラビリティに制限がある |

## 影響

### ポジティブ

- Fargate によりサーバー管理不要。パッチ適用・スケーリングが自動化される
- Docker コンテナにより開発環境・ステージング・本番の一貫性を確保
- マルチ AZ 構成で可用性を確保しつつ、コスト最適化が可能
- Terraform による IaC で環境の再現性とバージョン管理を実現

### ネガティブ

- AWS ベンダーロックインのリスク（ただし、ヘキサゴナルアーキテクチャにより影響を最小化）
- Fargate は EC2 と比較してコスト単価が高い（ただし、管理コスト削減で相殺）
- ローカル開発環境と本番環境の差異が完全にはなくならない（Docker Compose vs ECS）

## コンプライアンス

- すべてのインフラリソースが Terraform で管理されていることを確認する（手動変更禁止）
- セキュリティグループが最小権限の原則に従っていることを定期レビューする
- ALB ヘルスチェック（`/actuator/health`）が設定されていることを確認する
- RDS の自動バックアップが有効であることを月次確認する
