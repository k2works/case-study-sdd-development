# ADR-003: 仕入先連携を partner API 境界に分離する

仕入先との連携は、顧客 / 管理者向け公開面と分離した partner API 境界で受けます。

日付: 2026-03-19

## ステータス

承認済み

## コンテキスト

- 仕入先は外部関係者であり、顧客向け画面や管理画面と同じ公開面に置くべきではない
- 発注、入荷、将来の仕入先連携は、認証方式、監査要件、障害切り分けが内部利用者と異なる
- インフラ構成レビューで、公開 `Load Balancer` 直結は信頼境界が弱いと指摘された
- 少人数運用でも、境界を先に切っておくことで将来の方式変更に耐えやすくなる

## 決定

仕入先との連携は `Partner Integration API` を通して受ける。顧客 / 管理者向け Web 経路とは別の境界として扱い、認証、監査、エラーハンドリングを分離する。

### 代替案

- 仕入先も公開 `Load Balancer` 配下の API を共用する
  - 却下理由: 顧客 / 管理者と同じ公開面になり、認証境界と監査が曖昧になる
- すべてを非同期ファイル連携に寄せる
  - 却下理由: 現時点では連携方式が確定しておらず、API 境界を切った方が将来の選択肢を残せる

## 影響

### ポジティブ

- 外部連携の信頼境界を明確にできる
- 認証、監査、障害対応を利用者向け機能と切り分けやすい
- 将来、非同期連携や専用認証方式へ移行しやすい

### ネガティブ

- API 境界が 1 つ増える
- MVP 時点では未使用の可能性があり、やや先行設計になる

## コンプライアンス

- 仕入先連携が顧客 / 管理画面と別経路で表現されていること
- 仕入先向け認証と監査ログの方針を後続の非機能要件 / 運用要件で定義すること
- 連携方式の具体化時に、別 ADR を追加して詳細を残すこと
