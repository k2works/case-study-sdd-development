# ADR-001: 発注作成時のトランザクション方針

## ステータス

承認済み

## コンテキスト

IT3 で発注機能（S09）を実装する。発注作成時に以下の操作が発生する：

1. PurchaseOrder エンティティの永続化
2. （将来）入荷時の StockLot 作成

IT2 の受注機能では、注文保存と在庫引当を 3 フェーズに分離した（準備→保存→引当保存）。発注機能でも同様のトランザクション設計方針を事前に決定する必要がある。

## 決定

**発注作成は単一トランザクションで実行する。**

### 理由

1. **操作の単純性**: 発注作成は PurchaseOrder の INSERT のみ。StockLot への影響は入荷受入時（IT4・S10）に発生するため、IT3 時点ではクロス集約のトランザクションは不要
2. **IT2 との一貫性**: 受注機能と同様に、集約単位でトランザクション境界を設計する方針を踏襲
3. **YAGNI**: 入荷受入機能（IT4）で StockLot 更新が必要になった時点でトランザクション設計を拡張する

### トランザクション境界

```
発注作成:
  BEGIN
    INSERT purchase_orders (単一テーブル操作)
  COMMIT
```

### 将来の入荷受入（IT4 で検討）

```
入荷受入:
  BEGIN
    UPDATE purchase_orders SET status = '入荷済み'
    INSERT stocks (新規 StockLot 作成)
  COMMIT
```

入荷受入では PurchaseOrder と StockLot の 2 集約を更新するため、IT4 で改めてトランザクション方針を ADR として記録する。

## 影響

- PurchaseOrderUseCase は単一リポジトリ呼び出しのみで完結するため、実装がシンプルになる
- Prisma の `$transaction` は IT3 では不要
- IT4 の入荷受入機能で、クロス集約トランザクションの ADR を追加作成する

## 関連

- IT2 ふりかえり T7: トランザクション設計を事前に ADR 化
- IT2 受注機能: 3 フェーズ分離（準備→保存→引当保存）
