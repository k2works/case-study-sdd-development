# ADR-002: Heroku PaaS の採用

## ステータス

承認

## コンテキスト

インフラ環境の選定が必要。開発者 1-2 名でインフラ専任がいない。運用負荷を最小化しつつ、頻繁なデプロイが可能な環境が求められる。

## 検討した選択肢

### A: Heroku PaaS

- 利点: git push デプロイ、マネージド DB、運用負荷最小
- 欠点: カスタマイズ性の制限、コストの透明性

### B: AWS ECS Fargate

- 利点: 高いカスタマイズ性、スケーラビリティ
- 欠点: インフラ管理の負荷が高い、学習コスト

### C: Render

- 利点: Heroku 類似の簡易さ、コストパフォーマンス
- 欠点: Heroku ほどのエコシステムがない

## 決定

**A: Heroku PaaS** を採用する。

## 理由

1. **運用負荷の最小化**: マネージドサービスで DB・SSL・ログが自動管理される
2. **デプロイの簡易さ**: git push または `heroku pipelines:promote` でデプロイ完了
3. **Rails との親和性**: Rails アプリケーションのデプロイに最適化されている
4. **段階的スケーリング**: Dyno の追加で水平スケール可能
5. **将来の移行性**: 必要に応じて AWS ECS への移行も可能

## 影響

- ステージング・本番の 2 環境を Heroku Pipeline で管理
- DB は Heroku Postgres（Mini → Basic の段階的アップグレード）
- CI/CD は GitHub Actions → Heroku デプロイ
- 同時接続 50 超の場合は AWS 移行を検討
