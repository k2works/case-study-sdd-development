# ADR-001: 発注作成時のトランザクション方針

発注作成処理は単一集約の保存のみのため、明示的なトランザクション制御を導入せず Prisma のデフォルト（単一操作の暗黙的トランザクション）で運用する。

日付: 2026-03-18

## ステータス

承認済み

## コンテキスト

S09「単品を発注する」機能の実装にあたり、発注作成時のトランザクション境界を明確にする必要がある。

### 現在の処理フロー

1. `ItemRepository.findById()` で単品情報を取得（読み取り）
2. `PurchaseOrder.createNew()` でドメインオブジェクトを生成（インメモリ）
3. `PurchaseOrderRepository.save()` で発注レコードを永続化（書き込み）

### 考慮すべき点

- 現在の発注作成は **単一テーブルへの INSERT のみ** で完結する
- 単品情報の読み取り（ステップ 1）と発注の書き込み（ステップ 3）の間に、他のトランザクションによる単品マスタの変更は業務上ほぼ発生しない（マスタメンテナンスは別タイミング）
- 将来的に「発注作成と同時に在庫引当を行う」「発注と連動して仕入先への通知を送る」などの要件が追加される可能性がある
- プロジェクトのコードベースでは現時点で `prisma.$transaction` は未使用

## 決定

**発注作成処理において、Prisma のデフォルトの暗黙的トランザクション（単一操作ごとの自動コミット）を使用する。明示的な `prisma.$transaction` は導入しない。**

### 理由

1. **単一集約の保存**: 現在の処理は `purchaseOrder` テーブルへの `INSERT` 1 件のみであり、複数テーブルへの書き込みが存在しない
2. **Prisma のデフォルト保証**: `prisma.purchaseOrder.create()` は単一操作として原子的に実行される
3. **YAGNI 原則**: 必要になるまで複雑さを導入しない。将来の要件変更時に明示的トランザクションを追加すればよい

### 将来のトランザクション導入基準

以下のいずれかの条件を満たした場合、明示的トランザクション（`prisma.$transaction`）を導入する：

| 条件 | 例 |
|------|-----|
| 複数集約の同時更新 | 発注作成 + 在庫引当 |
| 楽観的ロック | 同一単品への同時発注の排他制御 |
| Saga パターン | 外部システム（仕入先 API）との連携 |

### 代替案

#### A: 即座に `prisma.$transaction` を導入する

- **却下理由**: 現時点では単一テーブルへの INSERT のみのため過剰。コードの複雑さが増し、テストも煩雑になる。Repository インターフェースにトランザクションコンテキストを渡す設計変更が必要になり、影響範囲が大きい

#### B: UseCase 層でトランザクションを管理する（Unit of Work パターン）

- **却下理由**: MVP フェーズにおいて過剰な抽象化。現在の Repository パターンとの整合性を保ちつつ導入するには、全 Repository のインターフェース変更が必要。複数集約の同時更新が実際に必要になった時点で検討する

## 影響

### ポジティブ

- コードがシンプルに保たれる（UseCase → Repository の直線的なフロー）
- テストが容易（トランザクション管理のモックが不要）
- 既存の Repository インターフェースを変更せずに済む

### ネガティブ

- 将来、複数集約の同時更新が必要になった場合、トランザクション境界の設計と Repository インターフェースの変更が必要
- 単品情報の読み取りと発注の書き込みの間に理論上のレースコンディションが存在する（業務上のリスクは極めて低い）

## コンプライアンス

- `PurchaseOrderUseCase.createPurchaseOrder()` で `prisma.$transaction` が使われていないことを確認
- `PrismaPurchaseOrderRepository.save()` が単一の `prisma.purchaseOrder.create()` で完結していることを確認
- 将来の変更時に本 ADR を参照し、トランザクション導入基準に該当するか判断する
