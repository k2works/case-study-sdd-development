# ADR-003: 顧客向けとスタッフ向けを単一 Next.js アプリで構成する

顧客向けとスタッフ向けは、単一の Next.js アプリケーション内でルート分割して構成します。

日付: 2026-03-20

## ステータス

承認済み

## コンテキスト

本システムには、顧客向けの短い注文導線と、スタッフ向けの一覧中心業務画面という性質の異なる UI が共存します。

- 顧客向けは初期表示と SEO を重視したいです
- スタッフ向けはフィルタ、再読込、差分確認など SPA 的操作性を重視します
- チーム規模が小さいため、別フロントエンド化による認証、CI/CD、デザインシステムの二重管理は避けたいです

## 決定

**`apps/web` に単一の Next.js アプリを置き、`(shop)` と `(staff)` のルート群で責務を分けます。顧客向けは SSR / ISR を活用し、スタッフ向けは Client 中心の操作性を優先します。専用 BFF は初期段階では導入しません。**

### 変更箇所

- 顧客向けとスタッフ向けを単一アプリ内のルート群で分離する
- API 連携は REST API を前提にし、`lib/api/` に集約する
- 認証情報を中継する必要がある場合のみ、Next.js の Route Handler / Server Component を same-origin の薄い中継層として使う

### 代替案

- 純粋な SPA
  - スタッフ画面には向くが、顧客向け導線の初期表示や SEO 対応の追加コストが高いため却下しました
- 顧客用 / スタッフ用の別アプリ
  - 小規模チームに対して運用対象が増えすぎるため却下しました
- 初期から専用 BFF を導入
  - 認証や API 集約の複雑性がまだ不足しており、現時点では過剰なため却下しました

## 影響

### ポジティブ

- 顧客向けとスタッフ向けの技術基盤を統一できます
- デザインシステム、テスト基盤、CI/CD を共通化できます
- 顧客向け SEO とスタッフ向け操作性を同じ技術スタックで両立できます

### ネガティブ

- ルート設計と Feature 分割が甘いと、顧客向けとスタッフ向けの責務が混ざりやすくなります
- 認証連携が複雑化した場合、後から BFF を導入する判断が必要になります

## コンプライアンス

- `apps/web` が単一アプリであり、`(shop)` と `(staff)` で責務分離されていること
- 共通 UI は `components/`、業務知識は `features/` に置かれていること
- API DTO をそのまま UI に露出せず、ViewModel へ変換していること
