# システムユースケース - フレール・メモワール WEB ショップシステム

## UC-001: 🖥️花束を注文する🌊

**コンテキスト**: 得意先が大切な記念日に花束を届けるため、WEB ショップから注文する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 得意先

**利害関係者と利益**:

- 得意先: 確実に指定日に花束が届くこと
- フレール・メモワール（経営者）: 受注が正確に記録され、在庫引当が行われること
- 仕入スタッフ: 在庫推移に受注が即時反映されること

**事前条件**:

- 商品（花束）がマスタに登録されている
- 得意先のクレジットカードが事前登録済み

**トリガー**: 得意先が WEB ショップにアクセスし商品を選択する

**最低保証**: 受注データが不整合な状態で保存されない

**成功時保証**: 受注が登録され、在庫引当が行われ、注文確認メールが送信される

**主成功シナリオ**:

1. 得意先が商品（花束）を選択する
2. システムが商品の詳細（構成花材・価格）を表示する
3. 得意先が届け日を選択する
4. システムが選択可能な届け日範囲（注文日 + リードタイム〜注文日 + 30 日）を提示する
5. 得意先が届け先情報（氏名・住所・電話番号）を入力する
6. 得意先がメッセージカードの内容を入力する
7. システムが注文内容の確認画面を表示する
8. 得意先が注文を確定する
9. システムが在庫引当を行い、受注を登録し、注文確認メールを送信する

**拡張**:

- 3a. 選択した届け日の在庫が不足している:
    - 3a1. システムが在庫不足を通知し、代替の届け日を提案する
    - 3a2. 得意先が別の届け日を選択する（ステップ 3 へ）
- 5a. 過去の届け先を利用する:
    - 5a1. 得意先が届け先コピー機能で過去の届け先を選択する（UC-002 を参照）
- 8a. 得意先が注文を取りやめる:
    - 8a1. システムが仮引当を解除し、処理を終了する

## UC-002: 🖥️届け先をコピーして再注文する🌊

**コンテキスト**: リピーターの得意先が、過去と同じ届け先に手軽に花束を再注文する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 得意先（リピーター）

**利害関係者と利益**:

- 得意先: 住所の再入力が不要で手軽に注文できること
- フレール・メモワール: リピート率の向上

**事前条件**:

- 得意先に過去の注文履歴がある

**トリガー**: 得意先が再注文を開始する

**最低保証**: 過去の届け先データが破損しない

**成功時保証**: 過去の届け先がコピーされ、新しい注文が登録される

**主成功シナリオ**:

1. 得意先が注文入力画面で「過去の届け先を利用」を選択する
2. システムが過去の届け先一覧を表示する
3. 得意先が使用する届け先を選択する
4. システムが届け先情報を注文入力画面にコピーする
5. 得意先が必要に応じて届け先情報を修正する
6. 以降は UC-001 のステップ 6 に合流する

**拡張**:

- 2a. 過去の届け先が存在しない:
    - 2a1. システムが「過去の届け先がありません」と表示する
    - 2a2. 得意先が新しい届け先を手入力する

## UC-003: 🖥️届け日を変更する🌊

**コンテキスト**: 得意先が注文後に届け日の変更を希望する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 得意先

**支援アクター**: 受注スタッフ

**利害関係者と利益**:

- 得意先: 届け日を柔軟に変更できること
- 仕入スタッフ: 在庫引当が正しく再計算されること

**事前条件**:

- 受注が「受付済み」状態である
- 変更期限（届け日の 3 日前）内である

**トリガー**: 得意先が届け日変更を依頼する

**最低保証**: 変更処理中に在庫データが不整合にならない

**成功時保証**: 届け日が変更され、在庫引当が再計算される

**主成功シナリオ**:

1. 得意先が届け日変更を依頼する
2. システムが変更期限を確認する
3. システムが新しい届け日の在庫状況を確認する
4. システムが現在の在庫引当を解除し、新しい届け日で再引当する
5. システムが届け日変更を確定し、変更完了を通知する

**拡張**:

- 2a. 変更期限を超過している:
    - 2a1. システムが「変更期限を過ぎています」と通知する
    - 2a2. 終了
- 3a. 新しい届け日の在庫が不足している:
    - 3a1. システムが在庫不足を通知し、変更可能な代替日を提案する
    - 3a2. 得意先が代替日を選択するか、変更を取りやめる

---

## UC-004: 🖥️注文をキャンセルする🌊

**コンテキスト**: 得意先が注文のキャンセルを希望する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 得意先

**事前条件**:

- 受注が「受付済み」状態である
- キャンセル期限（届け日の 3 日前）内である

**トリガー**: 得意先がキャンセルを依頼する

**最低保証**: キャンセル処理中に在庫データが不整合にならない

**成功時保証**: 受注がキャンセル済みとなり、在庫引当が解除される

**主成功シナリオ**:

1. 得意先がキャンセルを依頼する
2. システムがキャンセル期限を確認する
3. システムが在庫引当を解除する
4. システムが受注ステータスを「キャンセル済み」に更新する
5. システムがキャンセル完了を通知する

**拡張**:

- 2a. キャンセル期限を超過している:
    - 2a1. システムが「キャンセル期限を過ぎています」と通知する
    - 2a2. 終了

## UC-005: 🖥️受注状況を確認する🌊

**コンテキスト**: 受注スタッフが受注の一覧とステータスを管理画面で確認する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 受注スタッフ

**事前条件**: なし

**トリガー**: 受注スタッフが管理画面を開く

**最低保証**: データの表示が正確である

**成功時保証**: 受注状況が把握でき、対応が必要な受注を識別できる

**主成功シナリオ**:

1. 受注スタッフが受注管理画面を開く
2. システムが受注一覧を表示する
3. 受注スタッフがステータスや日付で絞り込む
4. システムが条件に合致する受注を表示する
5. 受注スタッフが個別の受注詳細を確認する

## UC-006: 🖥️出荷処理を行う🌊

**コンテキスト**: 出荷日に結束済みの花束を出荷する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 受注スタッフ

**事前条件**:

- 受注が「出荷準備中」状態で結束が完了している

**トリガー**: 出荷日（= 届け日の前日）になる

**最低保証**: 出荷データが不整合な状態で保存されない

**成功時保証**: 出荷が記録され、受注ステータスが「出荷済み」になる

**主成功シナリオ**:

1. 受注スタッフが出荷管理画面で出荷対象を確認する
2. 受注スタッフが出荷処理を実行する
3. システムが出荷情報を記録し、受注ステータスを「出荷済み」に更新する
4. システムが得意先に出荷通知メールを送信する

---

## UC-007: 🖥️在庫推移を確認する🌊

**コンテキスト**: 仕入スタッフが毎朝の発注判断のために在庫推移を確認する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 仕入スタッフ

**事前条件**:

- 単品マスタが登録されている

**トリガー**: 仕入スタッフが管理画面を開く（通常は毎朝）

**最低保証**: 在庫データの表示が正確である

**成功時保証**: 在庫推移が可視化され、発注判断の材料が得られる

**主成功シナリオ**:

1. 仕入スタッフが在庫推移画面を開く
2. システムが単品ごとの日別在庫推移を表示する（在庫残・出庫予定・入荷予定・廃棄予定）
3. 仕入スタッフが品質維持期限アラート（期限まで残り 2 日以内）を確認する
4. 仕入スタッフが発注が必要な単品を特定する

**拡張**:

- 3a. 品質維持期限が間近の在庫がある:
    - 3a1. システムがアラートを表示する
    - 3a2. 仕入スタッフが優先的に活用する判断を行う

---

## UC-008: 🖥️仕入先に発注する🌊

**コンテキスト**: 仕入スタッフが在庫推移を確認した結果、不足する花材を仕入先に発注する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 仕入スタッフ

**事前条件**:

- 発注が必要な単品が特定されている
- 仕入先がマスタに登録されている

**トリガー**: 仕入スタッフが発注を決定する

**最低保証**: 発注データが不整合な状態で保存されない

**成功時保証**: 発注が登録され、入荷予定として在庫推移に反映される

**主成功シナリオ**:

1. 仕入スタッフが発注登録画面を開く
2. 仕入スタッフが単品・数量・入荷予定日を入力する
3. システムが仕入先情報を自動設定する（単品ごとに仕入先が決まっている）
4. 仕入スタッフが発注を確定する
5. システムが発注を登録し、入荷予定を在庫推移に反映する

---

## UC-009: 🖥️入荷を受け入れる🌊

**コンテキスト**: 仕入先から花材が届き、在庫に登録する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 仕入スタッフ

**事前条件**:

- 発注済みの注文がある

**トリガー**: 仕入先から単品が届く

**最低保証**: 在庫データが不整合な状態で保存されない

**成功時保証**: 入荷が記録され、在庫が増加し、在庫推移が更新される

**主成功シナリオ**:

1. 仕入スタッフが入荷登録画面を開く
2. 仕入スタッフが発注番号を選択する
3. 仕入スタッフが入荷数量を入力する
4. システムが入荷日（= 本日）を品質維持日数の起算日として設定する
5. システムが在庫ロットを作成し、在庫を更新する

**拡張**:

- 3a. 入荷数量が発注数量と異なる:
    - 3a1. システムが差異を表示する
    - 3a2. 仕入スタッフが実際の入荷数量で登録する

---

## UC-010: 🖥️商品マスタを管理する🌊

**コンテキスト**: 受注スタッフが商品と単品のマスタデータを管理する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 受注スタッフ

**事前条件**: なし

**トリガー**: 新商品の追加やマスタ情報の変更が必要になる

**最低保証**: マスタデータが不整合な状態で保存されない

**成功時保証**: 商品・単品のマスタデータが登録・更新される

**主成功シナリオ**:

1. 受注スタッフが商品管理画面を開く
2. 受注スタッフが商品（花束）の情報を登録・更新する
3. 受注スタッフが単品マスタ（品質維持日数・購入単位・リードタイム・仕入先）を管理する
4. 受注スタッフが花束構成（花束を構成する単品と数量）を定義する
5. システムがマスタデータを保存する

---

## UC-011: 🖥️結束対象を確認する🌊

**コンテキスト**: フローリストが出荷日当日に結束すべき花束と必要な花材を確認する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: フローリスト

**事前条件**:

- 出荷日の受注が存在する

**トリガー**: 出荷日当日の朝

**最低保証**: データの表示が正確である

**成功時保証**: 結束すべき花束と必要な花材が把握できる

**主成功シナリオ**:

1. フローリストが結束一覧画面を開く
2. システムが出荷日（= 本日）の結束対象一覧を表示する
3. フローリストが各花束に必要な花材と数量を確認する
4. フローリストが結束作業を行う
