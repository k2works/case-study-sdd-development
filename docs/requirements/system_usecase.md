## UC01: 🖥️ 花束を注文する 🌊

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 得意先

**利害関係者と利益**:

- 得意先: 新鮮な花束を大切な記念日に届けたい
- フレール・メモワール: 受注を正確に記録し、確実に届けたい

**事前条件**:

- 得意先がクレジットカードを事前登録済みである
- 商品（花束）が WEB ショップに登録されている

**トリガー**: 得意先が WEB ショップで花束を選択する

**最低保証**: 注文が中断されてもデータの不整合が発生しない

**成功時保証**: 受注が登録され、届け日に向けた在庫引当が行われる

**主成功シナリオ**:

1. 得意先が商品一覧から花束を選択する
2. システムが商品の詳細と構成を表示する
3. 得意先が届け日を指定する
4. 得意先が届け先情報（住所・電話番号）を入力する
5. 得意先がお届けメッセージを入力する
6. システムが注文内容の確認画面を表示する
7. 得意先が注文を確定する
8. システムが受注を登録し、在庫を引き当てる

**拡張**:

- 4a. 過去の届け先を再利用したい:
    - 4a1. 得意先が届け先コピー機能を使用する（UC02）
- 3a. 指定した届け日に出荷不可:
    - 3a1. システムが出荷可能な日付を提示する
    - 3a2. 得意先が別の届け日を選択する
- 7a. 得意先が注文をキャンセル:
    - 7a1. システムが注文データを破棄する

## UC02: 🖥️ 届け先をコピーする 🐟

**スコープ**: WEB ショップシステム

**レベル**: サブ機能

**主アクター**: 得意先

**事前条件**:

- 得意先に過去の注文履歴がある

**主成功シナリオ**:

1. システムが過去の届け先一覧を表示する
2. 得意先が再利用する届け先を選択する
3. システムが届け先情報を注文画面にコピーする

## UC03: 🖥️ 届け日を変更する 🌊

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 得意先

**利害関係者と利益**:

- 得意先: 予定変更に柔軟に対応してほしい
- フレール・メモワール: 顧客満足を維持しつつ、出荷業務に支障をきたさない

**事前条件**:

- 対象の受注が「注文済み」状態である

**トリガー**: 得意先が届け日の変更を依頼する

**最低保証**: 変更不可の場合、得意先に迅速に通知される

**成功時保証**: 届け日が変更され、在庫引当が再計算される

**主成功シナリオ**:

1. 得意先が変更後の届け日を指定する
2. システムが変更後の届け日で出荷可否を判定する
3. システムが届け日を変更し、在庫引当を再計算する
4. システムが得意先に変更完了を通知する

**拡張**:

- 2a. 変更後の届け日で出荷不可:
    - 2a1. システムが変更不可を得意先に通知する

## UC04: 🖥️ 受注を確認する 🌊

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 受注スタッフ

**主成功シナリオ**:

1. 受注スタッフが受注一覧画面を表示する
2. システムが受注一覧（注文済み・出荷準備中・出荷済み）を表示する
3. 受注スタッフが特定の受注の詳細を確認する
4. システムが受注詳細（商品・届け日・届け先・メッセージ）を表示する

## UC05: 🖥️ 在庫推移を確認する 🌊

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 仕入スタッフ

**利害関係者と利益**:

- 仕入スタッフ: 適切な発注判断を行いたい
- フレール・メモワール: 廃棄ロスを最小化したい

**主成功シナリオ**:

1. 仕入スタッフが在庫推移画面を表示する
2. システムが単品ごとの日別在庫予定数を表示する
3. 仕入スタッフが品質維持日数を考慮した在庫状況を確認する

**技術およびデータのバリエーション**:

- ステップ 2: 在庫予定数 = 現在庫 + 入荷予定 - 受注引当 - 品質維持日数超過分

## UC06: 🖥️ 単品を発注する 🌊

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 仕入スタッフ

**事前条件**:

- 単品マスタに仕入先・購入単位・リードタイムが登録されている

**主成功シナリオ**:

1. 仕入スタッフが発注する単品を選択する
2. システムが単品の仕入先・購入単位・リードタイムを表示する
3. 仕入スタッフが発注数量を指定する
4. システムが購入単位に基づく数量を確認する
5. 仕入スタッフが発注を確定する
6. システムが発注を記録し、入荷予定日を計算する

**拡張**:

- 3a. 数量が購入単位の倍数でない:
    - 3a1. システムが購入単位の倍数に切り上げた数量を提示する
