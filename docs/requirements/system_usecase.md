## UC-01: 🖥️商品を注文する🌊

**使用時のコンテキスト**: 得意先が記念日向けの花束を WEB ショップから注文する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 得意先

**利害関係者と利益**:

- 得意先: 記念日に間に合う注文を簡単に完了したい。
- 受注スタッフ: 受注漏れや転記ミスなく注文を受け取りたい。
- フレール・メモワール: 受注処理を効率化し、リピーター体験を改善したい。

**事前条件**:

- 販売可能な商品が登録されている。
- 得意先が注文画面を利用できる。

**最低保証**:

- 入力途中で失敗しても既存受注は壊れない。

**成功保証時**:

- 受注が登録される。
- 出荷準備の対象として参照できる。

**トリガー**: 得意先が注文画面で商品を選択し、注文確定を開始する。

**主成功シナリオ**:

1. 得意先が商品を選択する。
2. システムが商品内容と入力フォームを表示する。
3. 得意先が届け日、届け先、メッセージを入力する。
4. システムが入力内容を検証し、確認画面を表示する。
5. 得意先が注文を確定する。
6. システムが受注を登録する。
7. システムが注文完了画面を表示する。

**拡張**:

- 3a. 入力必須項目が不足している:
  - 3a1. システムが不足項目を表示し、再入力を促す。
- 4a. 指定した届け日が受け付け対象外である:
  - 4a1. システムが受け付け可能な届け日を案内する。
- 5a. 得意先が確定前に入力を修正したい:
  - 5a1. システムが入力画面へ戻す。

**技術およびデータのバリエーションリスト**:

- 3. 届け先は新規入力または過去履歴からの選択で指定できる。
- 6. 受注は 1 受注 1 商品 1 届け先で登録する。

**関連情報**:

- 対応 BUC: BUC-01
- 対応ストーリー: US-01、US-02

## UC-02: 🖥️受注を確認する🌊

**使用時のコンテキスト**: 受注スタッフが登録済み受注を一覧と詳細で確認する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 受注スタッフ

**利害関係者と利益**:

- 受注スタッフ: 受注状況を見失わず、顧客問い合わせに対応したい。
- フローリスト: 出荷準備対象を正確に把握したい。

**事前条件**:

- 受注が 1 件以上登録されている。

**最低保証**:

- 受注データの内容は参照のみで変更されない。

**成功保証時**:

- 対象受注の一覧または詳細が確認できる。

**トリガー**: 受注スタッフが受注管理画面を開く。

**主成功シナリオ**:

1. 受注スタッフが受注管理画面を開く。
2. システムが受注一覧を表示する。
3. 受注スタッフが対象受注を選択する。
4. システムが受注詳細を表示する。

**拡張**:

- 2a. 対象受注が存在しない:
  - 2a1. システムが受注なしを表示する。

**技術およびデータのバリエーションリスト**:

- 2. 出荷日、届け日、顧客名などで絞り込みできる。

**関連情報**:

- 対応 BUC: BUC-01
- 対応ストーリー: US-03

## UC-03: 🖥️届け日を変更する🌊

**使用時のコンテキスト**: 得意先からの依頼にもとづき、受注スタッフが届け日変更を処理する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 受注スタッフ

**利害関係者と利益**:

- 得意先: 可能な範囲で柔軟に届け日を変えたい。
- 受注スタッフ: 在庫や出荷準備状況に反した変更を避けたい。

**事前条件**:

- 対象受注が存在する。
- 受注が出荷済みではない。

**最低保証**:

- 変更不可でも元の受注内容が維持される。

**成功保証時**:

- 変更可否が判断され、可能な場合は新しい届け日が保存される。

**トリガー**: 受注スタッフが変更依頼を受け付ける。

**主成功シナリオ**:

1. 受注スタッフが対象受注を選択する。
2. システムが現在の届け日と関連情報を表示する。
3. 受注スタッフが新しい届け日を入力する。
4. システムが在庫推移と出荷条件を確認する。
5. システムが変更可能と判断する。
6. システムが届け日を更新する。
7. システムが変更結果を表示する。

**拡張**:

- 4a. 出荷準備済みである:
  - 4a1. システムが変更不可を表示する。
- 4b. 新しい届け日に在庫不足が発生する:
  - 4b1. システムが変更不可を表示する。

**技術およびデータのバリエーションリスト**:

- 4. 判定には在庫予定、出荷状態、品質維持期限を用いる。

**関連情報**:

- 対応 BUC: BUC-02
- 対応ストーリー: US-08

## UC-04: 🖥️届け先をコピーする🌊

**使用時のコンテキスト**: リピーターが過去の届け先を新しい注文入力へ反映する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 得意先

**利害関係者と利益**:

- 得意先: 再注文時の入力を減らしたい。
- フレール・メモワール: 注文完了率を高めたい。

**事前条件**:

- 得意先に届け先履歴が存在する。

**最低保証**:

- 履歴が選べなくても新規入力に戻れる。

**成功保証時**:

- 選択した届け先情報が注文入力へ反映される。

**トリガー**: 得意先が届け先再利用を選ぶ。

**主成功シナリオ**:

1. 得意先が届け先選択画面を開く。
2. システムが過去の届け先一覧を表示する。
3. 得意先が 1 件を選択する。
4. システムが注文入力へ届け先情報を反映する。

**拡張**:

- 2a. 履歴が存在しない:
  - 2a1. システムが新規入力を案内する。

**技術およびデータのバリエーションリスト**:

- 2. 過去の届け先は顧客単位で保持する。

**関連情報**:

- 対応 BUC: BUC-03
- 対応ストーリー: US-07

## UC-05: 🖥️在庫推移を確認する🌊

**使用時のコンテキスト**: 仕入スタッフが将来の在庫状態を日別に確認する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 仕入スタッフ

**利害関係者と利益**:

- 仕入スタッフ: 発注判断を人間として適切に行いたい。
- 経営者: 廃棄ロスを減らしたい。

**事前条件**:

- 受注、商品構成、入荷予定、単品情報が登録されている。

**最低保証**:

- 算出失敗時でも元データは保持される。

**成功保証時**:

- 日別在庫予定数と注意対象が表示される。

**トリガー**: 仕入スタッフが在庫推移画面を開く。

**主成功シナリオ**:

1. 仕入スタッフが確認対象期間を指定する。
2. システムが受注、入荷予定、品質維持日数をもとに在庫推移を算出する。
3. システムが日別在庫予定数を表示する。
4. システムが不足見込みと廃棄注意を識別して表示する。

**拡張**:

- 2a. 必要な単品マスタが不足している:
  - 2a1. システムが算出不能な対象を表示する。

**技術およびデータのバリエーションリスト**:

- 2. 算出は単品単位で行う。
- 4. 表示は単品別、日付別で切り替えできる。

**関連情報**:

- 対応 BUC: BUC-04
- 対応ストーリー: US-04

## UC-06: 🖥️発注する🌊

**使用時のコンテキスト**: 仕入スタッフが在庫推移を見て発注内容を確定する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 仕入スタッフ

**利害関係者と利益**:

- 仕入スタッフ: 仕入先別に正しい数量を発注したい。
- 仕入先: 明確な発注内容を受け取りたい。

**事前条件**:

- 発注候補が確認できている。

**最低保証**:

- 送信に失敗しても発注内容を再確認できる。

**成功保証時**:

- 発注が作成・送信される。

**トリガー**: 仕入スタッフが発注画面から送信を実行する。

**主成功シナリオ**:

1. 仕入スタッフが発注対象を選択する。
2. システムが仕入先別の発注案を表示する。
3. 仕入スタッフが数量を確認し、発注を確定する。
4. システムが発注を保存する。
5. システムが仕入先へ発注内容を送信する。

**拡張**:

- 5a. 外部連絡に失敗する:
  - 5a1. システムが未送信として保持し、再送を可能にする。

**技術およびデータのバリエーションリスト**:

- 2. 購入単位に合わせて数量を丸める。

**関連情報**:

- 対応 BUC: BUC-05
- 対応ストーリー: US-05

## UC-07: 🖥️入荷を記録する🌊

**使用時のコンテキスト**: 仕入スタッフが納品結果を登録する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 仕入スタッフ

**利害関係者と利益**:

- 仕入スタッフ: 実績と予定の差分を把握したい。
- フローリスト: 利用可能な花材を正確に把握したい。

**事前条件**:

- 対象発注が存在する。

**最低保証**:

- 入荷未完了でも途中状態を保持できる。

**成功保証時**:

- 入荷実績が登録され、在庫予定が更新される。

**トリガー**: 仕入スタッフが入荷登録を開始する。

**主成功シナリオ**:

1. 仕入スタッフが対象発注を選択する。
2. システムが発注内容を表示する。
3. 仕入スタッフが入荷数量を入力する。
4. システムが入荷実績を保存する。
5. システムが在庫予定を更新する。

**拡張**:

- 3a. 一部入荷である:
  - 3a1. システムが発注状態を一部入荷として保持する。

**技術およびデータのバリエーションリスト**:

- 3. 予定数量との差分を記録する。

**関連情報**:

- 対応 BUC: BUC-06
- 対応ストーリー: US-06

## UC-08: 🖥️出荷対象を確認する🌊

**使用時のコンテキスト**: フローリストが出荷前日に当日作業対象を確認する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: フローリスト

**利害関係者と利益**:

- フローリスト: 必要花材と結束対象を漏れなく把握したい。
- 受注スタッフ: 出荷漏れを防ぎたい。

**事前条件**:

- 出荷日の前日までに受注が登録されている。

**最低保証**:

- 参照失敗時でも受注データは壊れない。

**成功保証時**:

- 出荷対象一覧と必要花材が表示される。

**トリガー**: フローリストが出荷一覧画面を開く。

**主成功シナリオ**:

1. フローリストが出荷日を指定する。
2. システムが対象受注を一覧表示する。
3. システムが各受注に必要な花材を表示する。

**拡張**:

- 2a. 対象受注が存在しない:
  - 2a1. システムが対象なしを表示する。

**技術およびデータのバリエーションリスト**:

- 3. 必要花材は花束構成にもとづいて算出する。

**関連情報**:

- 対応 BUC: BUC-07
- 対応ストーリー: US-09

## UC-09: 🖥️出荷を確定する🌊

**使用時のコンテキスト**: 受注スタッフが花束結束後の出荷実績を確定する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 受注スタッフ

**利害関係者と利益**:

- 受注スタッフ: 出荷状態を正確に管理したい。
- 得意先: 予定どおり出荷されたことを前提にサービスを受けたい。

**事前条件**:

- 出荷対象が確認済みである。

**最低保証**:

- 出荷未確定の状態を維持できる。

**成功保証時**:

- 出荷実績が保存される。
- 対象受注が出荷済みになる。

**トリガー**: 受注スタッフが出荷確定を実行する。

**主成功シナリオ**:

1. 受注スタッフが対象出荷を選択する。
2. システムが対象内容を表示する。
3. 受注スタッフが出荷確定を実行する。
4. システムが出荷実績を保存する。
5. システムが受注状態を出荷済みに更新する。

**拡張**:

- 3a. 必要花材不足で保留中である:
  - 3a1. システムが確定不可を表示する。

**技術およびデータのバリエーションリスト**:

- 4. 出荷実績には出荷日と担当者を保持する。

**関連情報**:

- 対応 BUC: BUC-08
- 対応ストーリー: US-10
