# システムユースケース - フレール・メモワール WEB ショップシステム

## UC-001: 🖥️商品を注文する🌊

**使用時のコンテキスト**: 得意先が WEB ショップから花束を選択し、届け日・届け先・メッセージを指定して注文する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 得意先

**利害関係者と利益**:

- 得意先: 手軽に花束を注文し、大切な記念日に届けたい
- 経営者: 受注件数を増加させ、売上を拡大したい
- 受注スタッフ: 正確な注文情報を受け取りたい

**事前条件**:

- 得意先が WEB ショップにアクセスできる
- 商品（花束）がシステムに登録されている

**最低保証**: 不完全な注文はシステムに保存されない

**成功時保証**: 注文が「注文受付」ステータスで登録され、受注スタッフが確認できる状態になる

**トリガー**: 得意先が WEB ショップにアクセスし、商品を選択する

**主成功シナリオ**:

1. 得意先が商品一覧から花束を選択する
2. システムが商品の詳細（構成する花材、価格）を表示する
3. 得意先が届け日を指定する
4. 得意先が届け先（氏名、住所、電話番号）を入力する
5. 得意先がお届けメッセージを入力する
6. システムが注文内容の確認画面を表示する
7. 得意先が注文を確定する
8. システムが受注を「注文受付」ステータスで登録する

**拡張**:

- 4a. 過去の届け先を再利用する場合:
    - 4a1. 得意先が過去の届け先一覧から選択する（UC-011 届け先をコピーする）
- 3a. 指定した届け日に対応できる在庫がない場合:
    - 3a1. システムが対応可能な最寄りの届け日を提案する
    - 3a2. 得意先が代替の届け日を選択する
- 7a. 得意先が注文をキャンセルする場合:
    - 7a1. システムが注文を破棄する
    - 7a2. 終了

**技術およびデータのバリエーションリスト**:

- ステップ 4: 届け先情報は手入力またはコピー機能による自動入力

**関連情報**: BUC-01 に対応

## UC-002: 🖥️届け日を変更する🌊

**使用時のコンテキスト**: 注文済みの花束の届け日を変更したい場合に、在庫状況を確認して変更の可否を判断する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 受注スタッフ

**利害関係者と利益**:

- 得意先: 都合に合わせて届け日を変更したい
- 受注スタッフ: 変更要望に迅速に対応したい
- 仕入スタッフ: 変更による在庫への影響を把握したい

**事前条件**:

- 対象の受注が「注文受付」または「受付済み」ステータスである
- 受注が「出荷準備中」以降のステータスでない

**最低保証**: 変更に失敗した場合、元の届け日が維持される

**成功時保証**: 届け日が新しい日付に変更され、在庫推移に反映される

**トリガー**: 得意先から届け日変更の依頼を受ける

**主成功シナリオ**:

1. 受注スタッフが対象の受注を検索する
2. システムが受注詳細を表示する
3. 受注スタッフが新しい届け日を入力する
4. システムが新しい届け日の在庫推移を確認する
5. システムが変更可能であることを表示する
6. 受注スタッフが変更を確定する
7. システムが届け日を更新し、在庫推移を再計算する

**拡張**:

- 4a. 新しい届け日で在庫が確保できない場合:
    - 4a1. システムが在庫不足の単品と対応可能な代替日を表示する
    - 4a2. 受注スタッフが得意先に変更不可を通知する
    - 4a3. 終了

**関連情報**: BUC-02 に対応

## UC-003: 🖥️受注を確認・管理する🌊

**使用時のコンテキスト**: 受注スタッフが日々の受注状況を確認し、ステータスを管理する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 受注スタッフ

**利害関係者と利益**:

- 受注スタッフ: 受注状況を一覧で把握し、漏れなく処理したい
- 経営者: 受注処理の効率化による生産性向上

**事前条件**:

- 受注スタッフがシステムにログインしている

**最低保証**: 受注データの整合性が維持される

**成功時保証**: 対象の受注ステータスが正しく更新される

**トリガー**: 受注スタッフが受注一覧画面にアクセスする

**主成功シナリオ**:

1. 受注スタッフが受注一覧画面を表示する
2. システムが受注一覧をステータス別に表示する
3. 受注スタッフが対象の受注を選択する
4. システムが受注詳細（商品、届け日、届け先、メッセージ）を表示する
5. 受注スタッフが受注を「受付済み」に更新する
6. システムがステータスを更新する

**拡張**:

- 2a. 絞り込み検索する場合:
    - 2a1. 受注スタッフが期間・ステータス等の条件を指定する
    - 2a2. システムが条件に合う受注を表示する

**関連情報**: BUC-03 に対応

## UC-004: 🖥️在庫推移を確認する🌊

**使用時のコンテキスト**: 仕入スタッフが品質維持日数を考慮した日別の在庫予定数を確認し、発注判断の材料にする。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 仕入スタッフ

**利害関係者と利益**:

- 仕入スタッフ: 在庫状況を正確に把握し、適切な発注判断をしたい
- 経営者: 廃棄ロスを最小化し、利益率を改善したい

**事前条件**:

- 仕入スタッフがシステムにログインしている
- 単品マスタに品質維持日数が登録されている

**最低保証**: なし（参照のみの操作）

**成功時保証**: 仕入スタッフが在庫状況を把握できる

**トリガー**: 仕入スタッフが在庫推移画面にアクセスする

**主成功シナリオ**:

1. 仕入スタッフが在庫推移画面を表示する
2. システムが単品ごとの日別在庫予定数を表示する
3. システムが品質維持日数に基づく廃棄予定を表示する
4. 仕入スタッフが特定の単品の詳細を確認する
5. システムが入荷予定・受注引当・廃棄予定の内訳を表示する

**拡張**:

- 2a. 特定の単品に絞り込む場合:
    - 2a1. 仕入スタッフが単品を指定する
    - 2a2. システムが指定された単品の在庫推移を表示する

**技術およびデータのバリエーションリスト**:

- ステップ 2: 在庫推移は表形式またはグラフ形式で表示

**関連情報**: BUC-04 に対応

## UC-005: 🖥️発注する🌊

**使用時のコンテキスト**: 在庫推移を確認した結果、仕入スタッフが不足する単品を仕入先に発注する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 仕入スタッフ

**利害関係者と利益**:

- 仕入スタッフ: 必要な花材を適切なタイミングで発注したい
- 仕入先: 正確な発注情報を受け取りたい
- 経営者: 発注判断を人間が行い、無駄な仕入れを防ぎたい

**事前条件**:

- 仕入スタッフがシステムにログインしている
- 単品マスタに仕入先・購入単位・リードタイムが登録されている

**最低保証**: 不完全な発注はシステムに保存されない

**成功時保証**: 発注が「発注済み」ステータスで登録され、在庫推移に入荷予定として反映される

**トリガー**: 仕入スタッフが在庫推移を確認し、発注が必要と判断する

**主成功シナリオ**:

1. 仕入スタッフが発注する単品を選択する
2. システムが単品の仕入先・購入単位・リードタイムを表示する
3. 仕入スタッフが発注数量を入力する
4. 仕入スタッフが希望納品日を入力する
5. システムが発注内容の確認画面を表示する
6. 仕入スタッフが発注を確定する
7. システムが発注を「発注済み」で登録し、在庫推移に入荷予定を反映する

**拡張**:

- 3a. 購入単位の倍数でない数量を入力した場合:
    - 3a1. システムが購入単位に切り上げた数量を提案する

**関連情報**: BUC-05 に対応

## UC-006: 🖥️入荷を受け入れる🌊

**使用時のコンテキスト**: 仕入先から単品が届いた際に、入荷実績を登録して在庫に反映する。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: 仕入スタッフ

**利害関係者と利益**:

- 仕入スタッフ: 入荷を正確に記録し、在庫を最新の状態にしたい
- フローリスト: 結束に必要な花材の在庫を正確に把握したい

**事前条件**:

- 対応する発注が「発注済み」ステータスである
- 単品が実際に到着している

**最低保証**: 不整合な入荷データはシステムに保存されない

**成功時保証**: 入荷実績が登録され、在庫数量が更新される

**トリガー**: 仕入先から単品が届く

**主成功シナリオ**:

1. 仕入スタッフが対応する発注を選択する
2. システムが発注内容（単品、数量、仕入先）を表示する
3. 仕入スタッフが入荷数量を入力する
4. システムが入荷を登録し、在庫を更新する
5. システムが発注ステータスを「入荷済み」に更新する

**拡張**:

- 3a. 発注数量と入荷数量が異なる場合:
    - 3a1. 仕入スタッフが実際の入荷数量を入力する
    - 3a2. システムが発注ステータスを「一部入荷」に更新する

**関連情報**: BUC-06 に対応

## UC-007: 🖥️花束を結束する🌊

**使用時のコンテキスト**: 出荷日（届け日の前日）にフローリストが花材から花束を組み立てる。

**スコープ**: WEB ショップシステム

**レベル**: ユーザー目的

**主アクター**: フローリスト

**利害関係者と利益**:

- フローリスト: 結束に必要な花材と受注情報を正確に把握したい
- 得意先: 新鮮で品質の高い花束を受け取りたい

**事前条件**:

- 対象の受注が「受付済み」ステータスである
- 出荷日（届け日の前日）である
- 必要な花材の在庫がある

**最低保証**: 結束に失敗した場合、受注ステータスは変更されない

**成功時保証**: 受注ステータスが「出荷準備中」に更新され、在庫から花材が消費される

**トリガー**: 出荷日の到来

**主成功シナリオ**:

1. フローリストが本日の結束対象一覧を表示する
2. システムが対象受注と必要な花材の一覧を表示する
3. フローリストが花束を結束する
4. フローリストが結束完了を登録する
5. システムが在庫から花材を消費し、受注ステータスを「出荷準備中」に更新する

**関連情報**: BUC-07 に対応

**主成功シナリオ**:

1. 配送スタッフが本日の出荷対象一覧を表示する
